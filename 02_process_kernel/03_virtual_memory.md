# 仮想メモリ — 住所の翻訳表が生む隔離と自由

## 概要

この章では、プロセスごとの「専用のメモリの眺め」(アドレス空間)を実現している
仕組み——仮想メモリ、ページテーブル、TLB——を扱います。前提知識は前2章
(task_struct と mm、コピーオンライト、例外、CR3 と switch_mm)です。
基準環境は Linux 7.0 / Ubuntu Server 26.04 LTS です。

## 導入 — もし物理メモリを直接使ったら

これまでの章で、アドレス空間を「プロセスごとにカーネルが与える専用のメモリの
眺め。他のプロセスのメモリは見えも触れもできない」と説明し、詳細をこの章に
先送りしてきました。いよいよその中身です。

仕組みに入る前に、**仮想メモリが無い世界**を想像してみます。マシンに 16 GiB の
物理メモリ(DRAM)が載っていて、すべてのプログラムがその物理的な番地
(**物理アドレス**)を直接使うとしたら、何が困るでしょうか。

1. **隔離がない** — メモリは全プロセスに丸見えの1枚の平原です。nginx が
   番地 0x5000 に置いたパスワードを、隣の悪意あるプロセスも番地 0x5000 を
   読むだけで盗めます。バグによる誤爆も同じで、あるプログラムのポインタの
   計算間違いが、無関係なプロセスやカーネルのデータを破壊します
2. **場所の取り合いになる** — プログラムは「自分がどの番地に置かれるか」を
   コンパイル時には知りようがありません。2つのプログラムが同じ番地を前提に
   作られていたら同時に動けませんし、起動のたびに空いている場所を探して
   コード中の番地を全部書き換える(再配置)のは大仕事です
3. **物理メモリの量が上限になる** — 16 GiB のマシンでは、全プロセスの
   使用量の合計が 16 GiB を超えた瞬間に手詰まりです。「今は触っていない
   部分」も物理メモリを占有し続けます

実際、初期のコンピュータや一部の組み込みシステムはこの世界で動いており、
これらの問題と常に格闘していました。現代の OS が採った解決策は、コンピュータ
科学の常套手段——**間に対応表を1枚挟む**——です。

- 各プロセスには、物理メモリとは無関係の**仮想アドレス**の空間を与える。
  どのプロセスも「自分は 0 番地から始まる広大なメモリを独り占めしている」
  という同じ景色を見る
- プロセスが仮想アドレスにアクセスするたびに、**翻訳表**を引いて本当の
  物理アドレスに変換する。表はカーネルだけが書き換えられる

こうすると、3つの問題が一挙に解けます。翻訳表に載っていない物理メモリには
原理的に到達できないので**隔離**が手に入り(問題1)、全プロセスが「自分は
0 番地から」と思い込んだままカーネルが裏で別々の物理位置に割り付けられる
ので**場所の取り合いが消え**(問題2)、さらに「表を引いた瞬間に初めて実体を
用意する」「使っていない部分は実体を持たせない」といった**遅延と節約の芸**が
可能になります(問題3。前章までに登場したコピーオンライトはこの芸の一種です)。

この章の残りは、この翻訳表の正体(ページテーブル)、表引きを高速化する
キャッシュ(TLB)、そして「表に載っていなかった瞬間」に起きること
(ページフォールト)の3点を軸に進みます。

## 理論

本節の内容は、主にカーネルドキュメントのメモリ管理概説
(**Documentation/admin-guide/mm/concepts.rst**)と x86-64 のアドレス空間
仕様(**Documentation/arch/x86/x86_64/mm.rst**)に基づきます。

### ページ — 翻訳の単位

翻訳表を「1バイトごと」に持つのは論外です(表がメモリ本体より大きくなり
ます)。そこでメモリを固定長の区画に切り、区画単位で対応させます。この区画を
**ページ(page)**と呼び、x86-64 の基本サイズは **4 KiB** です。仮想アドレス側の
区画を単に「ページ」、物理メモリ側の区画を**ページフレーム(page frame)**と
呼び分けます。

```
 仮想アドレス空間(プロセスAの眺め)         物理メモリ(DRAM)
 ┌──────────┐                        ┌──────────┐
 │ ページ 0     │──────翻訳表──────→ │ フレーム 731 │
 │ ページ 1     │──────翻訳表──────→ │ フレーム 42  │
 │ ページ 2     │  (対応なし=実体なし)     │   ...      │
 │   ...       │                        │            │
 └──────────┘                        └──────────┘
   連続に見える                            実際は飛び飛びでよい
```

重要な帰結が2つあります。第一に、**仮想側で連続に見える領域が、物理側で
連続している必要はない**こと。物理メモリの断片化(空きが飛び飛びにしか
ないこと)は、翻訳表が吸収してくれます。第二に、**対応を持たないページを
作れる**こと。「アドレスとしては存在するが、実体はまだ(もう)ない」という
状態を表せることが、後述する遅延の芸すべての土台になります。

アドレスの下位 12 ビット(2^12 = 4096)はページ内の位置(オフセット)を
表し、翻訳では変換されずそのまま通ります。翻訳されるのは残りの上位ビット
(仮想ページ番号→物理フレーム番号)だけです。

### ページテーブル — 翻訳表の実体

この翻訳表が**ページテーブル(page table)**です。1エントリが1ページ分の
対応を持ち、物理フレーム番号に加えて**属性ビット**を記録します。x86-64 の
ページテーブルエントリ(PTE)の主なビットは次のとおりです。

| ビット | 意味 |
|---|---|
| Present(P) | この対応が有効か。0 のページに触れると CPU が例外を上げる |
| R/W | 書き込み可か。0 なら読み取り専用(CoW はこれを使う) |
| U/S | ユーザーモードからアクセス可か。0 ならカーネル専用 |
| NX | 実行禁止か。データ領域を 1 にしておくと、注入されたコードの実行を CPU が拒否する |
| Accessed | CPU がアクセス時に自動で立てる。「最近使われたか」の証拠 |
| Dirty | CPU が書き込み時に自動で立てる。「書き換えられたか」の証拠 |

前章の「カーネル空間のメモリはユーザーから読むことすらできない」は U/S
ビットの、コピーオンライトの「書き込み禁止の印」は R/W ビットの仕業です。
Accessed / Dirty は後でページ回収の話をするときに効いてきます。

### なぜ多段か — 疎な空間を平らな表で持てない

x86-64 の仮想アドレスは標準で **48 ビット**(256 TiB)の空間を持ちます。
これを 4 KiB ページで割ると 2^36 = 約 687 億ページ。1エントリ 8 バイトの
平らな表を作ると、**1プロセスあたり 512 GiB** の表が必要になり、本末転倒です。

実際のプロセスが使うのは広大な空間のごく一部(疎)です。そこで表自体を
ページ単位に分割し、**使っている範囲の分だけ表を用意する**——これが
**多段ページテーブル**です。x86-64 の標準は4段で、48 ビットのうち上位
36 ビットを 9 ビットずつ4回に分けて使います(9 ビット = 1段の表が
512 エントリ = ちょうど 4 KiB に収まる設計です)。

```
 仮想アドレス(48ビット)
 ┌─────┬─────┬─────┬─────┬────────┐
 │ 9bit  │ 9bit  │ 9bit  │ 9bit  │ 12bit    │
 └──┬──┴──┬──┴──┬──┴──┬──┴────┬───┘
    │       │       │       │         └─ ページ内オフセット(素通し)
    ▼       ▼       ▼       ▼
 ┌────┐ ┌────┐ ┌────┐ ┌────┐
 │PGD  │→│PUD  │→│PMD  │→│PTE  │→ 物理フレーム番号 + 属性
 └────┘ └────┘ └────┘ └────┘
    ▲ 第1段の表の物理アドレスを CR3 レジスタが指す
```

各段は「次の段の表がどこにあるか」を指し、最終段(PTE)が物理フレーム番号を
持ちます。使っていない範囲は途中の段で「この先なし」と印を付けておけば、
その下の表を丸ごと省略できます。Linux はこの4段(実際にはさらに1段増えた
5段構成も抽象化した共通モデル)を PGD → PUD → PMD → PTE という名前で
扱います(P4D を挟む5段・57 ビットの拡張は Linux 4.14 以降がサポートして
おり、大容量メモリのサーバー向けです。本書では4段を基本に説明します)。

前章の switch_mm で登場した **CR3 レジスタ**の正体がここで確定します。
CR3 は**この多段の表の根(第1段の物理アドレス)**を指しており、CR3 を
差し替えることは翻訳表一式を丸ごと取り替えること、すなわち「見える世界」の
交換だったのです。

### MMU と TLB — 表引きを誰がやるか

さて、この表引きは**すべてのメモリアクセスごと**に必要です。命令を1つ
読むのも、変数を1つ書くのも、全部仮想アドレスだからです。これをソフトウェアで
やっていたら計算になりません。表引きは CPU 内のハードウェア
**MMU(Memory Management Unit、メモリ管理ユニット)**が自動で行います。
カーネルの仕事は表を**書いておく**ことまでで、引くのは MMU です。

それでもまだ問題が残ります。4段の表引きは、**1回のメモリアクセスのために
追加で4回メモリを読む**ことを意味します(表自体がメモリ上にあるため)。
そこで MMU は、一度引いた翻訳結果(仮想ページ→物理フレーム)を
**TLB(Translation Lookaside Buffer、アドレス変換バッファ)**という CPU 内の
小さな高速キャッシュに溜めます。以後同じページへのアクセスは TLB を見る
だけで済み(TLB ヒット)、表を実際に歩くのは TLB に無い場合(TLB ミス)
だけになります。プログラムのメモリアクセスには強い局所性(同じ辺りを
続けて触る性質)があるため、通常はほとんどのアクセスがヒットします。

TLB のエントリ数は現代の CPU で千数百程度の桁です。4 KiB ページなら
「TLB が一度に覚えていられる範囲」は数 MiB 程度にしかなりません。この
「TLB の守備範囲(リーチ)」の狭さが、後述する Huge Page の動機になります。

ここで前章の宿題を回収します。**コンテキストスイッチで CR3 を差し替えると、
TLB の中身は原則すべて無効**です——翻訳結果は「どの表で引いたか」に依存する
のに、表が丸ごと入れ替わったからです。切り替え直後のタスクは TLB ミスの嵐から
始まる——これが前章で述べた「間接費」の一部です。この緩和策が
**PCID(Process Context Identifier)**で、TLB の各エントリに「どのアドレス
空間の翻訳か」を示す札を付け、CR3 切り替え時に全部捨てる代わりに
「札が違うエントリは使わない」で済ませます。頻繁に行き来する少数のプロセス
間では、戻ってきたときに前回の翻訳結果がまだ生きている、というわけです。
前章の KPTI(モード切替のたびに CR3 を切り替える)が実用に耐えるのも、
PCID がカーネル用/ユーザー用の表を別の札で共存させているからです。

### ページフォールト — 「表に無い」は失敗ではなく仕組み

MMU が表を引いたとき、Present ビットが 0 だったり、R/W 違反(読み取り専用
ページへの書き込み)だったりすると、CPU は**例外**を上げてカーネルに制御を
移します。前章の「3つの門」の3番目です。この例外を
**ページフォールト(page fault)**と呼びます。

名前に「フォールト(fault)」とありますが、**大半のページフォールトは異常では
なく、カーネルが仕掛けた正常な仕組みの発動**です。カーネルは「表にわざと
載せない」「わざと書き込み禁止にしておく」ことで、**最初に触られた瞬間に
呼び出される罠**を仕掛けられます。これが導入で予告した「遅延と節約の芸」の
正体で、代表例は次のとおりです。

- **デマンドページング(demand paging)** — メモリの割り当て要求
  (malloc や mmap)に対し、カーネルは**アドレス範囲の予約だけ**を記録して
  即座に返り、物理フレームは割り当てません。プログラムが実際にそのページに
  触れた瞬間、ページフォールトが起きて初めてフレームを割り当てます。
  「確保したが結局使わなかった」領域が物理メモリを一切消費しない仕組みです
- **コピーオンライト** — fork 時に親子のページテーブルを同じフレームに
  向けて R/W を 0 にしておき、書き込みのフォールトを合図にそのページだけ
  複製する(既出の総復習です。仕掛けの部品が全部そろいました)
- **ファイルの読み込みの遅延** — 実行ファイルや共有ライブラリは、exec の
  時点では中身をメモリに読みません。「このアドレス範囲はこのファイルの
  この部分」という対応だけ記録し、実際に実行・参照されたページだけを
  フォールト契機でディスクから読みます。巨大なプログラムの起動が一瞬で
  済むのはこのためです
- **スワップ(swap)** — 物理メモリが足りなくなったとき、カーネルは
  しばらく使われていないページ(Accessed ビットが手掛かり)の中身を
  ディスク上のスワップ領域に退避し、フレームを回収します。表には
  「実体はスワップのここ」と記録しておき、後で触られたらフォールト契機で
  読み戻します。物理メモリの量を超える「見かけの容量」はこうして作られます

ページフォールトは、ディスク I/O を伴うかどうかで**メジャーフォールト**
(スワップからの読み戻しやファイルの初回読み込みなど、ディスクを待つ)と
**マイナーフォールト**(ゼロページの割り当てや CoW など、メモリ操作だけで
済む)に分類され、`/proc` で別々に数えられています(実行例参照)。この区別は
トラブルシューティングで決定的に効きます——マイナーは µs 桁、メジャーは
ディスク往復ぶん(ms 桁になり得る)遅いからです。

そして、仕掛けた覚えのない場所へのアクセス——どの予約範囲にも入っていない
アドレス、権限のない方向のアクセス——だった場合、カーネルはそのプロセスに
**SIGSEGV** シグナルを送ります。プログラムを書いたことがある読者が一度は
見ている **Segmentation fault** の正体は、「MMU が例外を上げ、カーネルが
正当な理由を見つけられなかったページフォールト」なのです(シグナルの
仕組み自体は `05_signals_ipc.md` で扱います)。

## 内部動作の詳細

### mm_struct と VMA — カーネルが持つ「予約の台帳」

前節で「カーネルはアドレス範囲の予約だけを記録する」と繰り返しました。
その記録先を見ます。task_struct が参照する **mm_struct**(アドレス空間の
台帳)は、ページテーブルの根(pgd)と並んで、
**VMA(vm_area_struct、仮想メモリ領域)**のリストを持っています。
VMA 1個が「連続する仮想アドレス範囲1つの予約」を表し、
次を記録します(カーネルソース `include/linux/mm_types.h`)。

- 範囲(開始・終了アドレス)
- 許可(読み/書き/実行——ページテーブルの属性ビットの「あるべき姿」)
- 中身の出どころ: ファイルの一部か(**ファイルバック**。実行ファイル、
  共有ライブラリなど)、それとも無名か(**アノニマス**。ヒープ、スタック
  など、ゼロから始まりスワップに退避されるメモリ)

つまりアドレス空間の実像は、**「予約の台帳(VMA)」と「翻訳表(ページ
テーブル)」の二層**です。VMA が「本来どうあるべきか」を、ページテーブルが
「いま実際にどうなっているか」を持ちます。mmap システムコール
(`man 2 mmap`。POSIX が規定するメモリ割り当ての基本インタフェース)が
やることは本質的に **VMA を1個追加するだけ**であり、ページテーブルは
フォールトが起きるまで白紙のままです。

なお、1プロセスの VMA は数千個に達することがあり、フォールトのたびに
「このアドレスはどの VMA か」を高速に探す必要があるため、Linux 6.1 以降は
**メイプルツリー(maple tree)**という範囲検索向けのデータ構造で管理されて
います(6.0 以前は赤黒木+リンクリスト。この置き換え自体が大規模プロセスの
性能改善のためでした)。

`/proc/<PID>/maps` は、この VMA リストをそのまま人間向けに印字したものです
(`man 5 proc`)。典型的なプロセスのアドレス空間はこう並んでいます。

```
 低位アドレス
 ┌───────────────┐
 │ text(実行コード)     │ r-x  実行ファイルのファイルバック
 │ data / bss(大域変数) │ rw-  同上(書き込みで CoW)+アノニマス
 │ heap(malloc の領域)  │ rw-  アノニマス。brk/mmap で伸びる
 │        ↓伸長         │
 │                       │
 │ mmap 領域             │      共有ライブラリ(libc.so 等)、
 │  (ライブラリ・大きな   │      巨大 malloc、[vdso] もここ
 │   割り当てが並ぶ)      │
 │                       │
 │        ↑伸長         │
 │ stack                 │ rw-  アノニマス。下向きに伸びる
 ├───────────────┤ ← ここから上はカーネル空間
 │ カーネル領域           │      (U/S ビットで全プロセス共通に
 └───────────────┘       マップされているが触れない)
 高位アドレス
```

各領域の開始位置は、攻撃者にアドレスを予測させないため起動ごとにランダムに
ずらされます(**ASLR**、Address Space Layout Randomization。分野07で再訪)。
また前章の vDSO が「全プロセスに貼り付けられる」の実体は、exec のたびに
カーネルが [vdso] という VMA を自動で1個追加することです。

### ページフォールトハンドラの分岐 — 1枚のフローに集約する

ここまでの部品を全部使うのが、カーネルのページフォールト処理
(x86-64 では `arch/x86/mm/fault.c` から `mm/memory.c` へ)です。CPU から
「どのアドレスで、読み/書き/実行のどれで、ユーザー/カーネルどちらのモードで」
という情報を受け取り、次の順に判定します。

```mermaid
graph TD
    A[ページフォールト発生<br>フォールトアドレスを CR2 から取得] --> B{アドレスを含む<br>VMA はあるか}
    B -- ない --> SEGV[SIGSEGV 送付<br>= Segmentation fault]
    B -- ある --> C{アクセス種別は<br>VMA の許可の範囲内か<br>例: 書き込み vs rw-}
    C -- 違反 --> SEGV
    C -- 適合 --> D{ページテーブルの<br>いまの状態は?}
    D -- "エントリなし・アノニマス" --> E[新しいフレームを割り当て<br>ゼロで埋めて表に記入<br>(デマンドページング/マイナー)]
    D -- "エントリなし・ファイルバック" --> F[ページキャッシュを探す<br>なければディスクから読む<br>(あればマイナー/なければメジャー)]
    D -- "書き込み禁止だが VMA は書き込み可" --> G[フレームを複製し<br>自分専用にして R/W を立てる<br>(CoW/マイナー)]
    D -- "実体はスワップ上" --> H[スワップから読み戻して記入<br>(メジャー)]
    E --> I[ユーザー空間へ復帰<br>同じ命令を再実行 → 今度は成功]
    F --> I
    G --> I
    H --> I
```

このフローが示す重要な点が2つあります。第一に、**SIGSEGV かどうかを決める
のは VMA(予約の台帳)**であって、ページテーブルの状態ではないこと。表が
白紙でも、正当な予約があればカーネルは黙って実体を用意します。第二に、
処理の最後は必ず**「フォールトを起こした命令そのものを再実行」**で終わる
こと。プログラムから見ると、何事もなかったかのように1回のメモリアクセスが
(少し遅れて)成功しただけです。前々章で「CoW はプロセスに透明」と述べた
のは、この再実行の仕組みのおかげです。

ファイルバックのページで登場した**ページキャッシュ(page cache)**について
一言だけ補足します。カーネルは一度ディスクから読んだファイルの中身を
ページ単位でメモリに残し、次回以降の読み込みやプロセス間の共有に使い回します
(同じ libc.so.6 のコードのフレームが全プロセスで1組しかないのは、全員の
ページテーブルが同じページキャッシュのフレームを指しているからです)。
ページキャッシュは「空きメモリの有効活用」であり、必要になれば即座に
回収されます。詳細はファイルシステムの分野
(`03_filesystem_storage/01_vfs_basics.md`)で扱います。

### メモリ不足のときに起きること — 回収と OOM killer

デマンドページングとオーバーコミット(次の段落)の世界では、「物理メモリが
足りない」はフォールト処理の途中で発覚します。フレームを割り当てたいのに
空きがないとき、カーネルは回収(reclaim)を試みます。優先的に狙われるのは
(1) ページキャッシュのうち最近使われていないもの(ディスクに実体があるので
捨てるだけでよい。Dirty なら書き戻してから)、(2) アノニマスページのうち
最近使われていないもの(スワップへ退避。スワップが無ければ回収不能)です。
「最近使われていない」の判定に、PTE の Accessed ビットを定期的に見て回る
LRU(Least Recently Used)ベースの仕組みが使われます。

それでも足りない最終局面では、カーネルは **OOM killer(Out Of Memory
killer)**を発動し、メモリ使用量などから算出した点数(`/proc/<PID>/oom_score`)
が最大のプロセスを**強制終了してフレームを没収**します。乱暴に聞こえますが、
これは Linux が既定で**オーバーコミット**——物理メモリ+スワップの合計を
超える予約(VMA)を許す方針——を採っていることの帰結です。予約時に厳格に
断らない(実際にはほとんどのプログラムが予約を使い切らないので、断ると
メモリが大幅に無駄になる)代わりに、本当に使われて破綻したときの後始末が
OOM killer なのです(方針は `/proc/sys/vm/overcommit_memory` で変更可能。
`man 5 proc` および Documentation/mm/overcommit-accounting.rst)。

### 発展: Huge Page — TLB の守備範囲を広げる

TLB のリーチ(千数百エントリ × 4 KiB ≒ 数 MiB)は、数十 GiB のメモリを
使うデータベースのようなプロセスには狭すぎ、TLB ミスが性能を支配し始めます。
そこで x86-64 は PMD 段で表引きを打ち切る **2 MiB ページ**(さらに 1 GiB
ページ)をサポートしており、1エントリで 512 倍の範囲を覆えます。Linux では
明示的に確保する HugeTLB(Documentation/admin-guide/mm/hugetlbpage.rst)と、
カーネルが自動で 4 KiB ページを 2 MiB に「まとめ上げる」
THP(Transparent Huge Pages、Documentation/admin-guide/mm/transhuge.rst)の
2系統があります。表引きの段数が1段減る効果もあり、大メモリのサーバー
チューニングの定番ですが、断片化との相性など癖もあるため、ここでは
「TLB のリーチという制約と、その緩和策」という構図だけ覚えてください。

## 実行例 — 台帳と会計を観察する

前提は Ubuntu Server 26.04 LTS です。

ページサイズを確認する:

```console
$ getconf PAGESIZE
4096
```

VMA の台帳(/proc/PID/maps)を見る。自分自身の cat のアドレス空間:

```console
$ cat /proc/self/maps
5f8a3c200000-5f8a3c208000 r-xp 00003000 fc:02 1837 /usr/bin/cat   ← text(実行コード)
5f8a3c209000-5f8a3c20a000 rw-p 00008000 fc:02 1837 /usr/bin/cat   ← data
5f8a5d3f1000-5f8a5d412000 rw-p 00000000 00:00 0    [heap]          ← アノニマス
7b1e4a800000-7b1e4a9b5000 r-xp 00028000 fc:02 9214 /usr/lib/x86_64-linux-gnu/libc.so.6
7ffc8d5e2000-7ffc8d603000 rw-p 00000000 00:00 0    [stack]
7ffc8d60b000-7ffc8d60d000 r-xp 00000000 00:00 0    [vdso]          ← 前章のvDSO
```

列の意味は「範囲 / 許可(r 読み・w 書き・x 実行・p 非共有) / ファイル内
オフセット / デバイス / inode / 出どころ」。出どころが空欄や [heap] [stack] の
行がアノニマス、パスが入っている行がファイルバックの VMA です。

予約(仮想)と実際の使用(物理)の差を見る。VmSize が VMA の合計、
VmRSS(Resident Set Size)が実際に物理フレームを持つ量:

```console
$ grep -E 'VmSize|VmRSS' /proc/$$/status
VmSize:     11172 kB    ← 予約の合計(仮想)
VmRSS:       5888 kB    ← 実際に物理メモリにある量
```

VmSize > VmRSS の差が、デマンドページングの「まだ触っていない・実体のない」
部分です。大きなプログラムではこの差は桁違いになります。

ページフォールトの回数を種別ごとに見る:

```console
$ ps -o pid,min_flt,maj_flt,comm -p $$
    PID  MINFL  MAJFL COMMAND
   4210   3512      2 bash     ← マイナーが圧倒的多数なのが正常
```

コマンド1回の裏で起きるフォールトを数える:

```console
$ /usr/bin/time -v ls / 2>&1 | grep -i fault
        Major (requiring I/O) page faults: 0     ← 2回目以降はページキャッシュに
        Minor (reclaiming a frame) page faults: 98    乗っているのでメジャー0
```

システム全体のメモリの内訳。「free が少ない」がページキャッシュのせいで
あることを確認する:

```console
$ free -h
               total        used        free      shared  buff/cache   available
Mem:            15Gi       1.2Gi       340Mi        12Mi        14Gi        13Gi
Swap:          4.0Gi          0B       4.0Gi
                                ↑            ↑ページキャッシュ等   ↑本当の余力
                       「free」はほぼ0でも異常ではない
```

スワップの出入り(si/so 列)を見る。ここが恒常的に非ゼロなら物理メモリ不足:

```console
$ vmstat 1 3
procs -----------memory---------- ---swap-- -----io---- -system-- -------cpu-------
 r  b   swpd   free   buff  cache   si   so    bi    bo   in   cs us sy id wa st
 0  0      0 348120  81234 14523456   0    0     1     3  120  310  1  0 99  0  0
                                     ↑読み戻し ↑退避(ページ/秒)
```

## トラブルシューティング — 仮想と物理を混同しない

- **`top` の VIRT が巨大で心配になる**: VIRT は予約(VmSize)であり、物理
  メモリの消費ではありません。数十 GiB の VIRT でも RES(=VmRSS)が小さければ
  実害はありません。メモリ消費を論じるときは RES を、プロセス間で共有される
  ぶん(libc 等)まで厳密に按分したいときは `/proc/<PID>/smaps_rollup` の
  PSS を見ます
- **Segmentation fault の一次調査**: 本章のフローのとおり「VMA に無い/許可に
  反するアクセス」が起きています。カーネルは概要を記録しており、
  `dmesg` に `segfault at <アドレス> ip <命令位置>` の形で残ります。
  アドレスが 0 付近なら NULL ポインタ経由のアクセスが典型です(0 番地付近に
  VMA を置かないのはこれを確実に検出するための慣例です)
- **メモリは足りているはずなのに遅い**: `sar -B` や `ps -o maj_flt` で
  メジャーフォールトが増え続けていないかを見ます。恒常的なスワップの出入り
  (vmstat の si/so)+高い I/O 待ちの組は**スラッシング**——ページの退避と
  読み戻しが往復し続け、CPU がほとんど仕事を進められない状態——の徴候で、
  対処は負荷の削減か物理メモリの増設です
- **プロセスが突然消えた・killed とだけ出た**: OOM killer を疑い、
  `journalctl -k | grep -i oom`(または dmesg)を確認します。
  `Out of memory: Killed process ...` の記録があれば、直前のメモリ消費の
  内訳(記録に各プロセスの RSS 一覧が残ります)から犯人——殺されたプロセス
  ではなく、メモリを食い尽くした側——を特定します
- **free の空きが少ない、と報告してしまう前に**: 実行例のとおり、Linux は
  空きメモリを遊ばせずページキャッシュに使うため、健全なサーバーほど
  「free」は小さくなります。見るべきは **available** 列(回収可能なぶんを
  加味した実質の余力)です。「Linux ate my RAM」と呼ばれる古典的な誤解です
- **malloc が成功したのに、使った瞬間に死ぬ**: オーバーコミットの帰結です。
  予約(malloc)は物理の裏付けなしに成功し、実際に触った瞬間のフォールト
  処理で破綻が発覚して OOM killer が動きます。「malloc の戻り値検査だけでは
  メモリ不足は検出できない」ことは、この仕組みの重要な帰結として覚えて
  ください

## 演習・確認問題

1. 仮想メモリが解決する3つの問題(隔離・配置・容量)を、「翻訳表を挟む」
   ことがなぜそれぞれの解決になるのかとあわせて説明してください
2. x86-64 で平らな(1段の)ページテーブルが実用にならない理由を、表の
   サイズの概算とともに説明してください。多段化はこれをどう解決しますか
3. マイナーフォールトとメジャーフォールトの違いは何ですか。また「大半の
   ページフォールトは異常ではない」と言える理由を、デマンドページングと
   CoW を例に説明してください
4. コンテキストスイッチの間接費として「TLB の汚染」が挙げられる理由と、
   PCID がそれをどう緩和するかを説明してください。同じプロセス内の
   スレッド交代ではこの間接費がどうなるかにも触れてください
5. `top` で VIRT 32 GiB / RES 800 MiB のプロセスを見つけた同僚が
   「メモリを 32 GiB も食っている」と報告してきました。何がどう違うのか、
   VMA とページテーブルの二層構造を使って訂正してください

## まとめ

- 仮想メモリは「仮想アドレス→物理アドレスの翻訳表(ページテーブル)を
  挟む」仕組み。隔離・配置の自由・遅延と節約の芸(デマンドページング、
  CoW、スワップ、ページキャッシュ)がすべてここから生まれる
- 表は 4 KiB のページ単位・多段構成(PGD→PUD→PMD→PTE)で、根を CR3 が
  指す。表引きは MMU が行い、結果は TLB にキャッシュされる。CR3 の差し替え
  =世界の交換であり、TLB 汚染が交代の間接費になる(緩和策が PCID)
- アドレス空間の実像は「予約の台帳(VMA)」と「実態の表(ページテーブル)」の
  二層。mmap は VMA を足すだけで、実体はページフォールトが遅延して用意する
- ページフォールトの大半は正常な仕組みの発動であり、VMA に照らして正当性が
  確認できないものだけが SIGSEGV(Segmentation fault)になる
- 観察は /proc/PID/maps(台帳)、VmSize/VmRSS(予約と実際)、min_flt/maj_flt
  (フォールト会計)、free の available(実質の余力)、vmstat の si/so
  (スワップの出入り)で行う
