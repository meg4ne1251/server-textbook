# ネットワークネームスペース — スタックまるごと複製する

## 概要

前2章で追った「1台のマシンのネットワークスタック一式」(デバイス、
ルーティング表、検問所、conntrack の台帳、ソケット)を、**1台の中に
複数、互いに独立して持てる**仕組みがネットワークネームスペースです。
コンテナのネットワーク隔離の正体でもあります。前提知識は前2章
(送受信の道のり、netfilter)。基準環境は Linux 7.0 / Ubuntu Server
26.04 LTS。ネームスペース一般とコンテナへの応用は分野05
(`05_virtualization_containers/01_cgroups_namespaces.md`)が主で、
本章はネットワークに限って機構を掘り下げます。

## 導入 — 「1台に1式」をやめたくなるとき

ここまでの2章は、ずっと暗黙の前提を置いていました。**スタックは
1台のマシンに1式**——NIC の一覧は1つ、ルーティング表は1つ、
nftables のルールセットも1つ、ポート番号 80 を listen できるのも
1プロセスだけ、という前提です。

しかし、この前提が邪魔になる場面があります。

1. **同居させたい。** 1台の物理サーバーに複数の顧客・複数の
   アプリケーションを収容したい。それぞれに「自分専用のネットワーク」
   ——自分の IP、自分のルーティング表、自分のファイアウォール——を
   見せたい。ポート 80 の取り合いもさせたくない
2. **試したい。** ルータ3台とサーバー2台のネットワーク構成を、
   物理機材なしで、1台のノート PC の中に組んで実験したい
3. **隔離したい。** このプロセスには、ホストのネットワークを
   一切触らせたくない(あるいは特定の経路しか見せたくない)

思い出してほしいのは、分野02で学んだ仮想メモリの発想です。物理
メモリは1つしかないのに、プロセスごとに専用の「メモリの眺め」
(アドレス空間)を与えることで、隔離と同居を両立しました。Linux は
同じ型をネットワークにも適用します——**カーネルの中にスタック一式の
複製を作り、プロセスごとに「どの複製を見るか」を差し替える**のです。

この「カーネルが管理する資源の眺めを、プロセスのグループごとに
分ける」仕組みの総称が**ネームスペース(namespace)**です。分けられる
資源はネットワークのほかにもあり(マウントの眺め、PID の眺め、
ホスト名など)、その組み合わせがコンテナの土台になります——全体像は
分野05に譲り、本章はそのうち最も劇的な**ネットワークネームスペース
(network namespace、netns)**だけを扱います。「劇的」というのは、
複製されるものの多さです。新しいネットワークネームスペースを作ると、
その中は**配線される前の新品のマシン**のようになります——NIC は
ループバックすらまだ起きておらず、ルーティング表は空、検問所には
何も登録されていません。

## 理論

本節の内容は、主に **`man 7 network_namespaces`**、`man 7 namespaces`、
`man 8 ip-netns`、`man 4 veth`、および Linux カーネルソース
(include/net/net_namespace.h、drivers/net/veth.c、net/bridge/)に
基づきます。ネームスペース機構は Linux 2.4.19 のマウント
ネームスペースに始まり、ネットワークネームスペースは Linux 2.6.24
(2008年)から段階的に導入されました。

### 何が複製されるのか — 前2章の全登場人物

ネットワークネームスペースが複製する(=ネームスペースごとに独立に
なる)ものを、前2章の登場人物に重ねて確認します。

| 複製されるもの | 前章までの登場箇所 |
|---|---|
| ネットワークデバイスの一覧(lo 含む) | 前々章「送受信の道のり」の出入口 |
| IP アドレス、ルーティング表 | 前々章の送信経路(IP 層の表引き) |
| netfilter のフックとルールセット | 前章の検問所 |
| conntrack の台帳 | 前章の「顔パス」の台帳 |
| ソケットとポート番号の空間 | 前々章の4つ組・bind |
| `net.*` の sysctl(ip_forward 等) | 前章までの各種つまみ |
| `/proc/net` の中身 | 各章の観察窓 |

つまり**部分的な共有はありません**。ルーティング表だけ共有して
ルールは別、といった中間はなく、スタックの状態はまるごと1式が
単位です。これは設計上の選択で、理由は整合性にあります——前章で
見たとおり、検問所・台帳・ルーティング判断は密に連携して動きます
(DNAT はルーティング判断の前、NAT は conntrack の上、など)。
一部だけ差し替えられる設計にすると、この連携の整合性をあらゆる
組み合わせで保証しなければなりません。「全部で1式」はそれを
単純さで解決しています。

2つ、重要な帰結があります。

- **ポート番号の衝突が消える。** bind の一意性(前々章)は
  ネームスペースの中だけの話になり、100 個のネームスペースで
  100 個のプロセスがそれぞれ 80 番を listen できます
- **1つのデバイスは、常にどれか1つのネームスペースに属する。**
  物理 NIC も例外ではありません。eth0 をあるネームスペースへ
  移せば、元の場所からは**消えます**。複製されるのは「スタックの
  状態」であって、ハードウェアは増えないからです

### 孤島を配線する — veth とブリッジ

新品のネームスペースは完全な孤島です。ループバックしかなく、外との
通信手段がありません。物理 NIC を移してしまう手はありますが、NIC は
1枚しかないのが普通で、それでは元のネームスペースが孤島になります。
そこで、**仮想のデバイスで配線する**道具が用意されています。

**veth ペア(virtual Ethernet pair)**は、仮想 NIC を必ず**2枚1組**で
作るデバイスです。片方に送信したフレームが、そのままもう片方の受信に
なります——2枚の NIC を1本の LAN ケーブルで直結した状態を、カーネル内の
ソフトウェアだけで作ったものです。分野02のパイプ(2つの fd が1本の
バッファでつながる)のネットワーク版と考えると、既視感があるはず
です。ペアの片割れを別のネームスペースへ移せば、境界をまたぐ1本の
ケーブルになります。

```
  ホストのネームスペース          ネームスペース ns1
  ┌─────────────────┐            ┌─────────────────┐
  │  eth0(物理)      │            │       lo        │
  │  veth0 ━━━━━━━━━┿━━━━━━━━━━━━┿━━ veth1         │
  └─────────────────┘  veth ペア  └─────────────────┘
                      (仮想の直結ケーブル)
```

ネームスペースが2つならこれで足ります。しかし 10 個になったら、
1対1のケーブルを 45 本張るわけにはいきません。現実の機材と同じで、
集線装置が欲しくなります。それが**ブリッジ(bridge)**——カーネル内の
ソフトウェア実装による L2 スイッチです。ブリッジというデバイス
(br0 等)を作り、各 veth のホスト側の端を「ポート」として挿すと、
挿された端どうしはフレームの MAC アドレス学習に基づいて相互に
通信できます。L2 スイッチとしての動作原理(MAC 学習、フラッディング)
は姉妹書 network-guide の主題です
(`../../network-guide/02_vlan_vxlan_evpn/01_vlan_basics.md`)。

外の世界(物理ネットワーク)へ出るには、最後の一歩が要ります。
代表的な構成は、**ホストのネームスペースをルータとして使う**もの
です。ブリッジに IP アドレスを与えてホスト側のゲートウェイとし、
ホストでフォワーディング(`net.ipv4.ip_forward`)を有効にして、
前章の postrouting で masquerade(SNAT の一種。出ていく物理 NIC の
アドレスに書き換える)を掛けます。つまり——

**コンテナのネットワークの標準構成の正体は、「ネームスペース+
veth+ブリッジ+前章の NAT」の組み合わせです。** docker0 という
ブリッジ、ホストで見える大量の veth、iptables/nftables に自動追加
される masquerade ルール——コンテナ環境で目にするこれらの部品は、
本章と前章の道具立てそのものです(コンテナランタイムがこの配線を
どう自動化するかは分野05で扱います)。

### ホストの中の分割と、ホストをまたぐ延伸

ネームスペースは**1台のホストの中で**ネットワークを分割する道具
です。分割した島々を**複数のホストにまたがって**同じ L2 につなぎ
たい——コンテナ基盤やデータセンターで必ず出てくるこの要求に応える
のが、L2 フレームを L3 でトンネルする VXLAN です。プロトコルと
しての VXLAN・その制御プレーン(EVPN)は network-guide が主です
(`../../network-guide/02_vlan_vxlan_evpn/03_vxlan_fundamentals.md`)。
本書の側から言えるのは、Linux では VXLAN もまた「仮想デバイスの
一種」として実装されており、ブリッジに挿せば veth と同列のポートに
なる、という接続の仕方です。ネームスペースで分け、ブリッジで束ね、
VXLAN で延伸する——3つの道具は同じ「デバイスとブリッジ」の語彙で
組み合わせられます。

## 内部動作の詳細

### struct net — 「スタック一式」の実体

ネットワークネームスペースの実体は、カーネル内の **`struct net`**
(include/net/net_namespace.h)という1個の大きな構造体です。
デバイスのリスト、ルーティング表、netfilter のフック登録、conntrack
の台帳、sysctl の値——理論の節で「複製される」と列挙したものは、
すべてこの構造体のメンバ(またはそこから参照される表)です。

歴史的には、これらはカーネル内の**グローバル変数**でした。だから
こそ「1台に1式」だったのです。Linux 2.6.24 からの長い作業で、
ネットワークサブシステムの全域にわたりグローバル変数が
`struct net` のメンバへ移されました(この移行が「ネットワーク
スタックまるごと」という複製単位を可能にした地道な正体です)。
起動時から存在するホスト本来のネームスペースも、いまや特別な
実装ではなく、`init_net` という名の `struct net` の1インスタンス
にすぎません。

プロセスとの対応づけは、task_struct(分野02)から2段の参照で
たどります。

```
  task_struct ──→ nsproxy ──→ struct net(ネットワーク)
                     ├───────→ (マウントの眺め)
                     ├───────→ (UTS: ホスト名)
                     └───────→ (その他のネームスペース。分野05)
```

nsproxy は「このタスクが各種ネームスペースのどれに所属するか」を
束ねた小さな構造体で、所属が同じプロセスどうしは参照カウントで
共有します。fork した子は親と同じネームスペースを指す——環境変数や
fd テーブルと同じ「fork は眺めを引き継ぐ」の一例です。

ここで1つ、重要な例外があります。**ソケットの所属は、作成時の
ネームスペースに固定**されます(struct sock がネームスペースへの
参照を持ち、`sock_net()` でたどれます)。プロセスが後から別の
ネームスペースへ移動しても、**開設済みのソケットは元のスタックに
属したまま**です。前々章の「ソケットは fd の顔をした通信の端点」を
思い出してください——fd がプロセスに付いていくのに対し、通信の
実体(4つ組、キュー)はスタック側の存在なので、スタックから
切り離して持ち歩くことはできないのです。この性質は意図的に使う
こともできます(あるネームスペースでソケットを開いてから移動すれば、
「体は隔離、通信路は外」という構成が作れます)。

### 生成と寿命 — fd と bind mount が命綱

ネームスペースを作る・移る窓口は3つのシステムコールです。

- **`clone(CLONE_NEWNET)`** — 新しいネームスペースの中に子プロセスを
  作る(fork の亜種。分野02で見た clone のフラグの1つ)
- **`unshare(CLONE_NEWNET)`** — 呼び出したプロセス自身が、その場で
  新品のネームスペースへ移る
- **`setns(fd)`** — fd で指した**既存の**ネームスペースへ移る

3つ目の setns が示すとおり、ネームスペースは **fd で参照できる**
オブジェクトです。各プロセスの所属は `/proc/<PID>/ns/net` という
特殊なシンボリックリンクとして見えており、その指し先
`net:[4026531840]` の数字は inode 番号です——**inode 番号が
ネームスペースの識別子**であり、2つのプロセスが同じネームスペースに
いるかは、このリンクの指し先が同じかで判定できます(「すべては
ファイル」がここでも貫かれています)。この /proc のファイルを
open すれば、setns に渡す fd が得られます。

寿命は参照カウントで決まります。ネームスペースは、**所属する
プロセスが1つもなくなり、参照する fd もマウントもなくなった時点で
破棄**されます(その中の veth は消え、移されていた物理 NIC は
`init_net` へ帰ります)。ここで問題になるのが「プロセスを常駐させずに
ネームスペースだけ生かしておきたい」場合です。`ip netns add` は
これをファイルシステムで解決しています——新しいネームスペースを
作ると同時に、`/proc/self/ns/net` を `/run/netns/<名前>` へ
**bind mount** するのです。マウントが参照を握り続けるため、中に
プロセスがいなくてもネームスペースは生き続け、しかも名前の付いた
ファイルとして一覧・参照できるようになります。`ip netns` サブ
コマンド群は、この規約の上に乗った管理ツールにすぎません。

### veth の内部 — 送信がそのまま相方の受信になる

veth の実装(drivers/net/veth.c)は、前々章の道のりの言葉で正確に
言えます。veth デバイスへの送信処理(ドライバの xmit)は、ハード
ウェアに何かを書き込む代わりに、**skb の所属を相方のデバイスに
付け替えて、相方側の受信キューに積み、NET_RX の softirq を上げ
ます**。以降は物理 NIC からの受信とまったく同じ道です——相方の
ネームスペースの prerouting の検問所を通り、そのネームスペースの
ルーティング判断を受け、ソケットへ配達されます。

- 物理的な配線・信号は存在せず、コストはメモリ操作だけ
- 「送信の終わり=受信の始まり」なので、パケットが失われる要素も
  原則ない(受け側の処理が追いつかない場合の破棄はある)
- 検問所・台帳は**双方のネームスペースでそれぞれ独立に**通る。
  veth 越しの通信は、ns1 の output/postrouting を通ってから、
  ns2 の prerouting/input を通る——2台のマシン間の通信と同じ構図

### 通し稽古 — ネームスペースの中から外の世界へ

コンテナ型の標準構成(ns1 の veth1 ─ ブリッジ br0 ─ 物理 eth0、
masquerade あり)で、ns1 の中のプロセスが外部のサーバーへ
パケットを1個送るまでを、前2章の語彙だけで通してみます。

1. **ns1 の中**: ソケットへの write。ns1 の**ルーティング表**が
   引かれ、「デフォルトゲートウェイは br0 のアドレス、出口は
   veth1」と決まる。ns1 の output / postrouting の検問所を通過
   (ns1 にルールがなければ素通し)
2. **veth を渡る**: veth1 の送信=ホスト側 veth0 の受信。skb は
   ホストのネームスペースの世界に入る
3. **ブリッジ**: veth0 は br0 のポートなので L2 の転送判断を受け、
   宛先 MAC(ゲートウェイ=br0 自身)により、ホストのスタックに
   「br0 に届いたパケット」として上がる
4. **ホストの検問所**: prerouting を通り、ルーティング判断。宛先は
   外部なので**転送**——ここで `net.ipv4.ip_forward`(これも
   ネームスペースごとの sysctl)が 0 なら破棄されます。forward の
   検問所を通り、postrouting で masquerade が働いて差出人が
   eth0 のアドレスに書き換わり、**ホストの conntrack 台帳**に
   変換が記録される(前章)
5. **eth0 から送信**。応答パケットは逆をたどる——ホストの台帳で
   逆変換(顔パス)、forward、ブリッジ、veth、ns1 の prerouting /
   input、そしてソケットへ

検問所と台帳が「ネームスペースごとに独立」でありながら、1つの
パケットが**複数のネームスペースの検問所を順に通る**ことに注意して
ください。トラブルシューティングでは「いまどのネームスペースの
スタックの話をしているのか」を常に意識する必要があります。

**発展: 抽象 UNIX ドメインソケット** — 分野02で見た UNIX ドメイン
ソケットのうち、ファイルシステム上のパス名で待ち合わせるものは
「ファイルの世界」の存在なので、ネットワークネームスペースの管轄外
です(マウントの眺めの話になります)。一方、パス名の代わりに抽象
名前空間(名前が `@` で始まる流儀)を使うものは、ネットワーク
ネームスペースごとに独立します。「どの資源がどのネームスペースの
管轄か」を考える良い練習問題です。

## 実行例 — 孤島を作り、配線し、ポートの衝突が消えるのを見る

Ubuntu Server 26.04 LTS 上での例です。まず新品のネームスペースが
「配線前の新品マシン」であることを確認します。

```console
$ sudo ip netns add ns1
$ sudo ip netns exec ns1 ip addr show
1: lo: <LOOPBACK> mtu 65536 qdisc noop state DOWN ...
                          ← デバイスは lo だけ。しかも DOWN(127.0.0.1 すら使えない)
$ sudo ip netns exec ns1 ip route show
$                         ← ルーティング表も空
```

veth ペアで配線します。

```console
$ sudo ip link add veth0 type veth peer name veth1
$ sudo ip link set veth1 netns ns1          ← 片割れを ns1 へ移す
$ sudo ip addr add 192.0.2.1/24 dev veth0
$ sudo ip link set veth0 up
$ sudo ip netns exec ns1 ip addr add 192.0.2.2/24 dev veth1
$ sudo ip netns exec ns1 ip link set veth1 up
$ sudo ip netns exec ns1 ip link set lo up  ← lo は自分で起こす
$ ping -c 1 192.0.2.2
64 bytes from 192.0.2.2: icmp_seq=1 ttl=64 time=0.041 ms
                          ← 物理配線のない「2台のマシン」が疎通した
```

ポート番号の空間が独立していることを、同じ 8080 番の2重 listen で
確かめます。

```console
$ python3 -m http.server 8080 &             ← ホスト側で 8080 を listen
$ sudo ip netns exec ns1 python3 -m http.server 8080 &
$                         ← 同じ 8080 で bind に成功(ホストなら EADDRINUSE のはず)
$ ss -tln | grep 8080
LISTEN 0  5  0.0.0.0:8080  0.0.0.0:*
$ sudo ip netns exec ns1 ss -tln | grep 8080
LISTEN 0  5  0.0.0.0:8080  0.0.0.0:*
                          ← ss の見え方もネームスペースごと(/proc/net が別物)
```

所属の識別子(inode 番号)も見ておきます。

```console
$ readlink /proc/$$/ns/net
net:[4026531840]          ← ホスト(init_net)の識別子
$ sudo ip netns exec ns1 readlink /proc/self/ns/net
net:[4026532333]          ← 別の番号 = 別のネームスペース
$ ip netns list
ns1
$ ls /run/netns/
ns1                       ← ip netns add が作った bind mount の実体
```

## トラブルシューティング

- **ネームスペースの中で 127.0.0.1 にすら接続できない**: 新品の
  ネームスペースでは **lo も DOWN で始まります**。ローカルの
  デーモン間通信・テストが全滅するので、`ip link set lo up` を
  忘れないこと(コンテナランタイムは自動でやってくれるため、
  手組みのときだけ踏む古典です)
- **veth を作って IP も振ったのに疎通しない**: 確認は3点セット
  です——両端とも up にしたか(片方だけ up では LOWER_UP に
  ならない)、相方は意図したネームスペースにいるか(`ip link set
  ... netns` を忘れて両端ともホストにいる)、そして同じサブネットの
  アドレスか
- **ネームスペースの中から外の世界に出られない**: 通し稽古の
  各段を順に疑います。(1) ns 内にデフォルトルートがあるか
  (`ip netns exec ns1 ip route`)、(2) ホストの
  `net.ipv4.ip_forward` が 1 か(**sysctl はネームスペースごと**。
  ホスト側で確認)、(3) masquerade のルールがあるか、(4) ホストの
  forward の検問所で落とされていないか。前章の counter・nftrace が
  ここでも武器になります
- **名前解決だけ失敗する**: `/etc/resolv.conf` は**ファイルの
  世界**の存在で、ネットワークネームスペースを分けても共有された
  ままです(ホストの設定が 127.0.0.53 のスタブリゾルバを指して
  いると、ns 内にはそのリゾルバがいないため死にます)。
  `ip netns exec` には `/etc/netns/<名前>/resolv.conf` があれば
  `/etc/resolv.conf` に重ねてマウントする規約があり、これで
  ネームスペース専用の設定を与えられます
- **コンテナのネームスペースが `ip netns list` に出ない**:
  `ip netns` が管理するのは `/run/netns/` の bind mount の分だけ
  です。コンテナランタイムは独自の場所で管理するため、ここには
  現れません。プロセスから入るのが確実です——`lsns -t net` で
  一覧し、`nsenter -t <PID> -n <コマンド>` で対象プロセスの
  ネームスペースに入って観察します
- **ホストの input チェーンで遮断したのに、コンテナ宛の通信が
  通ってしまう**: 通し稽古のとおり、ネームスペース宛のパケットは
  ホストのスタックでは**転送**であり、input ではなく **forward の
  検問所**を通ります。「ホスト自身への通信」を守るルールは
  コンテナには効きません。コンテナ管理ツールが自前の forward
  ルールを差し込むことと合わせて、事故の多い領域です(方針の
  設計は分野07で扱います)
- **conntrack の台帳に期待したエントリがない**: 台帳も
  ネームスペースごとです。ホストで `conntrack -L` を打っても
  ns 内で完結したフローは見えません。`ip netns exec ns1
  conntrack -L` のように、**見たいスタックの中で**観察コマンドを
  実行します(ss、ip route、nft list ruleset もすべて同様)

## 演習・確認問題

1. ネットワークネームスペースが複製するものを5つ以上挙げ、
   それぞれが前2章のどの仕組みに対応するかを対応づけてください。
   また「ルーティング表だけ共有する」といった部分共有が提供されて
   いない理由を、前章の DNAT・conntrack とルーティング判断の関係から
   説明してください
2. 新しいネームスペースを作った直後の状態(デバイス、ルーティング表、
   ルールセット)を述べ、そこから外部と通信できるようになるまでに
   必要な配線・設定を順に挙げてください
3. veth ペアの送信処理は、物理 NIC のドライバと何が違いますか。
   「ns1 から ns2 へ veth 越しに送った1つのパケットが通る検問所」を
   すべて、通る順に挙げてください
4. プロセスが setns で別のネームスペースへ移動したとき、移動前から
   開いていたソケットはどうなりますか。その設計の理由を「fd と
   通信の実体の所属の違い」から説明してください
5. 中にプロセスが1つもいないネームスペースが破棄されずに残るのは
   どういう場合ですか。`ip netns add` が作るネームスペースが
   それに該当する仕組みを説明してください
6. コンテナ宛の通信をホストの nftables で制御したいとき、input と
   forward のどちらのフックにルールを書くべきですか。パケットの
   道筋を根拠に説明してください

## まとめ

- ネットワークネームスペースは、スタック一式(デバイス・ルーティング
  表・検問所・conntrack・ソケット/ポート空間・sysctl)を1単位として
  複製する仕組み。実体は `struct net` で、ホスト本来のスタックも
  その1インスタンス(init_net)にすぎない
- 新品のネームスペースは配線前の孤島。veth ペア(仮想の直結
  ケーブル)で境界をまたぎ、ブリッジ(カーネル内 L2 スイッチ)で
  束ね、フォワーディング+masquerade で外へ出す——コンテナの
  ネットワークはこの組み合わせの自動化にすぎない
- 所属は task_struct → nsproxy → struct net とたどられ、fd /
  /proc/PID/ns/net / bind mount が参照(=寿命)を握る。ソケットは
  作成時のネームスペースに固定される
- 1つのパケットは複数のネームスペースの検問所を順に通る。観察
  コマンド(ss / ip / nft / conntrack)は「見たいスタックの中で」
  実行するのが調査の大原則
- スタックの複製と配線はできた。次章は、送信の道のりで予告した
  最後の登場人物——パケットを**どの順で・どれだけの速さで**送り出す
  かを決める tc / qdisc——を扱う
