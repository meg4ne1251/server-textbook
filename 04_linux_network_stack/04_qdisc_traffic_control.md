# tc / qdisc — パケットスケジューリングと「賢い行列」

## 概要

分野04の最終章。前々々章の「送信の道のり」で予告した最後の登場人物
——IP 層とドライバの間に挟まる**キューイング層(qdisc)**を扱います。
どのパケットを先に出すか(順番)、どれだけの速さで出すか(帯域)、
どれだけ並ばせるか(遅延)を采配する、パケット版のスケジューラです。
前提知識は本分野の前3章と、分野02のスケジューラの章
(`02_process_kernel/04_scheduler.md`)。基準環境は Linux 7.0 /
Ubuntu Server 26.04 LTS。

## 導入 — 行列があるところにしか、采配の余地はない

前々々章の送信の道のりを思い出してください。write されたデータは
ソケット層 → TCP 層 → IP 層と降りて、最後に **NIC ごとの送信待ち
行列**に並んでから電線に出ていきました。そもそも、なぜここに行列が
できるのでしょうか。

理由は速度の差です。CPU がパケットを作る速さは NIC が送り出す速さと
無関係で、しかも**バースト的**です——アプリケーションは静かにして
いたかと思うと、突然 1 MB を一気に write します。一方 NIC は、
リンク速度で決まった一定のペースでしか送れません。作る側が瞬間的に
速く、出す側が一定なら、間に行列ができるのは必然です。

ここで発想を切り替えます。行列は我慢の場所ではなく、**唯一の采配の
場所**です。パケットがひとたび NIC を出てしまえば、もうカーネルに
できることはありません。逆に行列に並んでいる間なら、選べます——

1. **順番** — 誰を先に出すか。バルク転送の大群の後ろで SSH の
   1パケットを待たせるのか、割り込ませるのか
2. **速度** — どれだけの速さで出すか。リンクは 1 Gbps でも、
   「この仮想マシンには 100 Mbps まで」と絞りたいことがある
3. **長さ** — どれだけ並ばせるか。そして満杯になったら、
   誰を諦める(捨てる)か

「限りある資源(送信機会)を、待っている者たちにどう配るか」——
これは分野02のスケジューラの章で解いた問題そのものです。実際、
これから見る道具立て(重み付き公平分配、階層、貸し借り)は CPU
スケジューラと同族で、あの章で「パケット側の系譜と分野04で再会する」
と予告した回収がこの章です。

ただし、パケットには CPU のタスクと違う事情が2つあります。

- **横取り(プリエンプション)がない。** タスクは走行中でも中断して
  交代させられますが、送信し始めたパケットは最後まで送るしか
  ありません。采配の手段は「行列の中での並べ替え」だけです
- **捨てることが正当な道具である。** タスクを勝手に kill する
  スケジューラは論外ですが、パケットの世界では違います。TCP は
  損失を「速度を落とせ」の合図として使う(前々々章)ので、
  **賢く捨てることは、送信元に速度を伝える正規の通信手段**なのです。
  この視点は後半(bufferbloat)で主役になります

## 理論

本節の内容は、主に **`man 8 tc`** とそのサブページ群
(`man 8 tc-fq_codel`、`man 8 tc-htb`、`man 8 tc-tbf`、
`man 8 tc-netem`)、RFC 8289(CoDel)/ RFC 8290(fq_codel)、
および Linux カーネルソース(net/sched/、net/core/dev.c)に
基づきます。

### qdisc — 機構と方針の分離、パケット版

この行列の仕組みを、Linux は **qdisc(queueing discipline /
キューイング規律)**と呼びます。カーネル本体と qdisc の間の契約は
拍子抜けするほど小さく、本質的には2つの操作だけです。

- **enqueue(skb)** — このパケットを預かれ
- **dequeue()** — 次に送るべきパケットを1つよこせ

カーネル本体(送信経路)は「預けて、送信機会が来たら1つ受け取って
ドライバに渡す」ことしか知りません。**預かってから渡すまでの間に
何をするか——並べ替えるか、遅らせるか、捨てるか——は qdisc の
自由**です。これは分野02で見た「機構(context_switch)と方針
(pick_next_task)の分離」のパケット版であり、VFS の操作表
(分野03)と同じ、関数ポインタによる差し替え可能な契約です。
そして方針を設定するユーザーランドのコマンドが **tc**
(traffic control。iproute2 に含まれる)——LVM で見た「方針は
ユーザーランド、機構はカーネル」の分業がここでも貫かれています。

qdisc は大きく2種類に分かれます。

- **クラスレス qdisc** — 内部に構造を持たない1個の規律。
  素朴な FIFO(pfifo)、優先度付き FIFO(pfifo_fast)、帯域制限
  (tbf)、公平キューイング(fq_codel)、遅延・損失の注入
  (netem)など
- **クラスフル qdisc** — 内部に**クラス**という子の枠を持ち、
  クラスの下にさらに qdisc をぶら下げられるもの(htb、prio 等)。
  どのパケットをどのクラスに入れるかは **filter(分類器)**が
  決める。全体として「根 → クラス → 子 qdisc」の**木**になる

```
            root qdisc(htb 1:)
              │
      ┌───────┴───────┐
   クラス 1:10      クラス 1:20      ← filter が振り分ける
  (保証 800Mbps)   (保証 200Mbps)
      │               │
   fq_codel        fq_codel          ← 葉にはまた qdisc
```

NIC 1枚には必ず根(root)の qdisc が1つ付いています。歴史的な
既定は **pfifo_fast**(TOS ビット由来の優先度で3レーンに分けた
FIFO。長さの上限は txqueuelen で既定 1000 パケット)でしたが、
現在の既定は後述の **fq_codel** です(`net.core.default_qdisc`
sysctl。systemd が設定ファイルで fq_codel に変更しており、Ubuntu も
これに従う——実機での確認は実行例で)。

### 帯域を配る — トークンバケットと HTB

「速度を絞る」の標準原理が**トークンバケット(token bucket)**です。

- バケツにトークンが**一定レートで**補充される(rate)
- バケツには深さがあり、あふれた分は捨てられる(burst)
- パケットを1つ送るには、そのバイト数分のトークンが必要。
  足りなければ、貯まるまでパケットは行列で**待つ**

長期的な送信速度は補充レートを超えられません。一方、しばらく
静かにしていればトークンが貯まり、その分だけ**瞬間的なバースト**は
許される——「平均は厳格に、瞬間は寛容に」を1つの単純な機構で
実現します。クラスレスな実装が tbf(token bucket filter)です。

超過分を「待たせる」やり方を**シェーピング(shaping / 整形)**、
待たせずに「捨てる」やり方を**ポリシング(policing / 取り締まり)**と
呼びます。シェーピングは行列が必要なので**送信側でしかできません**
(受信は後述)。

階層が欲しくなったときの道具が **HTB(Hierarchical Token
Bucket)**です。クラスごとに2つの値を持ちます。

- **rate** — 保証帯域。混雑していても、このクラスはここまで使える
- **ceil** — 天井。**親に余りがあれば**、ここまで借りられる

「全体 1 Gbps を、Web に 800 Mbps・バッチに 200 Mbps 保証。ただし
相手が使っていないときは互いに借りてよい」——余った資源を遊ばせず、
必要なときは保証を守る。この「保証+貸し借り」は、分野02の
nice/weight による CPU 時間の重み付き分配の帯域版です(兄弟クラス間の
借用配分も、公平キューイングと同じラウンドロビンで行われます)。

### 公平に混ぜる — 公平キューイング(分野02の再会)

帯域の上限を決めても、まだ「順番」の問題が残ります。1本の FIFO
では、大きなファイル転送が数百パケットを連続で積んだ後ろに、SSH の
キー入力1パケットが並ぶ——CPU の世界で「長い計算が対話タスクを
待たせる」のと同じ図です。

**公平キューイング(fair queueing)**の答えは、**フロー(おおむね
前々々章の4つ組)ごとに行列を分け、行列の間をラウンドロビンで
巡回する**ことです。1巡につき各フローから quantum バイト
(既定はほぼ MTU 1個分)ずつ取り出す方式(DRR: Deficit Round
Robin)により、パケット数ではなくバイト量で公平を測ります。
大食いのフローは自分の行列だけが長く伸び、**遅延も損失もまず
自分に降りかかる**。小さな対話フローは、他人の行列の長さと無関係に
毎巡すぐ順番が来ます——nice 値のいらない自動の公平です。

分野02で予告した系譜の再会がここにあります。EEVDF の原論文
(Stoica & Abdel-Wahab, 1995)はもともと資源共有一般の理論で、
その源流はネットワークの公平キューイング研究にあります。CPU も
パケットも、「重み比で帯域を分ける」という同じ問題の別の顔なのです。

### 満杯の哲学 — bufferbloat と CoDel

最後に「長さ」の問題です。直感的には、バッファは大きいほど
親切に思えます——捨てずに済むのだから。この直感が大規模に間違って
いたことが2010年代に広く認識され、**bufferbloat(バッファ
ブロート)**と名付けられました(Gettys らによる問題提起。
CACM 2012)。

理由は TCP の設計にあります。TCP は**損失を「回線が一杯だ」の
合図として速度を調整します**。バッファが深いと、回線が一杯に
なっても損失がなかなか起きず、送信元は速度を上げ続け、バッファは
満タン近くで安定してしまう。スループットは 1 bit も増えないのに、
**すべてのパケットが満杯の行列を律儀に通過する**ようになり、遅延
だけが恒常的に膨らみます。具体的な算数: txqueuelen 1000(既定)が
1500 バイトのパケットで満たされると 1.5 MB。1 Gbps なら 12 ms
ですが、10 Mbps のアップリンクでは **1.2 秒**です。アップロードを
1本始めた瞬間に SSH も DNS も1秒待たされる——家庭の回線からデータ
センターまで、いたるところで起きていた病気です。

解決には「いつ捨てるか」の再発明が必要でした。**CoDel(Controlled
Delay。RFC 8289)**の洞察は、**行列の長さではなく、パケットの
滞留時間(enqueue から dequeue までの時間)を測る**ことです。

- 滞留時間の最小値が **target(5 ms)** を **interval(100 ms)**
  にわたって下回らなかったら、「これは瞬間のバーストではなく
  恒常的な滞留だ」と判定して、パケットを捨て始める
- 捨てるのは行列の**先頭**から(最も古く並んだもの)。損失の合図が
  送信元に最速で届く
- 改善するまで、捨てる間隔を徐々に詰めていく

バーストは容認し(target を短時間超えるのは正常)、恒常的な滞留
だけを罰する。「捨てることは失敗ではなく、送信元への合図」を機構に
した qdisc です。

そして現在の既定 **fq_codel(RFC 8290)**は、この2つの合成です
——フローごとの行列(公平キューイング。既定 1024 本、ハッシュで
振り分け)を DRR で巡回し、**各行列に CoDel を適用する**。混ぜる
公平と、遅延の抑制を同時に手に入れた「ほぼ全員にとって正しい既定」
であり、systemd がわざわざ既定値を変更している理由です。

**発展: fq(sch_fq)** — 名前の似た別の qdisc として fq があります。
こちらは TCP 自身が計算する**ペーシング速度**(バーストせず等間隔で
送る)を執行する送信側専用の設計で、BBR 輻輳制御とともに使われ
ます。輻輳制御の詳細は本書の範囲外です(前々々章の TCP と同じ
線引き)。

## 内部動作の詳細

### dev_queue_xmit — 誰の CPU 時間で送るのか

送信経路の本丸は net/core/dev.c の `__dev_queue_xmit()` です。
IP 層がルーティングを終えた skb はここに来ます。

```
  送信プロセスのカーネル時間                     softirq(NET_TX)
  ┌────────────────────────────────┐
  │ q->enqueue(skb)   ← 預ける      │
  │ __qdisc_run(q):                │      送信完了割り込み /
  │   loop:                        │      quota 切れの続きは
  │     skb = q->dequeue()         │  ←── ksoftirqd などが
  │     ドライバの xmit(リングへ)  │      同じループを回す
  └────────────────────────────────┘
```

注目すべきは、**enqueue した本人が、続けて dequeue のループまで
回す**ことです。write したプロセスは、自分のカーネル時間で qdisc
から取り出し、ドライバに渡すところまで働いてから復帰します。この
とき取り出されるのは自分のパケットとは限りません——**たまたま
行列の先頭にいた誰かのパケット**です(公平の采配は qdisc が
済ませているので、運び役は誰でもよい)。

ループが終わる条件は主に3つ: 行列が空になった、NIC(リング)が
満杯だった、**quota(dev_tx_weight。既定 64 パケット)を使い
切った**。後の2つの場合、続きは **NET_TX softirq** に予約されます
——前々々章で `/proc/softirqs` に見えていた NET_TX の正体は、この
「送信の続き」です。NIC が送信完了の割り込みを上げてリングに空きが
できたときも、同じく NET_TX 経由で qdisc の続きが汲み出されます。

なお root qdisc はロックで守られており、多コアが1つの行列に殺到
すると競合します。複数の送信キューを持つ現代の NIC では、既定で
**mq** という「器」の qdisc が付き、**ハードウェアの送信キューごとに
独立した fq_codel** をぶら下げてこの競合を避けます(`tc qdisc show`
で mq の下に fq_codel が並ぶのはこの構成です)。

### noqueue — 詰まらないデバイスに行列は要らない(前章の回収)

前章の実行例で、ネームスペース内の lo が `qdisc noop`(DOWN 時)と
表示されていました。veth や lo を up にすると、今度は **noqueue** と
表示されます——**行列を持たない**という選択です。

理由は前章の veth の内部を思い出せば明らかです。veth の送信処理は
「相方の受信キューに積んで NET_RX を予約する」だけで、**失敗
しません**。物理 NIC のような「電線が一定速度でしか流せないから
待つ」という状況が原理的にないので、行列も采配も不要なのです。
「行列は速度差があるところにできる」の裏返しです。

運用上の帰結が1つあります。コンテナの帯域を絞りたければ、noqueue の
ままでは絞れません——veth に明示的に tbf や htb を付ける(または
ホスト側の物理 NIC の egress で絞る)必要があります。コンテナ
ランタイムの帯域制限機能の実体は、まさにこの自動化です(分野05)。

### qdisc の下の隠れた行列 — リングバッファと BQL

qdisc の采配には1つ弱点があります。dequeue されてドライバの
**リングバッファ**(前々々章の受信側で見たものの送信版。数百〜数千
記述子)に入ったパケットは、もう並べ替えられません。リングが深々と
埋まっていると、fq_codel がどれだけ賢く順番を決めても、その先に
「馬鹿な FIFO」が控えている構図になります——**采配が効くのは、
行列が方針の見える場所(qdisc)にできるときだけ**です。

これを塞ぐのが **BQL(Byte Queue Limits)**です(Linux 3.3 以降。
lib/dynamic_queue_limits.c)。「リンクを遊ばせないために必要な
最小限のバイト数」を送信完了の実績から動的に見積もり、それ以上を
リングに入れずに qdisc 側に留めます。行列は賢い側に溜まり、
リングは常に浅く保たれる。sysfs の
`/sys/class/net/<dev>/queues/tx-*/byte_queue_limits/` で現在の
見積もりを観察できます。

### 入口側 — ingress と、tc / netfilter の境界(発展)

ここまですべて送信(egress)の話でした。受信側を制御できないのには
本質的な理由があります——**パケットがあなたの NIC に届いた時点で、
回線の帯域はもう消費済み**だからです。届いたものを行列に並べても、
過去は変わりません。できることは2つだけです。

- **ポリシング** — 超過分を受信直後に捨てる。TCP なら損失が合図に
  なり、送信元が**間接的に**減速する
- **ifb への迂回** — ingress で受けたパケットを ifb という仮想
  デバイスへリダイレクトし、その **egress** として shaping の顔に
  変換する迂回路

このための取り付け点が **ingress qdisc(現代は clsact)**で、
場所はドライバがパケットをスタックに上げた直後
(`__netif_receive_skb_core`)——前章までの言葉で言えば、
**prerouting よりさらに手前**です。前々章の発展で触れた netfilter の
ingress フックと同じ「ドライバとスタックの境界」に同居しており
(評価は tc が先)、tcpdump が覗くのはそのさらに手前です。なお、
この tc の分類器・アクションの枠組みは eBPF プログラムの主要な
取り付け点の1つでもありますが、本書では深追いしません。

## 実行例 — 既定を眺め、遅延を注射する

Ubuntu Server 26.04 LTS 上での例です。まず既定の姿を確認します。

```console
$ sysctl net.core.default_qdisc
net.core.default_qdisc = fq_codel   ← systemd による既定の変更
$ tc qdisc show dev enp1s0
qdisc mq 0: root
qdisc fq_codel 0: parent :1 limit 10240p flows 1024 quantum 1514
  target 5ms interval 100ms memory_limit 32Mb ecn drop_batch 64
                          ← mq の下に送信キューごとの fq_codel。
                             target 5ms / interval 100ms が CoDel の値
$ tc qdisc show dev veth0
qdisc noqueue 0: root refcnt 2  ← veth は行列なし(詰まらないデバイス)
```

統計は `-s` で見えます。采配の痕跡(drops、backlog)の観察窓です。

```console
$ tc -s qdisc show dev enp1s0
qdisc fq_codel 0: parent :1 ...
 Sent 123456789 bytes 98765 pkt (dropped 12, overlimits 0 requeues 3)
 backlog 0b 0p                  ← いま並んでいる量。恒常的に大きければ詰まり
```

netem で送信側に 100 ms の遅延を注射し、qdisc が本当に送信の道の
上にいることを確かめます(**実験用の回線で行うこと**。SSH 接続
中の NIC に掛けると自分も 100 ms 遅くなります)。

```console
$ ping -c 2 192.0.2.2 | tail -1
rtt min/avg/max/mdev = 0.412/0.435/0.458/0.023 ms
$ sudo tc qdisc replace dev enp1s0 root netem delay 100ms
$ ping -c 2 192.0.2.2 | tail -1
rtt min/avg/max/mdev = 100.398/100.437/100.476/0.039 ms
                          ← 行きだけ +100ms(egress にしか効かない)
$ sudo tc qdisc del dev enp1s0 root   ← 後始末。既定(fq_codel)に戻る
```

tbf で帯域を絞ると、トークンバケットの動きが数字で見えます。

```console
$ sudo tc qdisc replace dev enp1s0 root tbf rate 10mbit burst 32kbit latency 50ms
$ tc -s qdisc show dev enp1s0
qdisc tbf 8001: root rate 10Mbit burst 4Kb lat 50ms
 Sent 52428800 bytes 34952 pkt (dropped 861, overlimits 40213 requeues 0)
                          ← overlimits = トークン切れで待たせた回数、
                             dropped = latency 相当を超えて諦めた分
$ sudo tc qdisc del dev enp1s0 root
```

## トラブルシューティング

- **ダウンロード側の帯域制限が効かない**: shaping は egress 専用
  です。受信を絞りたい場合は、(1) 相手側の egress で絞る(正攻法)、
  (2) ingress のポリシングで捨てて間接的に減速させる、(3) ifb へ
  リダイレクトして egress 化する、のいずれかになります。「received
  済みのものは制御できない」が原則です
- **tbf を付けたらほぼ何も通らなくなった**: burst がパケット
  サイズより小さいと、そのパケットは永遠にトークンが足りません。
  burst は最低でも MTU 以上(実務では rate に応じて余裕を持って)
  必要です。また GSO(前々々章の発展)で数十 KB の塊が流れてくる
  環境では、会計が粗くなることも意識してください
- **コンテナ / veth に制限を掛けたはずが効かない**: veth の既定は
  noqueue で、qdisc を付ける先を間違えやすい急所です。「どちらの
  ネームスペースの、どちらの端に付けたか」を確認します(前章の
  大原則——観察も設定も、見たいスタックの中で)。veth の egress は
  「コンテナから出る向き」と「コンテナへ入る向き」が端によって
  逆になる点にも注意
- **アップロード中だけ SSH がもたつく**: bufferbloat の古典的
  症状です。`tc -s qdisc` で backlog が恒常的に大きくないか、
  root が古い pfifo_fast のままでないかを確認します。fq_codel なら
  対話フローは別の行列に入るため、この症状は大きく緩和されます
- **tc -s の dropped が増えているが異常か**: 必ずしも異常では
  ありません。CoDel の drop は「送信元への正規の合図」であり、
  設計どおりの動作です。問題なのは drop の有無ではなく、遅延
  (backlog の滞留)やスループットが要件を満たしているかです
- **実験用の netem を消し忘れて「ネットワークが遅い」**: qdisc は
  再起動まで残ります。原因不明の遅延・損失では、`tc qdisc show` で
  netem や tbf が残っていないかを疑ってください(`tc qdisc del
  dev <dev> root` で既定に戻ります)
- **HTB で分類されないトラフィックが制限をすり抜ける**: htb の
  `default` パラメータを指定しないと、どの filter にも一致しない
  パケットは**どのクラスにも入らず素通し**で送信されます。
  「絞ったつもりの通信が全速で流れる」ときは、まず default 指定と
  filter の一致条件を確認します

## 演習・確認問題

1. qdisc の enqueue / dequeue という契約は、分野02のスケジューラに
   おける「機構と方針の分離」とどう対応しますか。また、CPU の
   タスクにはあってパケットにはない手段(横取り)と、パケットには
   あってタスクにはない手段(破棄)が、それぞれの采配の設計を
   どう変えているか説明してください
2. トークンバケットの rate と burst は、それぞれ長期・短期の
   何を決めますか。シェーピングとポリシングの違いと、シェーピングが
   egress でしかできない理由を述べてください
3. 公平キューイング(fq_codel)は、nice 値のような明示的な優先度
   設定なしに、なぜバルク転送から対話フローを守れるのですか。
   フローごとの行列と DRR の観点から説明してください
4. bufferbloat はなぜ「バッファを大きくした親切」から生まれるの
   ですか。TCP の輻輳制御との関係で説明し、CoDel が行列の長さでは
   なく滞留時間を判定基準に選んだ理由を述べてください
5. write したプロセスが、他人のパケットをドライバに渡す仕事まで
   してから復帰することがあるのはなぜですか。また NET_TX softirq が
   出動するのはどんなときか、3つの条件を挙げてください
6. veth の既定 qdisc が noqueue である理由を、veth の送信処理の
   実装(前章)から説明してください。その veth に帯域制限を
   掛けたい場合、何をどこに付ける必要がありますか

## まとめ

- qdisc は IP 層とドライバの間に挟まるパケットスケジューラ。
  enqueue / dequeue という小さな契約(機構)の中身を tc で
  差し替える(方針)——CPU スケジューラ・VFS と同じ分離の型
- 帯域はトークンバケット(rate / burst)で絞り、HTB が保証
  (rate)と貸し借り(ceil)の階層を作る。順番は公平キューイングが
  フローごとの行列と DRR で自動的に守る
- 深すぎる FIFO は損失の合図を遅らせ、遅延だけを膨らませる
  (bufferbloat)。CoDel は滞留時間で判定して先頭から捨てる——
  「捨てることは失敗ではなく、送信元への合図」。既定の fq_codel は
  公平キューイングと CoDel の合成
- 送信の実働は enqueue した本人のカーネル時間で行われ、続きは
  NET_TX softirq が引き継ぐ。行列が采配の見える場所にできるよう
  BQL がリングを浅く保ち、詰まらない仮想デバイス(veth)は
  そもそも行列を持たない(noqueue)
- 本分野の4章で、ソケットという窓口(01)、通り道の検問所(02)、
  スタックの複製(03)、出口の行列(04)が揃った。次の分野05では、
  ネームスペースに cgroups を加えて「コンテナ」という1つの答えに
  組み上げる
