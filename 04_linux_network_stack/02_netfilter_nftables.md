# netfilter / nftables — パケットの検問所

## 概要

前章で追った「パケットの道のり」の途中には、検査と書き換えのための
決まった検問所が置かれています。この章では、その検問所の枠組みである
**netfilter** のフック機構と、そこに載る現行のルール処理エンジン
**nftables** を扱います。前提知識は前章(送受信の道のり、sk_buff)です。
基準環境は Linux 7.0 / Ubuntu Server 26.04 LTS。ファイアウォールの
運用(方針設計・堅牢化)は分野07、NAT やルーティングのプロトコル上の
意味は姉妹書 `network-guide` が主で、本章はカーネルの機構を受け持ちます。

## 導入 — 配達人に検査の仕事を足す

前章のカーネルは、忠実な配達人でした。write されたバイト列をパケットに
仕立てて送り出し、届いたパケットを4つ組で表引きしてソケットに配達する。
どのパケットも平等に、黙々と運んでいました。

しかし実際のサーバーで、無条件の配達は困ります。

1. **遮断したい。** sshd は動かしたいが、社外からの 22 番宛ては
   受け取りたくない。配達の前に「差出人と宛先を見て断る」仕事が要る
2. **書き換えたい。** 1つのグローバルアドレスを何台かで共用したい、
   届いた依頼を裏の別のマシンに横流ししたい——宛先や差出人を通過の
   途中で書き換える仕事(NAT)が要る
3. **記録したい。** 誰が何回断られたかを数え、必要なら記録に残したい

問題は、**この仕事をどこでやるか**です。アプリケーションに任せるのは
筋が悪い——sshd に届く前に断りたいのだし、アプリの数だけ検査の実装が
増えるのは論外です。かといって NIC ドライバに埋め込めば、ドライバの
数だけ実装が増えます。

Linux の答えは、**全パケットが必ず通る共通の道の要所に、固定の検問所を
設けておく**ことです。検問所そのものは「場所」にすぎず、普段は素通し
です。検査・書き換え・記録といった個々の機能は、必要になったときに
検問所へ**登録**します。

- **netfilter** — 検問所の枠組みそのもの。スタック内の5箇所に置かれた
  **フック(hook)**と、そこへ処理を登録する仕組み
- **nftables** — 検問所に立つ現行の審査官。「ルールの一覧」を受け取り、
  パケットごとに照合して通す/落とすを判定するエンジン
- **iptables** — 先代の審査官。長年の標準だったが世代交代した
  (歴史と理由は理論の節で)

日常語の「ファイアウォール」は、この仕組みの上に実現される機能の
名前です。ufw や firewalld といった管理ツールも、最終的には nftables の
ルールを書いているにすぎません(分野07で扱います)。

## 理論

本節の内容は、主に **netfilter プロジェクトの公式文書(netfilter.org)**、
`man 8 nft`、`man 8 iptables`、Linux カーネルソース
(include/linux/netfilter.h、net/netfilter/)、およびカーネル
ドキュメント(Documentation/networking/nf_conntrack-sysctl.rst)に
基づきます。netfilter のフック機構は Linux 2.4(2001年)で導入され、
nftables は Linux 3.13(2014年)で追加されました。

### 5つのフック — 道のりのどこに検問所があるか

パケットがカーネルを通る道筋は、3通りしかありません。
**自分宛てに届く**(受信)、**自分から出ていく**(送信)、そして
**通り抜けていく**(転送。ルータとして働く場合)です。netfilter の
5つのフックは、この3つの道筋を過不足なく見張れる位置に置かれています。

```
                受信・転送の入口                転送
  NIC ──→ [prerouting] ──→ ルーティング判断 ──→ [forward] ──→ [postrouting] ──→ NIC
                                │                                  ↑
                             宛先が自分                             │
                                ↓                                  │
                             [input]                               │
                                ↓                                  │
                         ローカルプロセス                           │
                                ↓                                  │
                            [output] ──→ ルーティング判断 ──────────┘
```

- **prerouting** — 届いたパケットすべてが最初に通る(宛先が自分か
  他人かを判断する**前**)
- **input** — 宛先が自分だと決まったパケットが、ソケットへの配達前に通る
- **forward** — 通り抜けるパケットだけが通る
- **output** — 自分のプロセスが生んだパケットが、送信経路の最初に通る
- **postrouting** — 出ていくパケットすべて(転送も自分発も)が最後に通る

配置の鍵は**ルーティング判断との位置関係**です。たとえば「80番宛てに
届いた依頼を裏のマシンに横流しする」(宛先の書き換え、DNAT)は、
ルーティング判断の**前**にやらなければ意味がありません——宛先を
見てから配達先を決めるのですから、書き換えは見られる前に済ませる必要が
あります。だから prerouting が要るのです。逆に差出人の書き換え(SNAT)は、
経路が決まった**後**の最後の瞬間、postrouting でやるのが自然です。
また input と forward が分かれているのは、「サーバー自身への通信」と
「通り抜ける通信」は別の方針で扱いたいのが普通だからです(自分宛ては
厳しく、転送はそもそも許可しない、など)。

検問所での判定は **verdict(評決)** と呼ばれ、代表は次の2つです。

- **ACCEPT** — 通す。道のりの次の段階へ進む
- **DROP** — 落とす。パケットは**黙って**破棄され、差出人には何も
  通知されない(この「黙って」が運用上きわめて重要です——
  トラブルシューティングで回収します)

1つの検問所には、複数の処理が**優先度順**に並べます。同じ input を、
コネクション追跡(後述)が先に見て、フィルタのルールが後に見る——
という順序は優先度の数値(小さいほど先)で決まっています。フックは
あくまで「場所」であり、そこで何をするかは登録した側の自由です。

### コネクション追跡 — 台帳が「ステートフル」を可能にする

パケットを1個ずつ孤立して見る検査(ステートレス)には、根本的な
弱点があります。「**内から外への通信は自由にさせ、外から内へは
その応答だけを通したい**」——ほぼすべてのサーバーが求めるこの方針を、
パケット単体の情報だけで書くのは至難だからです。届いた1個のパケットが
「こちらから始めた会話の返事」なのか「見知らぬ相手からの初手」なのかは、
**過去を覚えていなければ**区別できません。

そこで netfilter は、**コネクション追跡(connection tracking、通称
conntrack)**という台帳を持ちます。conntrack は prerouting と output
(=パケットが生まれる/入ってくる、2つの入口)に高い優先度で登録され、
全パケットを観察して「フロー」(前章の4つ組が中心)ごとのエントリを
ハッシュ表に記録します。以後のパケットは台帳と照合され、次の状態が
付与されます。

- **NEW** — 台帳にないフローの最初のパケット(会話の初手)
- **ESTABLISHED** — 台帳にあるフローの一員。**両方向のパケットが
  観測済み**の会話
- **RELATED** — 既存の会話から派生した新しいフロー(FTP のデータ
  接続、既存の通信に対する ICMP エラーなど)
- **INVALID** — 台帳と矛盾する(どの会話にも属せない)パケット

注意すべきは、これが **TCP の接続状態とは別物**だという点です。
conntrack の ESTABLISHED は「両方向の往来を見た」という意味しか
持たないので、接続の概念がない UDP のフローにも付きます(DNS の
問い合わせと応答は、応答が観測された時点で ESTABLISHED です)。

この台帳の上で、ステートフルファイアウォールの基本形はたった2行で
書けます——「ESTABLISHED / RELATED は通す」「NEW は方針に照らして
選別する」。そして性能面の含意が重要です:
**ルールの精査は最初の1通だけ、2通目からは台帳で顔パス**。
ルールが何百行あっても、確立済みの会話のパケットは先頭の
ESTABLISHED ルール1回の照合で通過します。

### NAT — 台帳に基づく書き換え

NAT(Network Address Translation)がなぜ要るか・アドレス設計として
どう使うかは `network-guide` の主題です。本書が扱うのはカーネル内の
機構で、要点は1つ——**NAT は conntrack の上に実装されている**ことです。

書き換えのルール(prerouting の DNAT / postrouting の SNAT)が評価
されるのは、フローの**最初の1パケット(NEW)だけ**です。決まった
変換は conntrack のエントリに刻まれ、以後のパケットは——**逆方向の
返事も含めて**——ルールを見ずに台帳どおり自動で書き換えられます。
返りのパケットの宛先を元に戻す逆変換を毎回ルールで書く必要がないのは、
台帳が「行きをこう変えたなら、帰りはこう戻す」を覚えているからです。
ここでも「精査は最初の1通、以後は台帳で顔パス」が貫かれています。

### iptables から nftables へ — 審査官の世代交代

長年の標準だった iptables には、構造的な限界が4つありました。

1. **世界がプロトコルごとに分立していた。** IPv4 は iptables、IPv6 は
   ip6tables、ARP は arptables、ブリッジは ebtables——ほぼ同じ機能の
   実装とルールを4通り維持する必要があった
2. **ルールは先頭から線形に評価される。** 1000 行のルールは、最悪
   毎パケット 1000 回の照合を意味する。「1000 個のアドレスを断る」の
   ような集合の照合が苦手だった(補助ツール ipset が別付けされた)
3. **マッチも動作もカーネルのコードだった。** 新しい条件で照合したく
   なるたび、カーネルモジュール(とユーザー空間の対応コード)を書いて
   組み込む必要があった
4. **更新の原子性が不完全だった。** ルールの一括入れ替えはテーブル
   単位の置き換えで、複数テーブルにまたがる変更は一瞬の隙間を生んだ

nftables の設計は、この4つへの直接の回答です。発想の転換は
**「カーネルには意味を持ち込まず、小さな仮想機械だけを置く」**こと
でした。カーネル側にあるのは「パケットのこの位置から n バイト読む」
「レジスタと比較する」「集合を引く」といった少数の汎用命令(式)を
評価するインタプリタだけです。「TCP の宛先ポートが 22 なら通す」と
いったルールの**意味**は、ユーザー空間の `nft` コマンドが命令列に
翻訳してカーネルへ送り込みます。

- 新しい照合条件は命令の組み合わせで表現でき、**カーネルの変更が
  不要**になった(3への回答)
- 命令は L3 プロトコルに中立で、**inet ファミリ**1つで IPv4/IPv6 を
  同時に扱える(1への回答)
- 集合(set)と対応表(map)が言語に内蔵され、要素数によらない
  ハッシュ/区間木の表引きで照合できる(2への回答)
- ルールセットの更新は1つのトランザクションとして**原子的に**
  適用される(4への回答)

現在の Ubuntu / Debian では、`iptables` コマンド自体が nftables
バックエンドへの互換層(iptables-nft)になっており、古い書式で
書いても実体は nftables の枠組みに載ります。

## 内部動作の詳細

### フックの機構 — nf_hook_ops と優先度の列

処理の登録は、`struct nf_hook_ops`(どのファミリの・どのフックに・
どの優先度で・どの関数を)をカーネルに渡すことで行います。パケットが
検問所に差しかかると、カーネルは登録済みの関数を優先度順に呼び、
返ってきた verdict に従います——NF_ACCEPT なら次の関数へ、NF_DROP
なら skb を解放してそこで終わりです(このほか、パケットを預かって
流れから抜くもの、ユーザー空間の判定に委ねる NF_QUEUE があります)。

ipv4 の主要な登録の優先度は次のとおりで、同じ検問所の中に
**「台帳→宛先書き換え→フィルタ→差出人書き換え」の順序**が
埋め込まれています。

| 優先度 | 登録者 | 仕事 |
|---|---|---|
| -200 | conntrack | 台帳と照合し、状態を skb に付与する |
| -100 | NAT(dstnat) | 宛先の書き換え(prerouting) |
| 0 | filter | いわゆるファイアウォールのルール |
| 100 | NAT(srcnat) | 差出人の書き換え(postrouting) |

前章の道のりに重ねると、受信側の prerouting / input は softirq の
文脈(NET_RX の処理中、IP 層でのヘッダ検証の直後とソケット表引きの
前)で走ります。つまり**検査のコストを払うのはパケットを受け取った
CPU であって、宛先のプロセスではありません**。フィルタのルールが
重ければ、その分だけ softirq の滞在時間が伸びます。なお、どの
検問所にも何も登録されていなければ、フックの通過コストは実質ゼロに
なるよう実装されています(素の状態のサーバーが遅くならない理由です)。

### nftables のオブジェクト — テーブル・チェーン・ルール

nftables の世界は3層でできています。

- **テーブル** — ルール一式を入れる名前空間。作るときに**ファミリ**
  (ip / ip6 / inet / arp / bridge / netdev)を決める。iptables と
  違って定義済みのテーブルは存在せず、**素の状態は完全に空**
  (=すべて素通し)
- **チェーン** — ルールの列。**ベースチェーン**は
  「フック・優先度・ポリシー」を宣言して検問所に登録されるもので、
  **通常チェーン**はどこにも登録されず、他のチェーンから jump で
  呼ばれるサブルーチン
- **ルール** — 照合条件と動作の列。最後まで一致しなければチェーンの
  **ポリシー**(accept / drop)が適用される

ベースチェーンの宣言 `type filter hook input priority filter;` は、
機構の言葉に訳せば「input フックに優先度 0 で nf_hook_ops を登録せよ」
です。iptables で固定されていた filter / nat / mangle という区分も、
nftables では単なる「よくある優先度の別名」にすぎません。

ルールの実体は、前述の仮想機械の命令(**式**)の列です。

```
  nft のルール:  tcp dport 22 ct state new accept

  カーネル内の命令列(概念図):
    [meta]    L4 プロトコルを読む → TCP でなければ不一致で終了
    [payload] トランスポートヘッダの先頭+2 から2バイトをレジスタへ
    [cmp]     レジスタを 22 と比較 → 違えば不一致で終了
    [ct]      conntrack の状態をレジスタへ
    [cmp]     NEW と比較 → 違えば不一致で終了
    [verdict] accept
```

`payload`(パケットの中身)、`meta`(入出力インタフェースや mark
などの属性)、`ct`(conntrack 台帳)といった「読む」式と、比較・
カウンタ・verdict がすべてです。この翻訳の様子は
`nft --debug=netlink` で実際に観察できます。

### set と map — 線形評価からの脱出

`tcp dport { 22, 80, 443 } accept` と書いたとき、nftables は3つの
ルールを作りません。3要素の**集合(set)**を1つ作り、「読んだ
ポート番号で集合を1回引く」命令にします。照合コストは要素数に
よらず、ほぼ一定です(内部表現は要素の性質に応じてハッシュ表や
区間木が選ばれ、`10.0.0.0/8` のような範囲も1要素として持てます)。

**map / vmap** はその発展で、「値→値」「値→verdict」の対応表です。
たとえば `dnat to tcp dport map { 80 : 10.0.0.10, 443 : 10.0.0.11 }`
のように、**分岐そのものを表引きに畳み込めます**。iptables で
ルールを並べて表現していた分岐が、データ構造1回の参照になる——
これが「1000 行の線形評価」への nftables の答えです。

### conntrack の内部 — 2つのタプルを持つエントリ

台帳のエントリ1件は、**方向ごとの2つのタプル**を核に持ちます。

```
  conntrack エントリ(1フロー分)
  ├─ original タプル: src=192.0.2.10:52814 → dst=203.0.113.5:22
  ├─ reply    タプル: src=203.0.113.5:22   → dst=192.0.2.10:52814
  ├─ 状態(NEW / ESTABLISHED …)と TCP なら詳細状態
  ├─ タイムアウト(残り時間。パケットが通るたび更新)
  └─ NAT 変換情報(このフローをどう書き換えるか)
```

どちらの方向のパケットが来ても引けるよう、両タプルがハッシュ表に
登録されます。NAT が働くフローでは reply タプルが**書き換え後の**
姿で登録され、これが「帰りの逆変換を自動で行う」機構の正体です——
返りのパケットは reply タプルで台帳に命中し、original の姿へ戻されて
から先へ進みます。

エントリには上限(`net.netfilter.nf_conntrack_max`)があり、
使われないエントリはタイムアウトで消えます(確立済み TCP は既定で
5日、UDP は30秒〜数分の桁。正確な既定値は
Documentation/networking/nf_conntrack-sysctl.rst 参照)。上限に
達すると新しいフローのパケットは**破棄**され、カーネルログに
`nf_conntrack: table full, dropping packet` が残ります——
大量の接続をさばく NAT 箱・ロードバランサでの古典的な障害です。

### 通し稽古 — SSH 接続1本が通る検問所

前章の道のりに検問所を重ねて、SSH 接続の最初の数パケットを
追ってみます(受け側サーバー視点、input に「established は accept、
22 番の new は accept」のルールがある前提)。

1. **SYN 到着** — prerouting: conntrack が台帳にないことを確認し、
   NEW の仮エントリを作る。ルーティング判断で「宛先は自分」。
   input: filter ルールに `tcp dport 22 ct state new accept` が
   一致して通過。TCP 層が SYN キューへ(前章)
2. **SYN+ACK 送信** — output: conntrack が original の逆向きを観測。
   postrouting を抜けて送信。この往復が観測された時点でエントリは
   ESTABLISHED になり、台帳に確定する
3. **以後の全パケット** — prerouting で台帳に命中し ESTABLISHED が
   付与され、input では先頭の `ct state established accept` に
   1回で一致。**22 番のルールまで到達すらしない**(顔パス)

ルールが千行に育っても、この構図は変わりません。精査のコストは
会話1本につき最初の1通だけです。

**発展: ingress / egress フック** — 5つのフックとは別に、netdev
ファミリはインタフェースごとの **ingress** フック(ドライバが
パケットを上げた直後、prerouting よりさらに手前)と egress フックを
持ちます。プロトコル処理が始まる前の最安の位置で落とせるため、
DDoS 対策の早期破棄などに使われます。この「ドライバとスタックの
境界」という場所は、次々章の tc / qdisc(`04_linux_network_stack/04`)
とも接する領域です。

## 実行例 — 検問所を作って顔パスを観察する

Ubuntu Server 26.04 LTS 上での例です。素の状態を確認してから、
最小のステートフルファイアウォールを組みます。

```console
$ sudo nft list ruleset
$                            ← 何も出ない = テーブルが存在しない(全部素通し)
$ sudo nft add table inet filter
$ sudo nft add chain inet filter input \
    '{ type filter hook input priority filter ; policy accept ; }'
$ sudo nft add rule inet filter input ct state established,related counter accept
$ sudo nft add rule inet filter input tcp dport 22 ct state new counter accept
```

SSH でしばらく操作してから、ルールセットを表示します。

```console
$ sudo nft list ruleset
table inet filter {
        chain input {
                type filter hook input priority filter; policy accept;
                ct state established,related counter packets 1274 bytes 154820 accept
                tcp dport 22 ct state new counter packets 2 bytes 128 accept
        }
}
```

counter の偏りが、理論の節の「顔パス」の直接の証拠です——22 番の
NEW ルールに当たったのは接続を張った瞬間の2パケットだけで、以後の
千数百パケットはすべて先頭の ESTABLISHED ルールで通過しています。

台帳そのものは `conntrack` コマンド(conntrack パッケージ)で
観察できます。

```console
$ sudo conntrack -L
tcp   6 431982 ESTABLISHED src=192.0.2.10 dst=203.0.113.5 sport=52814 dport=22
      src=203.0.113.5 dst=192.0.2.10 sport=22 dport=52814 [ASSURED] use=1
   ↑ 1行に2組のタプル(original / reply)。431982 は残りタイムアウト秒
```

## トラブルシューティング

- **接続が refused ではなくタイムアウトする**: 前章の伏線の回収です。
  listen する者がいなければカーネルが RST を返すので、即座に
  refused になります。**何も返らずタイムアウトするなら、途中の
  どこかの検問所が DROP している**疑いが濃厚です(DROP は黙って
  捨てる)。即座にエラーを返して断りたい場合のために、ICMP エラー
  等で通知する verdict として reject も用意されています
- **どのルールで落ちているか分からない**: 疑わしいルールに
  `counter` を挟んで増え方を見るのが第一歩です。決定打は追跡機能で、
  `meta nftrace set 1` を先頭に足して `nft monitor trace` を
  実行すると、パケット1個ごとの「どのチェーンのどのルールに
  どう判定されたか」の全行程が出力されます
- **ルールを変えたのに挙動が変わらない**: 既存の会話は conntrack の
  台帳で顔パスしている可能性が高いです(遮断ルールを足しても、
  先頭の ESTABLISHED accept が先に一致する)。台帳のエントリを
  `conntrack -D` で消すか、接続を張り直して NEW から再評価させます
- **policy drop にしたら自分自身の通信まで壊れた**: ループバック
  (127.0.0.1)への通信も output / input を通ります。ローカルの
  デーモン間通信まで検査対象になるので、drop ポリシーの
  チェーンには `iif lo accept` を先頭に置くのが定石です
- **リモートから設定変更して閉め出された**: 定石は3つ——
  ESTABLISHED accept を必ず先頭に(作業中の SSH は既存の会話)、
  変更は逐次 add ではなく `nft -f` の原子的な置き換えで、そして
  戻す手段(一定時間後に旧ルールへ戻すタイマー)を仕掛けてから
- **カーネルログに nf_conntrack: table full, dropping packet**:
  台帳が上限(nf_conntrack_max)に達し、新規フローが捨てられて
  います。現在数は `conntrack -C` で見えます。上限を上げるか、
  追跡が不要な通信(たとえば大量の DNS)を `notrack` で台帳から
  外すかを検討します
- **iptables と nft の二重管理で混乱する**: iptables-nft 経由の
  ルールと nft 直接のルールは、**別々のチェーンとして同じ検問所に
  登録され、両方が順に評価されます**。片方を見て「ルールはない」と
  判断すると事故になります。`iptables -V` でバックエンドを確認し、
  管理は一方に寄せます(ufw / firewalld との関係は分野07で)

## まとめ

- netfilter はスタックの要所5箇所(prerouting / input / forward /
  output / postrouting)に置かれた検問所の枠組み。フックは「場所」で
  あり、conntrack・NAT・フィルタといった機能が優先度順に登録される
- conntrack は全パケットを観察するフローの台帳。NEW / ESTABLISHED /
  RELATED / INVALID の状態がステートフルファイアウォールを可能にし、
  NAT の自動逆変換もこの台帳の上に実装される。精査は最初の1通だけ、
  2通目からは台帳で顔パス
- nftables はカーネル内に汎用の仮想機械だけを置き、ルールの意味は
  ユーザー空間で命令列に翻訳する設計。inet ファミリでの v4/v6 統一、
  set / map による定数時間の照合、原子的な更新が iptables の限界への
  回答となった
- DROP は黙って捨てる——「refused ではなくタイムアウト」は検問所を
  疑う合図。counter と nftrace が調査の道具になる
- ここまでで、1台のマシンのスタック(経路・検問所・台帳)を貫いた。
  次章は、このスタック一式を**まるごと複製して増やせる**仕組み——
  ネットワークネームスペース——を扱う
