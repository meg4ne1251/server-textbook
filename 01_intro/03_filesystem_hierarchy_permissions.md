# ファイル階層(FHS)とパーミッションの基礎

## 概要

この章では、Linuxのファイルの世界を支える2つの土台——「どのマシンでも `/etc` や
`/var` という同じ地図が使える理由(ファイル階層)」と「誰が何を読み書きできるかを
決める仕組み(パーミッション)」——を扱います。前提知識は前2章(カーネル/
ユーザー空間の区別、fork/exec とファイルディスクリプタ)だけです。読み終えると、
`ls -l` の1行をすべて解読でき、「Permission denied」の原因を仕組みから絞り込める
ようになります。

## 導入 — 一本の木と、共同生活のルール

### ドライブレターのない世界

Windowsに触れたことがあれば、記憶装置ごとに `C:` `D:` という「別々の入り口」が
あることを知っているでしょう。Linux(およびUNIX系OS)の設計は根本から違います。
**ファイルの世界全体が、`/`(ルートディレクトリ)を根とするたった一本の木**です。
ディスクが何台あっても、USBメモリを挿しても、入り口は増えません。新しい装置の
中身は、**マウント(mount)**という操作で「木のどこかの枝に接ぎ木」され、
一本の木の一部として見えるようになります。

この設計のおかげで、プログラムは「どの装置に入っているか」を気にせず、
`/home/megane/data.txt` のような**パス(木の根からの道順)**だけでファイルを
指定できます。装置の違いを隠して統一的な木に見せる——これは前々章で説明した、
OSの仕事のひとつ「抽象化」そのものです。

### 地図が共通だから、迷わない

木が一本でも、枝の配置がマシンごとにばらばらでは困ります。「設定ファイルはどこ?」
「ログはどこ?」が管理者ごとに違ったら、ソフトウェアの配布も運用の引き継ぎも
成り立ちません。そこで、主要なディレクトリの名前と役割を取り決めた標準が
**FHS(Filesystem Hierarchy Standard、ファイルシステム階層標準)**です。
現行版は **Linux Foundation が公開する FHS 3.0(2015年)**で、主要な
ディストリビューションはこれに(細部の差はあれ)従っています。だからこそ、
初めて触るサーバーでも「設定は `/etc`、ログは `/var/log`」という地図が通用します。

### 共同生活には鍵が要る

サーバーは典型的な**マルチユーザーシステム**です。複数の人間(と、人間の代わりに
働く多数のデーモン)が同じ一本の木を共有します。共同住宅に部屋ごとの鍵があるように、
「このファイルは誰が読めて、誰が書けるのか」を決める仕組みが必要です。それが
**パーミッション(permission、アクセス権)**です。あなたが自分のファイルを読める
一方で `/etc/shadow`(パスワード情報)を読めないのは、カーネルがファイルごとの
パーミッションを照合し、依頼を許可・拒否しているからです。

## 理論

### 単一ツリーとマウント — POSIXの前提

POSIX(IEEE Std 1003.1)は、ファイルの世界を「ルートディレクトリを起点とする
単一の階層」としてモデル化しています。パス名の解決(`/home/megane/data.txt` の
ような文字列から実際のファイルへたどり着く処理)は、`/` から一段ずつ枝をたどる
操作として規定されています。

マウントとは、この木の**あるディレクトリを「別のファイルシステムの入り口」に
すり替える**操作です。たとえば2台目のディスクを `/data` にマウントすると、
以後 `/data` 以下へのアクセスはすべて2台目のディスクに向かいます。木は一本の
まま、裏側の装置だけが切り替わる——利用者からは継ぎ目が見えません
(マウントの内部機構であるVFSは `03_filesystem_storage/01_vfs_basics.md` の主題です)。

さらにLinuxでは、**ディスクですらないもの**も木に接ぎ木されています。前章の
実行例で使った `/proc` は、カーネルが自身の内部情報(プロセス一覧など)を
「ファイルのふり」をして見せてくれる疑似ファイルシステムです。同様に `/sys` は
デバイスやカーネル設定を見せます。装置も、カーネルの内部状態も、すべて
「ファイルの読み書き」という同じ作法で扱える——この「**すべてはファイル**」という
方針はUNIX設計哲学の柱で、本書でもこの先繰り返し登場します。

### FHS — 主要ディレクトリの地図

FHS 3.0 が定める主要ディレクトリの役割です。全部を暗記する必要はありません。
まず太字の6つを覚えれば、日常の作業で迷うことはなくなります。

| ディレクトリ | 役割 |
|---|---|
| **/etc** | システム全体の設定ファイル(editable text configuration と俗に言われる) |
| **/home** | 一般ユーザーのホームディレクトリ(`/home/megane` など) |
| **/var** | 可変データ。ログ(`/var/log`)、スプール、キャッシュなど運用中に増減するもの |
| **/usr** | インストールされたプログラムとライブラリの本体。読み取り専用にできる静的な領域 |
| **/tmp** | 一時ファイル。全ユーザーが書ける共有の作業場(後述のstickyビットが守る) |
| **/boot** | カーネル本体(vmlinuz)とブートローダ関連 |
| /bin, /sbin, /lib | 歴史的にはブート初期に必要な最小限のコマンド・ライブラリ置き場(現代の扱いは後述) |
| /root | rootユーザー専用のホームディレクトリ(`/home` の下ではない) |
| /dev | デバイスファイル。装置を「ファイルのふり」で見せる入り口 |
| /proc, /sys | カーネルが内部情報を見せる疑似ファイルシステム(FHSも言及するがLinux固有) |
| /opt | 自己完結型のアドオンソフトウェア一式を置く場所 |
| /srv | このサーバーが提供するサービスのデータ(Web公開ディレクトリなど) |
| /mnt, /media | 一時的なマウント用(/mediaはリムーバブルメディア) |
| /run | 起動後に生成される実行時データ(PIDファイル、ソケット等)。再起動で消える |

でたらめに見える配置にも、FHSが明文化した**2軸の設計原理**があります。

- **静的(static)か可変(variable)か**: 運用中に書き換わらないもの(プログラム
  本体=`/usr`)と、日々増減するもの(ログ=`/var`)を分ける。静的な領域は
  読み取り専用マウントやバックアップ戦略の面で扱いが楽になる
- **共有可能(shareable)か固有(unshareable)か**: 複数マシンで共有できるもの
  (`/usr`、`/home`)と、そのマシン固有のもの(`/etc`、`/boot`)を分ける

「なぜプログラムは `/usr` でログは `/var` なのか」の答えがこれです。`/usr` は
静的・共有可能、`/var` は可変・(概ね)固有。性質の異なるデータを別の枝に
分けておけば、枝ごとに別のディスクを接ぎ木したり、マウントの属性(読み取り専用
など)を変えたりできます。単一ツリー+マウントという仕組みと、FHSの分類が
かみ合って初めて意味を持つ設計です。

なお `/bin` と `/usr/bin` の使い分けは歴史的経緯(初期UNIXのディスク容量不足に
由来する分割)であり、現代では統合が進んでいます。Ubuntu を含む主要ディストロは
`/bin` を `/usr/bin` へのシンボリックリンク(後述)にする「usrマージ」を完了して
おり、本書の基準環境(Ubuntu Server 26.04 LTS)でも `/bin` と `/usr/bin` は
実体として同じ場所です。

### ユーザーとグループ — カーネルにとっては番号

パーミッションの話に入る前に、「誰が」を決める仕組みを押さえます。

カーネルは「megane」という名前を知りません。カーネルが管理するのは
**UID(User ID、ユーザーID)**と**GID(Group ID、グループID)**という番号だけです。
プロセスは必ずUID/GIDを身分証として携えており(fork で子に引き継がれます)、
ファイルには所有者のUIDと所属グループのGIDが記録されています。名前と番号の
対応表がユーザーランド側のファイル `/etc/passwd` と `/etc/group` で、`ls -l` が
名前を表示できるのは、この対応表で番号を名前に引き直しているからです
(ユーザー管理の運用は `07_operations_security/01_user_permission_auth.md` で扱います)。

**グループ**は「ユーザーの集合」に付けた名前で、部署やチーム単位でファイルを
共有するための仕組みです。1人のユーザーは複数のグループに所属できます。

そして特別な存在が **root(スーパーユーザー、UID 0)**です。伝統的なUNIXの
モデルでは、UID 0 のプロセスはパーミッション検査を免除され、すべてのファイルに
アクセスできます(厳密には、現代のLinuxはこの「rootの全能性」をcapabilityという
単位に分割していますが、これは発展的話題として分野07で扱います。当面は
「root = 検査免除」という理解で本筋を追えます)。

### 3×3のパーミッションモデル

POSIXが規定するパーミッションの基本モデルは、驚くほど小さい表です。
ファイルごとに、**3種類の相手**に対して**3種類の操作**の可否を記録します。

|  | 読み r (read) | 書き w (write) | 実行 x (execute) |
|---|---|---|---|
| **所有者(owner/user)** | 可否 | 可否 | 可否 |
| **グループ(group)** | 可否 | 可否 | 可否 |
| **その他(others)** | 可否 | 可否 | 可否 |

計9個のon/offで、これが `ls -l` の先頭に並ぶ `rwxr-xr--` のような9文字の正体です
(3文字ずつ所有者・グループ・その他の順。`-` は不許可)。

このモデルの設計上の妙は「**ファイル側に許可リストを書く**」ことにあります。
相手ごとの許可を全ユーザー分列挙する方式(アクセス制御リスト)に比べ、
所有者・グループ・その他という3段の粗い分類は、たった9ビットで表現でき、
検査も一瞬で終わります。粗すぎて困る場面のための拡張(POSIX ACL)も存在しますが、
基本モデルの単純さこそがUNIXが50年間使い続けてきた理由です。

## 内部動作の詳細

### パーミッションはどこに記録されているか — inodeの先取り

`ls -l` が表示する情報(パーミッション、所有者、サイズ、更新日時)は、ファイルの
「中身」ではなく「属性」です。Linuxのファイルシステムでは、この属性一式は
**inode(アイノード)**というファイル1つにつき1個の管理台帳に記録されています。
重要なのは、**ファイル名はinodeに含まれない**ということです。名前は
ディレクトリ側が持つ「名前→inode番号」の対応表のエントリにすぎません。
この分離が後述する「削除の権限はファイルではなく親ディレクトリで決まる」の
伏線になります(inodeの本格的な構造は `03_filesystem_storage/03_inode_block_management.md`
の主題です)。

`ls -l` は各ファイルに対して `stat` システムコールを発行し、カーネルがinodeから
読み出した属性を受け取って整形表示しているだけです。1行を完全に解剖すると
こうなります。

```
 -  rwx r-x r--  1  megane  staff  4096  Jul 13 10:00  report.txt
 │   │   │   │  │    │       │      │        │            │
 │   │   │   │  │    │       │      │        │            └ ファイル名(ディレクトリ由来)
 │   │   │   │  │    │       │      │        └ 最終更新日時 ┐
 │   │   │   │  │    │       │      └ サイズ(バイト)      │
 │   │   │   │  │    │       └ グループ(GIDを名前に変換)  ├ ここから左は
 │   │   │   │  │    └ 所有者(UIDを名前に変換)            │ すべてinode由来
 │   │   │   │  └ ハードリンク数(分野03で扱う)            │
 │   │   │   └ その他の許可(r--: 読みのみ)                │
 │   │   └ グループの許可(r-x: 読みと実行)                │
 │   └ 所有者の許可(rwx: すべて)                          │
 └ ファイル種別(-:通常ファイル d:ディレクトリ l:シンボリックリンク)┘
```

9ビットの許可は**8進数3桁**でも表現されます。r=4、w=2、x=1 の重み付けで、
`rwxr-xr--` は 4+2+1 / 4+0+1 / 4+0+0 = **754** です。`chmod 644 file` のような
数値指定はこの記法で、各桁が所有者・グループ・その他に対応します。

### 検査のアルゴリズム — 最初に一致した1段しか見ない

カーネルがアクセス可否を判定する手順は、POSIX(IEEE Std 1003.1 の File Access
Permissions の節)に明確に規定されています。

1. プロセスのUIDがファイルの所有者UIDと一致するか? → 一致すれば
   **所有者の3ビットだけ**で判定(以降は見ない)
2. 一致しなければ、プロセスのGID(所属グループのいずれか)がファイルの
   GIDと一致するか? → 一致すれば**グループの3ビットだけ**で判定
3. どちらでもなければ、**その他の3ビットだけ**で判定

見落としがちなのは「**最初に一致した1段しか見ない**」という点です。3段は
「いずれかが許せばOK」の論理和ではありません。たとえばモード `044`
(所有者は何もできず、グループとその他は読める)のファイルは、**所有者だけが
読めない**という一見奇妙な状態になります。所有者はステップ1で判定が確定し、
グループ以降のビットは参照されないからです(所有者は `chmod` で自分の許可を
戻せるので、締め出されたままにはなりませんが)。

もう一つ、検査の**タイミング**が重要です。パーミッション検査が行われるのは
ファイルを**開く(open)瞬間の一度だけ**です。openが成功してファイル
ディスクリプタ(前章)を得たあとは、途中でファイルのパーミッションを変更されても、
そのfd経由の読み書きは継続できます。「fdは入場済みの通行証」と捉えてください。
ログを書き続けているデーモンからファイルの書き込み許可を取り上げても、開き直す
までは書き込みが続くのはこのためです。

### ディレクトリのrwxは意味が違う

同じrwxでも、ディレクトリに対しては意味が変わります。ディレクトリの実体が
「名前→inode番号の対応表」であることを思い出すと、素直に理解できます。

| ビット | ディレクトリでの意味 |
|---|---|
| r | 対応表を**読める** = 中のファイル名一覧を取得できる(`ls` ができる) |
| w | 対応表を**書き換えられる** = エントリの追加・削除・改名ができる |
| x | 対応表を**通過できる** = 名前からinodeへたどれる(中のファイルにアクセスできる) |

ここから2つの重要な帰結が出ます。

- **パス解決には、途中の全ディレクトリのxが必要**:
  `/home/megane/data.txt` を開くには、`/`、`/home`、`/home/megane` のすべてで
  x(通過)の許可が要ります。ファイル自身のパーミッションが完璧でも、途中の
  ディレクトリ1つがxを欠けば「Permission denied」です
- **ファイルを削除する権限は、ファイル自身ではなく親ディレクトリのwで決まる**:
  削除とは「ディレクトリの対応表からエントリを消す」操作だからです。自分に
  読み書きできないファイルでも、親ディレクトリにwがあれば削除できます。逆に、
  自分のファイルでも親ディレクトリにwがなければ削除できません

「rだけあってxがない」ディレクトリでは名前一覧は見えるのに中身にアクセスできず、
「xだけあってrがない」ディレクトリでは一覧は見えないのに名前を知っていれば
アクセスできる——という非対称も、対応表の読み取りと通過が別の操作である
ことから導けます。

### 新しいファイルのパーミッションはどう決まるか — umask

ファイルを作るとき、プログラムは `open` システムコールに「希望するモード」を
渡します(慣例的にファイルは `666`、ディレクトリは `777` を希望します)。しかし
実際に付くパーミッションは、そこから **umask** という値のビットを**取り除いた**
ものです(`man 2 umask`)。

```
実際のモード = 希望モード AND (NOT umask)
例: 希望 666、umask 022 → 644 (rw-r--r--)
```

umaskはプロセスごとの属性で、シェルから子プロセスへ引き継がれます。Ubuntuの
既定は `022`(グループとその他のwを削る)です。「新しく作ったファイルがなぜ
いつも644なのか」の答えは、この引き算にあります。umaskが「与える許可」ではなく
「**削る許可**」の指定であることに注意してください。

### 3つの特殊ビット — setuid・setgid・sticky

9ビットの上に、もう3つ特殊なビットがあります(8進で4桁目: setuid=4、
setgid=2、sticky=1)。仕組みの面白さが凝縮されているのが setuid です。

- **setuidビット**: 実行ファイルに付いていると、**起動されたプロセスは
  起動した人ではなく「ファイルの所有者」の身分(実効UID)で動きます**。
  典型例が `/usr/bin/passwd` です。パスワードの保存先 `/etc/shadow` は
  root以外読み書き禁止ですが、passwdコマンドは root所有+setuid なので、
  一般ユーザーが起動してもrootの身分で動き、shadowを更新できます。
  「特定の操作に限って、一時的に高い権限を貸し出す」ための正規の通用口ですが、
  裏を返せばsetuidプログラムの欠陥は権限昇格の穴になるため、数を絞り厳重に
  管理すべき対象です(運用面は分野07で扱います)
- **setgidビット**: 実行ファイルでは「ファイルのグループの身分で動く」。
  **ディレクトリに付けた場合は意味が変わり**、その中に作られる新しいファイルが
  親ディレクトリのグループを引き継ぐようになります。チームの共有ディレクトリで
  「誰が作ってもチームのグループになる」を実現する定番の使い方です
- **stickyビット**: ディレクトリに付けると、wがあっても**他人のファイルは
  削除できなくなります**(削除できるのはファイル所有者・ディレクトリ所有者・
  rootのみ)。全ユーザーが書ける `/tmp` が無法地帯にならないのはこのビットの
  おかげで、`ls -ld /tmp` のモード末尾に `t` として見えます

### シンボリックリンク — 名前が名前を指す

`ls -l` のファイル種別 `l` は**シンボリックリンク(symbolic link、symlink)**です。
実体は「別のパス名が書いてある小さな特殊ファイル」で、パス解決の途中で
シンボリックリンクに出会うと、カーネルは書かれているパスに読み替えて解決を
続けます。usrマージの `/bin → /usr/bin` はまさにこれです。

シンボリックリンク自身のパーミッションは `lrwxrwxrwx` と表示されますが、
これは飾りで、**アクセス可否はリンク先のパーミッションで判定されます**。
また、リンク先が存在しなければ「あるのに開けない」壊れたリンク(dangling link)に
なります。名前の付け替えにすぎない同種の仕組みとして**ハードリンク**もありますが、
inodeの理解が要るため `03_filesystem_storage/03_inode_block_management.md` に
委ねます。

## 実行例 — 仕組みを自分のマシンで確認する

前提はUbuntu Server 26.04 LTSです。

inode由来の属性を直接見る(`stat` は同名のシステムコールの結果を表示する):

```console
$ stat report.txt
  File: report.txt
  Size: 4096       ...  Inode: 1835028     ← inode番号(名前とは独立の管理台帳)
Access: (0644/-rw-r--r--)  Uid: (1000/megane)   Gid: (1000/megane)
                           ↑ 8進表記と記号表記、UID/GIDと名前の対応が並ぶ
```

「最初に一致した1段しか見ない」を体験する(所有者だけ読めないファイル):

```console
$ echo hello > test.txt
$ chmod 044 test.txt          # 所有者---、グループr--、その他r--
$ cat test.txt
cat: test.txt: Permission denied   ← 所有者なのに読めない(所有者段で判定確定)
$ chmod 644 test.txt               ← 所有者はchmodで自力復旧できる
```

ディレクトリのx(通過)の意味を体験する:

```console
$ mkdir -p box && echo data > box/file.txt
$ chmod 644 box               # rにするがxを外す(通過禁止)
$ ls box
ls: cannot access 'box/file.txt': Permission denied
file.txt                      ← 名前一覧(r)は読めるが……
$ cat box/file.txt
cat: box/file.txt: Permission denied   ← 通過(x)できず中身に届かない
$ chmod 755 box
```

umaskの引き算を確認する:

```console
$ umask
0022                          ← 現在のumask(グループ・その他のwを削る)
$ touch new.txt && ls -l new.txt
-rw-r--r-- 1 megane megane 0 ... new.txt   ← 666 AND NOT 022 = 644
```

setuid・stickyビットを実物で見る:

```console
$ ls -l /usr/bin/passwd
-rwsr-xr-x 1 root root ... /usr/bin/passwd   ← 所有者のxの位置が s = setuid
$ ls -ld /tmp
drwxrwxrwt 14 root root ... /tmp             ← 末尾の t = stickyビット
$ ls -ld /bin
lrwxrwxrwx 1 root root 7 ... /bin -> usr/bin ← usrマージ(シンボリックリンク)
```

## トラブルシューティング — Permission denied の絞り込み方

- **ファイルは644なのに読めない**: 疑うべきは**途中のディレクトリのx**です。
  パス解決は根から一段ずつ進むため、`/a/b/c.txt` なら `/`・`/a`・`/a/b` の
  すべてに通過(x)が必要です。`namei -l /a/b/c.txt` は各段のパーミッションを
  一覧表示してくれる調査の定番です
- **グループに追加してもらったのに、まだ読めない**: プロセスの所属グループは
  **ログイン時に確定**し、実行中のプロセスには反映されません。`groups` コマンドの
  表示が(現在のシェルの認識)と `/etc/group`(登録上の所属)で食い違っていたら
  これです。再ログインするか、`newgrp <グループ>` で新しいシェルを開きます
- **消せないはずのファイルが消えた/消せるはずが消せない**: 削除の可否は
  ファイル自身ではなく**親ディレクトリのw**(と、stickyビットの有無)で
  決まります。`ls -ld <親ディレクトリ>` から確認します
- **rootなのに書けない**: パーミッションの問題ではありません。典型は
  ファイルシステムが**読み取り専用でマウント**されている場合で、`mount` の
  出力に `ro` と見えます(異常検出時にカーネルが自衛のためroで再マウントする
  ことがあります。詳細は分野03)。ほかに immutable 属性(`lsattr` で確認)という
  拡張もあります
- **`sudo echo x > /etc/xxx.conf` が失敗する**: これはパーミッションの理解では
  なくシェルの理解の問題です。リダイレクトを処理するのは権限の上がる前の
  シェル自身——前章のトラブルシューティングを参照してください
- **パーミッションを絞ったのにデーモンが書き続けている**: 検査はopen時の
  一度だけで、取得済みのfdは影響を受けません。反映させるにはファイルを
  開き直させる(デーモンの再起動など)必要があります

## 演習・確認問題

1. `/usr` と `/var` が別のディレクトリに分かれている理由を、FHSの2軸
   (静的/可変、共有可能/固有)を使って説明してください
2. `-rwsr-xr-x 1 root root ... /usr/bin/passwd` という表示について、
   (a) `s` は何を意味するか、(b) この仕組みがないと一般ユーザーの
   パスワード変更はなぜ成立しないか、を説明してください
3. モード `044`・所有者megane のファイルを、(a) megane、(b) meganeと同じ
   グループの他ユーザー、(c) root、それぞれが読めるかどうか、検査の
   アルゴリズムに沿って答えてください
4. 「ファイルを削除できるかどうかは、そのファイルのパーミッションでは
   決まらない」と言われるのはなぜですか。ディレクトリの実体(名前→inodeの
   対応表)という観点から説明してください
5. umaskが `027` のとき、新規作成されるファイル(希望モード666)と
   ディレクトリ(希望モード777)のモードをそれぞれ8進数で答えてください

## まとめ

- Linuxのファイルの世界は `/` を根とする一本の木で、装置や疑似ファイルシステムは
  マウントで接ぎ木される。主要ディレクトリの配置は **FHS 3.0(Linux Foundation)**が
  標準化しており、静的/可変・共有可能/固有の2軸で整理されている
- パーミッションは所有者・グループ・その他 × rwx の9ビットで、ファイル名では
  なくinodeに記録される。検査は「最初に一致した1段だけ」を見て、openの瞬間に
  一度だけ行われる
- ディレクトリのrwxは「一覧を読む・エントリを書き換える・通過する」の意味になり、
  パス解決には途中の全ディレクトリのxが、削除には親ディレクトリのwが要る
- 新規ファイルのモードは「希望モードからumaskのビットを削った」結果になる
- setuid(所有者の身分で実行)・setgid(グループ継承)・sticky(他人のファイルの
  削除禁止)の3ビットが基本モデルを補い、passwdや/tmpを成立させている
