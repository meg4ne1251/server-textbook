# ネットワークストレージ基礎 — NFS と iSCSI

## 概要

分野03の最終章として、ストレージをネットワーク越しに使う2つの代表方式——
ファイルの層で共有する **NFS** と、ブロックの層で貸し出す **iSCSI** ——を
扱います。前提知識は `03_filesystem_storage/01`(VFS の契約)と
`03_filesystem_storage/04`(ブロックデバイスの抽象と積み重ね)です。
TCP/IP などネットワークプロトコル自体の詳細は `network-guide` の領分、
ソケットの内部機構は分野04で扱います。基準環境は Linux 7.0 /
Ubuntu Server 26.04 LTS です。

## 導入 — 「ディスクはこの筐体の中にある」という前提を疑う

前章は「ファイルシステムと物理ディスクの直結」を疑い、間に翻訳の層を
挟みました。しかし LVM も RAID も、ディスクが**このサーバーの筐体の中に
ある**ことは疑っていません。サーバーを何台も運用する段になると、この
前提が今度は問題になります。

1. **複数のサーバーで同じデータを使いたい。** Web サーバーを3台に増やした
   とき、ユーザーがアップロードしたファイルは3台のどこに置くのか。
   各自のローカルディスクに置けば、たちまち3つの食い違うコピーが生まれます
2. **データの寿命とサーバーの寿命を切り離したい。** サーバー本体が故障しても
   データは残ってほしい。仮想マシン(分野05)を別の物理ホストへ移す
   ときも、ディスクの中身が特定の筐体に縛られていては動かせません
3. **ストレージを一括で管理したい。** 容量の追加・冗長化・バックアップを
   サーバー1台ごとにやるのではなく、専用の置き場に集約して面倒を見たい

答えの方向は明らかで、「ストレージをネットワークの向こうに置く」です。
問題は**どうやって**です。ここで、本分野で積み上げてきた2つの抽象を
思い出してください。

- **ファイルの顔**(VFS)——「open して read/write する」という契約。
  裏の実装が何であるかをアプリケーションは知らない(`03_filesystem_storage/01`)
- **ブロックの顔**(ブロックデバイス)——「番号順に並んだ区画をどこでも
  読み書きできる」という契約。裏が何台のディスクかを mount は知らない
  (`03_filesystem_storage/04`)

どちらの契約も「裏を知らない」のなら、**裏をネットワークにしてしまっても
契約は守れる**はずです。実際、ネットワークストレージの2大方式は、この
2つの顔のどちらを遠隔化するかにきれいに対応します。

```
   NFS(ファイルの層で切る)            iSCSI(ブロックの層で切る)

   ┌──────────────────┐               ┌──────────────────┐
   │ アプリケーション  │               │ アプリケーション  │
   ├──────────────────┤               ├──────────────────┤
   │       VFS        │               │       VFS        │
   ├──────────────────┤               ├──────────────────┤
   │ NFS クライアント  │←ここから遠隔  │    ext4 / XFS    │←台帳はこちら側
   └────────┬─────────┘               ├──────────────────┤
            ┆ ファイル単位の依頼       │    ブロック層     │
      ~~ ネットワーク ~~              ├──────────────────┤
            ┆ (LOOKUP/READ/WRITE)     │ iSCSIイニシエータ │←ここから遠隔
   ┌────────┴─────────┐               └────────┬─────────┘
   │   NFS サーバー    │                        ┆ ブロック単位の依頼
   │   ext4 等の FS    │←台帳はこちら側    ~~ ネットワーク ~~
   │ ブロックデバイス  │                        ┆ (SCSI READ/WRITE)
   └──────────────────┘               ┌────────┴─────────┐
                                      │ iSCSI ターゲット  │
                                      │ ブロックデバイス  │ 中身は関知しない
                                      └──────────────────┘
```

ファイルの層で切って共有する装置・構成を **NAS**(Network Attached
Storage)、ブロックの層で切って貸し出す構成を **SAN**(Storage Area
Network)と呼びます。この章で見ていくとおり、**どの層でネットワークを
挟むかが、何を共有できるか・整合性を誰が守るかを決めます**。これが
この章の背骨です。

## 理論

本節の内容は、主に **NFS の RFC(NFSv3: RFC 1813、NFSv4.0: RFC 7530、
NFSv4.1: RFC 8881)**、その土台の **ONC RPC(RFC 5531)/ XDR(RFC 4506)**、
**iSCSI の RFC 7143**、`man 5 nfs`、`man 5 exports`、および Linux カーネル
ドキュメント(Documentation/filesystems/nfs/、Documentation/target/)に
基づきます。SCSI コマンド自体の仕様は標準化団体 T10 の規格群
(SAM / SBC)で定められています。

### どの層で切るかが、責任の在り処を決める

2方式の違いは、通信の中身を比べると鮮明になります。

| | NFS(ファイルの層) | iSCSI(ブロックの層) |
|---|---|---|
| ネットワークを流れる依頼 | 「このファイルのここを読め」 | 「セクタ番号 N から M 個読め」 |
| ファイルシステムの台帳がある場所 | **サーバー側** | **クライアント側** |
| サーバーが知っていること | ファイル・所有者・パーミッション | ただのブロックの並び |
| 複数クライアントからの同時利用 | 前提(サーバーが調停する) | **不可**(調停者がいない) |
| クライアントから見えるもの | mount された FS の一部 | 新しいブロックデバイス(/dev/sdX) |

最重要の行は「台帳がある場所」です。NFS では、inode・ビットマップ・
ディレクトリといった台帳(`03_filesystem_storage/03`)を管理するのは
**サーバー側の**ファイルシステムです。クライアントが何台いても、台帳は
1つ、更新の入口も1つ——だから共有が成立します。iSCSI のサーバーは
ブロックの置き場を貸すだけで、そこにどんな台帳が書かれているかを
知りません。台帳の管理は完全にクライアント側の ext4 / XFS の仕事で、
**管理者が1人しかいないことを前提にした台帳を2人が同時に書けば壊れる**
——この帰結は iSCSI の節で詳しく見ます。

### NFS — VFS の契約をネットワークへ延ばす

NFS(Network File System)は、ファイル操作の依頼をネットワーク越しに
送る仕組みで、土台には **RPC(Remote Procedure Call / 遠隔手続き呼び出し)**
という汎用機構を使います(ONC RPC。RFC 5531)。「手続き番号と引数を
決まった形式(XDR)に詰めて送ると、相手がその手続きを実行して結果を
返す」——関数呼び出しの形をした通信です。NFS はこの上に LOOKUP、
GETATTR、READ、WRITE といった手続き一式を定義したものにすぎません。

依頼の宛先には**ファイルハンドル**を使います。パス名ではありません。
クライアントはまず LOOKUP(ディレクトリのハンドル, 名前)で目的の
ファイルのハンドルを受け取り、以後の READ / WRITE はハンドルだけで
行います。ハンドルの中身はクライアントには不透明(opaque)ですが、
サーバー側の実装では概ね「ファイルシステム ID + inode 番号 + 世代番号」
です——**名前と inode の分離**(`01_intro/03`、`03_filesystem_storage/03`)
という UNIX の設計が、そのままプロトコルの語彙に現れているわけです。
LOOKUP がパス解決の1成分ぶん、ハンドルが inode 参照に対応します。

歴史的に重要なのが、NFSv2 / v3(RFC 1813)の**ステートレス**という
設計判断です。サーバーは「どのクライアントが何を開いているか」を
**一切覚えません**。プロトコルに OPEN / CLOSE すら存在せず、READ は
毎回「ハンドル+オフセット+長さ」を完全に指定します。狙いは障害への
強さです。サーバーが再起動しても、失われる状態がそもそもないので、
クライアントは同じ依頼を再送するだけで続きから動けます。代償は、
状態が本質的に必要な機能——ファイルロック——をプロトコルの外
(NLM という別プロトコル)に追い出したことで、これは長年の不具合の
温床になりました。

NFSv4(RFC 7530、v4.1 は RFC 8881)は、この設計を転換して
**ステートフル**になりました。OPEN / CLOSE が導入され、状態は
**リース(lease)**——期限付きの貸与——として管理されます。
クライアントが定期更新を怠れば(=消息を絶てば)状態は期限切れで
自動回収されるため、ステートレスが持っていた「相手の死で状態が
腐らない」という利点をかなりの程度保ちます。ほかにも、複数の手続きを
1往復に束ねる COMPOUND、通信の TCP ポート 2049 番への一本化
(v3 で必要だった rpcbind 等の補助プロトコルが不要に)、ファイルの
読み書き権限を一定期間クライアントへ委任してローカル並みの動作を
許す**デリゲーション**、v4.1 ではメタデータとデータの経路を分離する
pNFS など、運用に効く改善が入っています。本書が基準とする現代の
構成では v4 系の利用が標準です。

### 分散するとキャッシュが問題になる — close-to-open 整合性

ローカルのファイルシステムでは、ページキャッシュ(`03_filesystem_storage/01`)
は常に「唯一の真実」でした。どのプロセスの read/write も同じカーネルの
同じキャッシュを通るので、食い違いようがありません。NFS はここに
新しい問題を持ち込みます。**クライアントごとに別々のカーネル、別々の
ページキャッシュ**があり、クライアント A の書き込みをクライアント B の
キャッシュは知らないのです。

すべての read/write を毎回サーバーに問い合わせれば正しさは保てますが、
キャッシュの意味がなくなり、性能はネットワーク往復の回数で決まって
しまいます。NFS が選んだ現実的な落とし所が **close-to-open 整合性**
(`man 5 nfs`)です。約束はこうです。

> クライアント A がファイルを **close** した後にクライアント B が
> **open** すれば、B は A の書き込みをすべて見る。

機構は2つの動作の組でできています。close 時に、そのファイルの
ダーティなページをすべてサーバーへ書き戻す(フラッシュ)。open 時に、
サーバーへ属性を問い合わせ、手元のキャッシュが古くなっていないかを
更新時刻(v4 では change 属性)で検証し、古ければキャッシュを捨てる。
つまり「**close で真実をサーバーに集め、open で真実と突き合わせる**」。
逆に言えば、open と close の**間**の同時アクセスについては何も約束
しません。また属性の問い合わせ自体も**属性キャッシュ**(既定でファイル
3〜60秒)で節約されるため、他クライアントの変更が見えるまで最大で
その窓ぶんの遅れがあります。

この約束は「1人が書いて保存し、他の人が後から開く」という共同作業の
典型パターンをちょうど安全にする設計です。複数クライアントから同じ
ファイルへ**同時に**書き込む用途は、NFS の保証の外側にあります
(必要ならロックを併用し、アプリケーション側で設計します)。

### iSCSI — ブロックデバイスの顔をした TCP 接続

iSCSI の理解は、SCSI を知れば半分終わります。**SCSI** はもともと
ディスクとコンピュータの間の会話——「セクタ番号(LBA)N から M 個
読め/書け」というコマンドの語彙——を定めた規格の一族(T10)で、
Linux カーネルのディスクドライバ(sd)は今もこの語彙でディスクと
会話しています。**iSCSI(RFC 7143)は、この SCSI コマンドを TCP の
上に載せてネットワークへ流す**、それだけの仕組みです。

登場人物は2人です。依頼する側(クライアント)を**イニシエータ
(initiator)**、ブロックを貸す側(サーバー)を**ターゲット(target)**と
呼びます。双方は IQN(iSCSI Qualified Name。`iqn.2026-07.com.example:...`
形式)という名前を持ち、イニシエータがターゲットへログインすると、
ターゲットが束ねている論理ディスク——**LUN(Logical Unit Number)**——が
イニシエータ側に見えるようになります。

クライアント側から見た結果は劇的に単純です。**新しいブロックデバイス
(/dev/sdb 等)が1本生える。以上。** カーネルにとっては裏が TCP で
あることを除けばローカルディスクと同じ顔なので、前章までのすべてが
そのまま通用します——mkfs でファイルシステムを作るのも、pvcreate で
LVM の材料にするのも、md の配列に加えることさえも。ブロックデバイスの
抽象が「積み重ねられる」ことの、最も遠くまで行った例です。

そして、先に予告した帰結がここで牙をむきます。**同じ LUN を2台の
ホストから同時に mount してはいけません。** ext4 / XFS は単一ホスト用の
ファイルシステムです。スーパーブロックや空きブロックビットマップを
メモリに読み込んで(`03_filesystem_storage/01`〜`03`)、自分だけが
それを更新すると信じて動きます。2台がそれぞれ手元のキャッシュ上で
台帳を書き換えれば、互いの変更を知らないまま同じブロックを別の用途に
割り当てる——即座に、確実に、破損します。RAID の write hole のような
確率的な事故ではなく、設計前提の違反です。共有ブロックデバイスの上で
複数ホストが同居するには、分散ロックで台帳の更新を調停する**クラスタ
ファイルシステム**(GFS2 / OCFS2)が必要で、これは分野08(高可用性
クラスタ)の話題です。iSCSI の通常の使い方は「1つの LUN は同時には
1台が使う」——共有ではなく、**ディスクの遠隔化**です。

なお、ブロックを遠隔化する仕組みは iSCSI だけではありません。専用網を
使う Fibre Channel(FC)は伝統的な SAN の主役ですし、近年は NVMe を
そのままネットワークに載せる NVMe over Fabrics(NVMe-oF)が高性能側の
標準になりつつあります。「ブロックの層で切る」という構図はどれも
同じです。

## 内部動作の詳細

### NFS クライアントの中 — 見慣れた VFS オブジェクト、遠い裏側

NFS クライアントはカーネル内の1つのファイルシステム実装(fs/nfs/)で、
mount すると superblock ができ、パス解決で dentry と inode が作られて
いきます——VFS の章で見た景色と完全に同じです。違いは操作の裏側だけで、
たとえば NFS の inode が持つ属性(サイズ、更新時刻…)は「最後に
GETATTR で聞いた答えのキャッシュ」であり、ローカル FS のような真実
そのものではありません。属性キャッシュの正体はこれです。

read はまずページキャッシュを引き、なければ READ RPC を発行して
埋めます(先読みも効きます)。write はページキャッシュに書いて
ダーティ印を付けて即座に戻り、後からライトバックが WRITE RPC に
なります——**「遅らせると、まとめられて、賢く決められる」の型が、
裏がネットワークになってもそのまま再演される**わけです。v3 以降の
WRITE には興味深い2段階があります。通常の WRITE は「サーバーの
メモリに届けばよい」(UNSTABLE)とし、後で COMMIT を送って永続化を
確認するのです。サーバーは再起動すると起動ごとの照合値(verifier)を
変えるため、クライアントは COMMIT の応答で「あの書き込みはサーバーの
クラッシュで消えた」ことを検出し、手元に保持していたデータを再送
できます。ライトバックと fsync の分業(`03_filesystem_storage/02`)が、
ネットワークを挟んで二重になっている構図です。

### サーバーが応えないとき — D 状態の伏線回収

RPC は応答が来なければ再送します。ではサーバーがダウンしていたら
いつまで再送するのか——ここに mount オプションの重大な選択があります
(`man 5 nfs`)。

- **hard(既定)** — 応答があるまで**無期限に**再送し続ける。依頼した
  プロセスはその間、割り込み不能な待ち——**D 状態**——で眠る
- **soft** — 規定回数の再送で諦め、アプリケーションにエラー(EIO)を
  返す

分野02で「D 状態のプロセスは kill できない」「ロードアベレージは
NFS の詰まりでも上がる」と述べた伏線が、ここで回収されます。NFS
サーバーが無応答になると、そのマウントに触れたプロセスが次々と D 状態で
積み上がり、CPU は暇なのにロードアベレージだけが伸びていきます。
なお現代のカーネルでは NFS の待ちは TASK_KILLABLE(`02_process_kernel/05`
で触れた「致命シグナルだけは受け付ける眠り」)なので、SIGKILL に
限っては効きます。

soft にすれば固まらないのに、なぜ既定は hard なのか。EIO を返すのは
**書き込みの失敗をアプリケーションに押し付ける**ことだからです。
エラー処理の甘いプログラムなら、書けなかったことに気づかずデータを
失います。「遅くなっても正しさを守る」hard が既定であるのは、
ファイルシステムとしては当然の選択です(サーバーは復旧する前提で
運用し、復旧すれば D 状態の行列は何事もなく流れ出します)。

### iSCSI の経路 — I/O がソケットに変わるまで

iSCSI イニシエータのカーネル内の経路は、層の積み重ねとして素直です。

```
  ファイルシステム(ext4 等)
        ↓ ブロックの読み書き
  ブロック層
        ↓ SCSI コマンドに翻訳(sd ドライバ)
  SCSI 中間層
        ↓ このディスクの転送手段は?
  iscsi_tcp(トランスポート)── PDU に包んで TCP ソケットへ
        ↓
  (以降は分野04の世界: ソケット → TCP/IP → NIC)
```

セッションの確立はユーザーランドのデーモン(open-iscsi の iscsid)が
担います。ターゲットの発見(discovery)、ログイン(必要なら CHAP
認証)、切断時の再接続もここの仕事で、確立済みセッション上のデータの
流れはカーネルが処理します——「方針はユーザーランド、機構はカーネル」
の分業がここにもあります。ターゲット側も Linux で完結でき、カーネル内
実装 **LIO** が configfs 経由(管理ツールは targetcli)で設定されます。
LUN の裏の実体(バックストア)にはブロックデバイスもファイルも指定
でき、「thin LV(前章)を LUN として貸す」といった積み重ねが普通に
行われます。

### 経路も冗長化する — dm-multipath

ストレージをネットワークの向こうに置くと、**経路**が新しい単一障害点に
なります。そこで NIC もスイッチも2系統にする——と、今度は奇妙な
ことが起きます。同じ LUN への経路が2本あるため、カーネルには
**同じディスクが /dev/sdb と /dev/sdc の2本に見える**のです。

これを解くのが device mapper の multipath ターゲットです。デバイスの
個体識別子(WWID)を突き合わせて「この2本は同じディスクだ」と認識し、
1つの仮想デバイス(/dev/mapper/mpatha)に束ねます。I/O は経路へ
振り分けられ、片方の経路が死ねば multipathd が検知して残りへ切り替え、
失敗した I/O は再送されます。前章の「翻訳表を挟む」がここでも使われて
いる——linear が位置を、raid が台数を吸収したように、multipath は
**経路**を吸収する翻訳です。ファイルシステムは常に mpatha だけを見て
いればよく、経路の故障に気づきもしません。

## 実行例 — ファイルの顔とブロックの顔

NFS は、サーバー側で公開設定(エクスポート)を書き、クライアントから
mount するだけで試せます(パッケージは nfs-kernel-server / nfs-common)。

```console
(サーバー側)
$ cat /etc/exports
/srv/share  192.0.2.0/24(rw,sync,root_squash)   ← 公開範囲と条件
$ sudo exportfs -ra

(クライアント側)
$ sudo mount -t nfs server:/srv/share /mnt
$ findmnt /mnt
TARGET SOURCE            FSTYPE OPTIONS
/mnt   server:/srv/share nfs4   rw,...,hard,proto=tcp   ← v4・hard が既定
```

mount 済みの NFS の内部統計は `/proc/self/mountstats` に RPC の内訳
まで出ます(nfsstat コマンドでも集計できます)。

iSCSI は、ターゲットを発見してログインすると、ブロックデバイスが
生えるのが見えます(パッケージは open-iscsi)。

```console
$ sudo iscsiadm -m discovery -t sendtargets -p 192.0.2.10
192.0.2.10:3260,1 iqn.2026-07.com.example:storage.lun0
$ sudo iscsiadm -m node --login
$ lsblk
NAME   SIZE TYPE MOUNTPOINTS
sda    100G disk            ← ローカルディスク
└─...
sdb     50G disk            ← いま生えた iSCSI の LUN
```

以後の `/dev/sdb` は普通のブロックデバイスです。mkfs も pvcreate も
できます——それが「ブロックの層で切る」ということの意味です。

## トラブルシューティング

- **ls や df が固まる。ロードアベレージが CPU 使用率と無関係に急騰**:
  NFS サーバー無応答の典型症状です。マウントに触れたプロセスが D 状態で
  積み上がっています(`ps` の STAT 列で確認)。対処の本筋はサーバー側の
  復旧で、復旧すれば行列は自然に解消します。プロセスを個別に止めたい
  場合は SIGKILL だけが効きます
- **他のクライアントの変更が見えない/古い内容が見える**: close-to-open
  整合性の約束どおりの動作の可能性がまずあります。書き手が close して
  いるか、読み手が open し直しているかを確認します。それでも見えない
  場合は属性キャッシュの窓(最大60秒)です。頻繁な相互参照が要件なら
  actimeo の短縮や noac がありますが、RPC が激増する代償を伴います
- **root なのに NFS 上で書けない(Permission denied)**: エクスポート
  既定の **root_squash** が原因です。クライアントの root(UID 0)は
  サーバー側では無権限ユーザーに読み替えられます。「クライアント1台の
  root 奪取が共有ストレージ全体の root になる」ことを防ぐ意図的な設計で、
  安易に no_root_squash にする前に運用を疑います
- **削除したのに `.nfsXXXX` という謎のファイルが残る**: ローカル FS では
  「開かれたまま unlink されたファイル」は参照カウントで生かされます
  (`03_filesystem_storage/01`)。ステートレスな NFS サーバーは誰が
  開いているかを知らないため、クライアントが**開いたまま削除された
  ファイルを `.nfsXXXX` へ rename して延命**します(silly rename)。
  close されれば消えます。残骸が残るのは「開いたまま消した」プロセスが
  どこかにいた証拠です
- **iSCSI の LUN を2台目のホストからも mount したら両方壊れた**:
  設計前提の違反です(理論の節を参照)。ext4 / XFS は単一ホスト専用で、
  修復は fsck でも困難です。複数ホストで同時に使うならクラスタ FS
  (分野08)、そうでなければ「同時には1台」を徹底します
- **再起動したら iSCSI / NFS 上のファイルシステムの mount で起動が
  止まる**: ネットワークが上がる前に mount しようとした順序の問題です。
  fstab では `_netdev` オプションで「ネットワーク後」と宣言します
  (systemd がこれを依存関係に翻訳します——分野06で扱います)
- **同じディスクが sdb と sdc の2本見える**: 経路が2本あるのに
  dm-multipath を構成していない状態です。片方を素で使ってしまうと
  経路故障の切り替えが働きません。multipathd を構成し、以後は
  /dev/mapper/ 側だけを使います

## 演習・確認問題

1. NFS と iSCSI を「どの層でネットワークを挟むか」の観点から比較し、
   ネットワークを流れる依頼の内容、ファイルシステムの台帳を管理する
   場所、複数クライアントから同時に使えるか否か、の3点の違いを
   説明してください
2. NFSv3 の「ステートレス」とはサーバーが何を覚えないことか、それが
   サーバー再起動時の復旧をどう単純にするか、そして何を犠牲にしたかを
   述べてください。NFSv4 のリースが、ステートフル化と障害への強さを
   どう両立させているかも説明してください
3. close-to-open 整合性が保証すること・しないことを述べ、close 時と
   open 時にクライアントがそれぞれ何をすることで実現されているかを
   説明してください
4. NFS の hard マウントでサーバーが無応答になると、プロセスの状態と
   ロードアベレージに何が起きますか。soft マウントが既定でない理由と
   あわせて説明してください(分野02の D 状態の知識を使うこと)
5. 単一ホスト用ファイルシステム(ext4 等)の載った iSCSI の LUN を
   2台のホストから同時に mount すると破損するのはなぜですか。
   「台帳のキャッシュ」と「調停者の不在」という言葉を使って説明し、
   NFS ではなぜ同じ問題が起きないのかも述べてください
6. dm-multipath は何という問題を解決する翻訳層ですか。前章の linear /
   raid ターゲットと並べて、「それぞれが何を吸収する翻訳か」を
   一言ずつで整理してください

## まとめ

- ネットワークストレージは、本分野の2つの抽象のどちらを遠隔化するかで
  2方式に分かれる。**ファイルの層で切る** NFS(NAS)と、**ブロックの
  層で切る** iSCSI(SAN)——切る層が、共有できるものと整合性の責任の
  在り処を決める
- NFS は RPC の上のファイル操作で、台帳はサーバー側にあるから共有できる。
  v3 のステートレスから v4 のリースベースのステートフルへ進化した。
  キャッシュの整合性は close-to-open という現実的な約束で、同時書き込みは
  保証の外
- サーバー無応答時、hard マウントのプロセスは D 状態で待つ(SIGKILL のみ
  有効)。ロードアベレージの急騰として現れる——「遅くても正しく」が
  ファイルシステムの既定
- iSCSI は SCSI コマンドを TCP に載せ、クライアントに普通のブロック
  デバイスを1本生やす。台帳はクライアント側にあり調停者がいないため、
  単一ホスト用 FS の LUN を複数ホストから mount すれば確実に壊れる——
  共有ではなくディスクの遠隔化として使う
- 経路の冗長化は dm-multipath——「翻訳表を挟む」の型が、位置(linear)、
  台数(raid)に続いて**経路**を吸収する。ネットワークストレージは
  遅延と切断という新しい故障モードを持ち込み、その運用は分野06以降の
  主題につながる
