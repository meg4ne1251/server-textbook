# VFS — 「すべてはファイル」を支える抽象化層

## 概要

この章から分野03(ファイルシステム・ストレージ)に入ります。最初の章として、
Linux のあらゆるファイル操作が必ず通る共通の窓口 **VFS(Virtual File System)**
を扱います。前提知識は `01_intro/03`(ファイル階層・パーミッション・マウント)と
分野02(システムコール、fdテーブル、ページキャッシュ)です。
基準環境は Linux 7.0 / Ubuntu Server 26.04 LTS です。

## 導入 — なぜ「翻訳層」が要るのか

ディスクの上にファイルをどう並べるか——どこにファイルの中身を置き、どこに
名前の一覧を置き、どこに空き領域の記録を置くか——という**データの並べ方の
流儀**を、**ファイルシステム(filesystem)**と呼びます。そしてこの流儀は
1種類ではありません。Ubuntu の標準は ext4、大容量ストレージでよく使われる
XFS、USB メモリの FAT、ネットワーク越しの NFS、さらにはディスクを一切
持たない `/proc` や `/sys`——中身の管理方式はてんでばらばらです。

ところが、使う側は違いをまったく意識しません。`cat` は ext4 上のファイルも
USB メモリ上のファイルも `/proc/cpuinfo` も同じように読めます。`cp` で
ext4 から FAT へコピーするとき、`cp` のプログラムに「FAT 用の処理」は
1行もありません。プログラムが呼ぶのはどれも同じ `open` / `read` / `write` の
システムコールです。

この「ばらばらの流儀」と「共通のシステムコール」の間を取り持つのが、
カーネル内の抽象化層 **VFS** です。飲食店の比喩で言えば、VFS は
**共通の注文票**を定めた本部にあたります。客(プロセス)はどの店
(ファイルシステム)に対しても同じ注文票(open / read / write …)で
注文でき、注文票をどう調理するか(ディスクのどこをどう読むか)は各店の
流儀に任される——客は店の厨房の違いを知る必要がありません。

分野03全体の地図も先に描いておきます。この分野では、この共通の窓口から
出発して、ディスクへ向かって一段ずつ降りていきます。

```
  アプリケーション(cat、nginx、PostgreSQL …)
      │  open / read / write / stat …(システムコール)
┌─────▼─────────────────────────────────┐
│  VFS ── 共通の窓口・共通のオブジェクトモデル(本章)        │
├──────┬──────┬───────┬───────┬─────────┤
│ ext4  │ XFS  │ tmpfs │ /proc │  NFS    │ ← 個々のファイルシステム
│(第2〜3章)   │(本章) │(本章) │(第5章)  │
├──────┴──────┴───┬───┴───────┴─────────┤
│ ブロック層・デバイスドライバ │ ← LVM / RAID はこの層(第4章)│
└─────────▼─────────┘
        ディスク(HDD / SSD)
```

`01_intro/03` で「すべてはファイル」という思想を導入しました——デバイスも
カーネル情報も、何もかもファイルとして見せる、という UNIX の設計です。
この章を読み終えると、その思想の**実現機構**が VFS であることが分かります。
「すべてはファイル」とは、正確には「すべては VFS の注文票を受け付ける」
という意味なのです。

## 理論

本節の内容は、主に **Linux カーネルドキュメント
(Documentation/filesystems/vfs.rst)** と `man 7 inode`、および POSIX
(IEEE Std 1003.1)のファイル操作の規定に基づきます。

### VFS の位置と役割 — システムコールの一次受付

`open` / `read` / `write` / `stat` / `mkdir` ——ファイルに関わるシステム
コールは、カーネルに入るとまず **VFS のコードに着地します**。VFS は
そこで共通の前処理(パス解決、パーミッション検査、キャッシュの検索)を行い、
ファイルシステム固有の仕事だけを個々の実装に**下請けに出します**。

この構造の価値は、**変更の影響範囲の遮断**にあります。新しいファイル
システムを1つ追加しても、アプリケーションはもちろん、システムコールの
実装さえ1行も変わりません。逆に、アプリケーションはファイルシステムを
選ばず動きます。`02_process_kernel/02` で見た「POSIX = インタフェース /
ABI = 機構」の役割分担と同じ発想の、カーネル内部版と言えます——
**インタフェース(VFS)と実装(各ファイルシステム)の分離**です。

### 共通のオブジェクトモデル — 4つの登場人物

VFS が定める「共通の注文票」は、実際には**4種類のオブジェクト(構造体)と、
それぞれに紐づく操作の表**として表現されています(vfs.rst)。まず4種類の
オブジェクトの役割から押さえます。

| オブジェクト | 表すもの | 一言で |
|---|---|---|
| **superblock** | マウントされたファイルシステム1つ | 「店1軒」の台帳 |
| **inode** | ファイル1つの実体(属性) | 品物1つの台帳 |
| **dentry** | パス名の1成分(名前→inode の対応) | 品物の「名札」 |
| **file** | あるプロセスが open した1回分 | 「注文1件」の伝票 |

inode は `01_intro/03` で先取りしました——パーミッション・所有者・サイズ
などを記録する「ファイル1つにつき1個の管理台帳」です。ここで重要な補足を
すると、VFS が扱う inode は**メモリ上の共通形式**です。ディスク上の
inode の形式はファイルシステムごとに違いますが(ext4 と XFS では別物)、
ディスクから読み込むときに各実装が VFS の共通形式へ「翻訳」します。
ディスク上の形式とブロック管理の詳細は本分野の第3章
(`03_inode_block_management.md`)で扱います。

dentry(directory entry)は本章の新顔です。`01_intro/03` で「ディレクトリの
実体は名前→inode 番号の対応表」と学びました。dentry はその対応表の
**1行分をメモリ上に持ったもの**——「`syslog` という名前はこの inode を
指す」という対応1件です。なぜわざわざオブジェクトにするのかは、
内部動作の節(パス解決)で明らかになります。

file(struct file)は「open 1回分」を表します。ファイルそのもの(inode)
とは別に「open した」という事実を表すオブジェクトが要る理由——それが
分野02から持ち越した宿題の答えになります。これも内部動作の節で扱います。

### 操作の表 — 関数ポインタによる「契約」

4つのオブジェクトには、それぞれ**操作の表**が付きます。file には
file_operations(read / write / mmap …)、inode には inode_operations
(lookup / mkdir / unlink …)、superblock には super_operations という
具合です。表の中身は**関数ポインタ**、つまり「この操作を頼まれたら
この関数を呼べ」という飛び先の一覧です。

ファイルシステムを実装するとは、**この表を自分の関数で埋めて VFS に
登録すること**です(register_filesystem)。VFS は read を頼まれたら、
相手が ext4 か NFS かを一切気にせず、その file に付いている表の read の
欄に載っている関数を呼ぶだけです。オブジェクト指向言語の「インタフェースと
実装」(ポリモーフィズム)を C 言語の関数ポインタで実現した構造で、
カーネル内では VFS に限らず頻出する定石です。

登録済みのファイルシステムの一覧は `/proc/filesystems` で見えます
(実行例で確認します)。

### 契約さえ満たせば、裏はディスクでなくてもよい

この設計から、面白い帰結が出ます。VFS が要求するのは「表を埋めること」
だけで、**表の中の関数が本当にディスクを読むかどうかは問われません**。

- `/proc` — read の欄の関数は、ディスクを読む代わりに**カーネル内の
  情報をその場で文字列に整形して返します**。`/proc/cpuinfo` の中身は
  読まれる瞬間に生成されており、ディスク上のどこにも存在しません
- `/sys` — 同様に、カーネル内のデバイス情報を見せる窓です
- **tmpfs** — 実体を**メモリ(ページキャッシュ)だけ**に持つファイル
  システムです。`/dev/shm`(`02_process_kernel/05` の POSIX 共有メモリの
  住処)や `/run` がこれで、再起動で消える代わりに高速です

こうした**疑似ファイルシステム(pseudo filesystem)**の存在が、
「すべてはファイル」を可能にしています。カーネルの内部状態だろうが
デバイスだろうが、VFS の契約(操作の表)さえ用意すれば、`cat` や `grep`
という既存の道具がそのまま使える——`/proc/<PID>/status` を分野02で
散々読んできたのは、この仕組みのおかげだったのです。

## 内部動作の詳細

### 全体図 — プロセスからディスクまでの参照の鎖

4つのオブジェクトは、プロセス側から次のような参照の鎖でつながっています。

```
task_struct
   │ files(fdテーブル)
   ▼
 ┌─────┬─────┬─────┐
 │ fd 0 │ fd 1 │ fd 3 │…        fd = ただの番号(添字)
 └──┬──┴──┬──┴──┬──┘
    │      │      └──────┐
    ▼      ▼             ▼
 struct file(open 1回分: 読み書き位置、モード、参照カウント)
    │ f_path.dentry
    ▼
 dentry(名前→inode。dcache に常駐)
    │ d_inode
    ▼
 inode(ファイルの実体: 属性、操作の表、データの所在)
    │ i_sb
    ▼
 superblock(マウントされたFS 1つの台帳)──→ 実装(ext4 / tmpfs / …)──→ ディスク
```

以降、この鎖を上から順に確定させていきます。

### fd → file → inode の三層 — 分野02の宿題の回収

`02_process_kernel/01` で「fork では fd テーブルは複製されるが、
**開いたファイルの状態(読み書き位置)は親子で共有される**。二層構造の
詳細は分野03で」と保留にしました。ここで回収します。

- **fd** はただの番号で、プロセスごとの fd テーブルの添字にすぎません。
  テーブルの各欄が持つのは **struct file への参照(ポインタ)**です
- **struct file** が「open 1回分」の状態——読み書き位置(f_pos)、
  アクセスモード(読み専用か等)、そして**参照カウント**——を持ちます。
  POSIX はこの層を**オープンファイル記述(open file description)**と
  呼んで fd と明確に区別しています(`man 2 open` の NOTES にも
  この区別の解説があります)
- **inode** がファイルそのものです。同じファイルを何度 open しても
  inode は1つで、struct file が open のたびに1個作られます

fork の挙動はこの図でそのまま説明できます。**複製されるのは fd テーブル
(ポインタの並び)だけ**で、指した先の struct file は親子で同一の実体の
ままです。だから子が read して f_pos が進めば、親から見ても進んでいる——
共有されていたのは「テーブルの先」だったのです。`dup` やシェルの
リダイレクト(`2>&1`)も同様に「同じ struct file を複数の fd が指す」
状態を作ります。逆に、親子がそれぞれ**別々に open し直せば** struct file は
2個できるので、読み書き位置は独立します。

```
fork 直後:                          別々に open した場合:
親 fd 3 ──┐                        親 fd 3 ──→ file A(pos=100)┐
          ├─→ file(pos 共有)─→ inode                          ├─→ inode
子 fd 3 ──┘                        子 fd 3 ──→ file B(pos=0) ─┘
```

struct file の**参照カウント**は運用上も重要です。close は「参照を
1つ減らす」操作にすぎず、struct file が解放されるのは参照が 0 になった
ときです。さらに inode 側にも「これを参照する名前(リンク)の数」と
「open 中の参照の数」の勘定があり、**ファイルの実体が消えるのは、名前が
すべて削除され、かつ誰も open していなくなったとき**です。`rm` は名前を
消す操作(システムコール名も unlink =「名前と inode の紐を解く」)で
あって、実体を消す操作ではありません——「削除したのにディスクが減らない」
というサーバー運用の定番トラブルの正体で、トラブルシューティングで
再訪します。なお1つの inode に複数の名前を付けるハードリンクの詳細は
第3章(`03_inode_block_management.md`)で扱います。

### dentry とパス解決 — 名前の世界を歩く

`open("/var/log/syslog")` と頼まれた VFS が最初にやる仕事は、この
**パス名の文字列から inode にたどり着く**ことです。これを**パス解決
(path resolution / path walk)**と呼びます(`man 7 path_resolution`)。
手順は `01_intro/03` で学んだ原理のとおり、根から1成分ずつです。

1. ルートディレクトリ(/)の inode から出発する
2. その「名前→inode 対応表」で `var` を引き、var の inode へ進む。
   このとき、そのディレクトリの **x(通過)権限を検査**する
3. 同様に `log` を引いて進む(x 検査)
4. 最後に `syslog` を引いて目的の inode に到達する

問題は頻度です。パス解決は open だけでなく stat、exec、ディレクトリ
作成——**ほぼすべてのファイル系システムコールの前段**で走ります。
そのたびに各ディレクトリの対応表をディスクから読んでいたら、成分の数だけ
ディスク I/O が発生してしまいます(共有ライブラリのパスのような深い
階層では致命的です)。

そこで VFS は、一度解決した「名前→inode」の対応1件ずつを **dentry**
としてメモリに残します。この dentry の集合体が **dcache(dentry cache)**
です。2回目以降のパス解決は、ディスクに触れずメモリ上の dcache を
たどるだけで済みます。dcache にない成分に出会ったときだけ、VFS は
その親ディレクトリの inode_operations の **lookup** を呼び、ファイル
システム実装にディスク上の対応表を引かせて、結果を新しい dentry として
登録します。

dcache には気の利いた工夫がもう1つあります。lookup の結果が「そんな
名前はない」だった場合、**「ない」という事実も負の dentry(negative
dentry)としてキャッシュします**。プログラムは存在しないファイルを
繰り返し探すことが意外に多く(たとえばライブラリや設定ファイルの
探索パスを順に試す動き)、ENOENT を毎回ディスクに聞きに行くのは
無駄だからです。

パス解決の途中には、`01_intro/03` で学んだ2つの仕掛けが機構として
組み込まれています。

- **シンボリックリンク**: 進んだ先の inode がシンボリックリンクなら、
  その中身のパス文字列に読み替えて解決を続けます(読み替え回数には
  上限があり、超えると ELOOP エラーになります——リンクのループ対策です)
- **マウントポイント**: dentry には「ここには別のファイルシステムが
  接ぎ木されている」という印が付けられます。パス解決がそこを通ると、
  VFS は**進む先をそのファイルシステムの根の dentry にすり替えます**。
  `01_intro/03` の「マウント=接ぎ木」の実体はこのすり替えであり、
  木が一本に見えるのは、パス解決が縫い目を自動で乗り越えているからです

なお、パス解決はあらゆる操作の前段だけに徹底的に最適化されており、
ロックをほぼ取らずに dcache を歩く高速経路(RCU-walk)と、取りながら
歩く通常経路(ref-walk)の2段構えになっています。詳細は **カーネル
ドキュメント(Documentation/filesystems/path-lookup.rst)** に譲りますが、
「カーネルで最も頻繁に走る経路の1つなので特別扱いされている」ことだけ
覚えておいてください。

### open の一部始終 — 通行証の発行

道具が揃ったので、open システムコールの全体を通して見ます。

1. **パス解決** — dcache(+ミス時は lookup)で目的の inode に到達する。
   途中の各ディレクトリで x 権限を検査する
2. **パーミッション検査** — 到達した inode の許可ビットと、プロセスの
   UID/GID、要求モード(読みたいのか書きたいのか)を照合する。
   `01_intro/03` の「検査は open の瞬間に一度だけ」の、その一度がここです
3. **struct file の生成** — 読み書き位置 0、要求されたモードで伝票を
   起こし、操作の表(file_operations)を inode から引き継ぐ
4. **fd の割り当て** — fd テーブルの**空いている最小の番号**に struct
   file への参照を置き、その番号を返す(「最小」の保証が、`01_intro/02`
   のリダイレクトで見た「fd 1 を閉じてから open すると 1 に入る」の根拠です)

以後の read / write は fd から始まる参照の鎖をたどるだけで、パス解決も
パーミッション検査もしません。「fd は入場済みの通行証」(`01_intro/03`)の
機構的な意味がこれです——検問(2)は発行時に済んでおり、通行証を持つ者は
素通りできます。

### read / write とページキャッシュ — 分野02の宿題の回収

`02_process_kernel/03` で「一度読んだファイルの中身はページキャッシュに
残る。詳細は分野03で」と予告しました。read / write の内部がその答えです。

**read(fd, buf, n)** の流れ:

1. fd → struct file → file_operations の read 系の欄をたどる。ほとんどの
   ディスクファイルシステムは、ここで VFS が用意する**汎用実装**に乗る
2. 汎用実装は、要求された範囲の**ページキャッシュを引く**。ページ
   キャッシュとは「この inode のオフセット何ページ目はこの物理ページ
   フレームにある」という対応の集合です
3. **ヒット**したら、そのフレームからユーザーのバッファへコピーして
   即座に戻る——ディスク I/O ゼロ。これが「2回目の cat は速い」の正体です
4. **ミス**したら、フレームを確保してファイルシステム実装に「この
   ページ分をディスクから読み込め」と依頼し、完了までタスクは **S 状態
   (または D 状態)で眠ります**。割り込みで I/O 完了が通知されると
   起こされ、コピーして戻ります——分野02で学んだ「ブロッキング read の
   一部始終」の、ブロックしている中身がこれです

**write** はさらに割り切っています。ユーザーのデータを**ページキャッシュの
フレームに書き込み、「ダーティ(dirty、ディスクと食い違っている)」の
印を付けたら、ディスクを待たずに即座に戻ります**。実際にディスクへ書き
戻す仕事(ライトバック)は、カーネルのスレッドが後から非同期に行います。

この遅延には見覚えがあるはずです——デマンドページングも CoW も
「今やらなくていいことは、必要になるまでやらない」でした。write の
遅延書き戻しはその I/O 版で、(a) アプリは遅いディスクを待たない、
(b) 同じページへの連打が1回の書き戻しにまとまる、(c) 連続した書き込みを
まとめて効率よく書ける、という利益があります。**代償は、write が戻った
時点ではデータがまだメモリにしかない**こと——電源断で失われます。
「この write は本当にディスクに固定してくれ」と明示的に頼むシステム
コールが **fsync** で(`man 2 fsync`)、データベースがトランザクションの
確定時に必ず呼ぶのはこのためです。「書き戻しの途中で電源が落ちたら
ファイルシステム自体は壊れないのか?」という当然の疑問が、次章
(ジャーナリング)の主題になります。

もう1つ、`02_process_kernel/03` で学んだ **mmap** との関係も、ページ
キャッシュで統一的に説明できます。ファイルを mmap したプロセスのページ
テーブルには、**ページキャッシュのフレームそのものを指す PTE** が
書かれます。つまり read / write 経由でも mmap 経由でも、見ている実体は
同じ1枚のフレームです。共有ライブラリが全プロセスで1組で済むのは、
全員の PTE が同じページキャッシュのフレームを指しているからでした。

### superblock とマウント — 店1軒の台帳

参照の鎖の最下段、superblock は**マウントされたファイルシステム1つ分の
台帳**です。ブロックサイズ、ルートディレクトリの inode、空き容量の
勘定、そしてダーティな inode の一覧(ライトバックの巡回先)などを
持ちます。mount システムコールの実体は、(1) 指定されたデバイスから
ファイルシステム実装に superblock を組み立てさせ、(2) その根の dentry を
指定されたディレクトリの dentry に接ぎ木する(前述の「すり替え」の印を
付ける)ことです。どのプロセスにどの木を見せるかを分ける「マウント
ネームスペース」という発展形がありますが、これはコンテナの土台として
分野05(`05_virtualization_containers/01`)で扱います。

## 実行例 — 参照の鎖とキャッシュを観察する

前提は Ubuntu Server 26.04 LTS です。

登録済みのファイルシステム一覧。`nodev` が付いているものは、裏に
デバイス(ディスク)を持たない疑似ファイルシステムです:

```console
$ cat /proc/filesystems | head -8
nodev   sysfs
nodev   tmpfs
nodev   proc        ← /proc 自身もここに登録された一実装にすぎない
nodev   devtmpfs
        ext4        ← ディスクを持つ実装には nodev が付かない
...
$ findmnt -t proc,tmpfs,ext4 -o TARGET,FSTYPE | head -5
TARGET     FSTYPE
/          ext4
/proc      proc
/run       tmpfs
/dev/shm   tmpfs
```

struct file の読み書き位置(f_pos)が実在することを、`/proc` 経由で
確認します。fd 3 でファイルを開き、100 バイト読むと:

```console
$ exec 3< /etc/passwd              ← fd 3 で open(struct file が1個できる)
$ head -c 100 <&3 > /dev/null      ← fd 3 から 100 バイト読む
$ grep ^pos /proc/$$/fdinfo/3
pos:    100                        ← struct file が覚えている読み書き位置
$ exec 3<&-                        ← close(参照カウントが減る)
```

ページキャッシュの効果。同じファイルの2回目の読み込みはディスクに
触れません:

```console
$ time cat /usr/bin/snap > /dev/null     ← 1回目: ディスクから(ミス)
real    0m0.184s
$ time cat /usr/bin/snap > /dev/null     ← 2回目: ページキャッシュから
real    0m0.009s
$ free -h | head -2

Mem:           7.8Gi   890Mi   5.1Gi  1.0Mi   1.8Gi       6.9Gi
                                              ↑ ページキャッシュ等はここに計上
```

dcache の規模感(dentry の総数と、そのうち未使用でいつでも回収できる数):

```console
$ cat /proc/sys/fs/dentry-state

↑ 総数   ↑ 未使用(回収可能)          ↑ 負の dentry の数
```

「削除済みなのに open 中」のファイルは、`/proc` の fd のリンク先に

## トラブルシューティング

- **ログを rm したのに df の使用量が減らない**: 前述のとおり、rm
  (unlink)は名前を消すだけで、**open 中の fd が残っている限り inode と
  データは解放されません**。`lsof +L1`(リンク数 0 なのに open 中の
  ファイルを列挙)か `ls -l /proc/*/fd | grep deleted` で犯人のプロセスを
  特定し、そのプロセスを再起動(またはログ再オープンのシグナルを送る)
  すれば解放されます。ログの肥大対応で最初に疑う定番です
- **Too many open files(EMFILE)**: fd テーブルには上限があります
  (`ulimit -n`、既定 1024 のことが多い)。上限到達の大半は「close 忘れ
  (fd リーク)」で、`ls /proc/<PID>/fd | wc -l` で本当に使っている数を
  数え、増え続けるならリークです。上限を上げる前に、まず数えるのが順序です
- **write は成功したのに、電源断でデータが消えた**: write の成功は
  「ページキャッシュに書けた」ことしか意味しません。確実に固定したい
  地点では fsync(またはアプリの同期書き込み設定)が必要です。
  ファイルシステム自体の整合性がどう守られるかは次章で扱います
- **df は空きがあるのに No space left on device**: 容量とは別に inode の
  個数にも上限があるファイルシステムが多く(ext4 は作成時に固定)、
  小さいファイルを大量に作ると inode 側が先に枯渇します。`df -i` で
  IUse% を確認してください。詳細は第3章で扱います
- **マウントしたらディレクトリの中身が見えなくなった**: マウントは
  接ぎ木なので、マウントポイントの**下にあった元のファイルは隠れます**
  (消えてはいません。アンマウントすれば再び見えます)。「df の合計と
  du の合計が合わない」ときの一因でもあります
- **`/proc` のファイルが ls -l でサイズ 0 なのに cat で読める**: 疑似
  ファイルシステムの中身は読まれる瞬間に生成されるため、stat の時点では
  サイズが分かりません(0 と報告されます)。「サイズ 0 = 空」が成り立つ
  のは通常のファイルだけです。`/proc` を定期収集するスクリプトが
  「サイズ 0 だからスキップ」して空振りする、は実際にある落とし穴です

## 演習・確認問題

1. `cat /etc/hostname` の1回の実行で、VFS の4オブジェクト(superblock /
   inode / dentry / file)がそれぞれどこで登場するかを、open から close
   までの時系列で説明してください
2. fork した親子が読み書き位置を共有する理由と、親子がそれぞれ open
   し直すと位置が独立する理由を、fd → struct file → inode の三層の言葉で
   説明してください
3. 「rm したのに df が減らない」が起きる条件と解消方法を、unlink・
   参照カウント・fd の言葉で説明してください。ゾンビプロセス
   (`02_process_kernel/01`)の「終わったのに残る」と構図がどう似ているか
   も考えてみてください
4. write システムコールが一瞬で戻るのに fsync が別に存在する理由を、
   ページキャッシュとダーティの言葉で説明してください。この設計は
   分野02で学んだどの「遅延の芸」と同じ発想ですか
5. `/proc/cpuinfo` はディスク上のどこにも存在しないのに cat で読めます。
   これが可能な理由を「操作の表(契約)」の言葉で説明し、「すべては
   ファイル」という思想と VFS の関係を1〜2文で述べてください

## まとめ

- VFS はシステムコールとファイルシステム実装の間の抽象化層。共通の
  オブジェクトモデル(superblock / inode / dentry / file)と関数ポインタの
  表(契約)により、アプリは実装の違いを知らずに済む。「すべてはファイル」は
  この契約の別名であり、/proc や tmpfs は契約だけ満たす疑似ファイルシステム
- fd はただの番号、struct file が「open 1回分」(読み書き位置・参照
  カウント)、inode がファイルの実体。fork で共有されるのは struct file、
  rm(unlink)が消すのは名前だけ——実体は参照がすべて絶えたときに消える
- パス解決は1成分ずつの表引きで、dcache(負の dentry 含む)が
  ディスク I/O を省く。シンボリックリンクの読み替えとマウントの縫い目の
  乗り換えもこの経路に組み込まれている
- read / write はページキャッシュを介する。read のヒットは I/O ゼロ、
  write はダーティを付けて即帰りし、書き戻しは非同期——固定の保証が
  必要な地点では fsync を使う
- 観察は /proc/filesystems、findmnt、/proc/PID/fdinfo(pos)、
  /proc/PID/fd(deleted)、free の buff/cache、df -i で行う
