# inode とブロック管理 — 台帳の実物と置き場所の決め方

## 概要

これまで inode を「ファイル1つ分の管理台帳」と呼んできました(`01_intro/03`)。
この章はその台帳の実物——ディスク上のどこに、どんな中身で置かれ、ファイルの
中身のブロックをどう指すのか——を扱います。前提知識は `01_intro/03`(inode と
「名前 → inode」の対応表)、前々章(VFS、unlink と参照カウント)、前章
(ジャーナリング)です。基準環境は Linux 7.0 / Ubuntu Server 26.04 LTS、
題材は ext4(対比として XFS)です。

## 導入 — 台帳に残された2つの宿題

`01_intro/03` で導入した inode の理解を、ここまでの章で少しずつ使ってきました。
「名前は inode に含まれず、ディレクトリが『名前 → inode 番号』の対応表を持つ」
「rm が消すのは名前で、実体は参照カウントが尽きたときに消える」(前々章)。
しかし、よく考えると2つの問いにまだ答えていません。

1. **inode は中身をどう指しているのか。** inode には所有者・パーミッション・
   サイズなどの属性が入っていると説明しましたが、肝心の**ファイルの中身**は
   inode の中にはありません。中身はディスク上のデータブロックに置かれます。
   では、「このファイルの先頭から 4096 バイト目以降はどのブロックにあるか」を、
   inode はどうやって知っているのでしょうか(**マッピング**の問題)
2. **新しいブロックや inode は、どうやって選ばれるのか。** ファイルに追記
   すると新しいブロックが割り当てられます。ディスクには何億ものブロックが
   ありますが、「どこが空いているか」をどう管理し、空きの中から**どこを**
   選ぶのでしょうか(**割り当て(アロケーション)**の問題)

この2つが本章の主題です。マッピングは「読むとき」の性能を、割り当ては
「書くとき」の性能と将来の断片化を決めます。そして両方の答えは、
ハードディスクという装置の物理的な性質——**近い場所はまとめて読めるが、
離れた場所への移動(シーク)は遅い**——への適応として設計されてきました。

もう1つ、`01_intro/03` からの持ち越しがあります。「名前 → inode」の対応が
1対1とは限らない——**1つの inode に複数の名前を付けられる**(ハードリンク)
——ことの仕組みも、台帳の実物を見たこの章で回収します。

## 理論

本節の内容は、主に **Linux カーネルドキュメントの ext4 ディスクフォーマット
仕様(Documentation/filesystems/ext4/ 以下)**、`man 7 inode`、`man 5 ext4`、
および XFS のドキュメント(Documentation/admin-guide/xfs.rst)に基づきます。

### ディスクの区画整理 — ブロックグループ

ext4 は、デバイス全体を先頭から**ブロックグループ(block group)**という
固定サイズの区画に等分します。ブロックサイズ 4 KiB の標準構成では
1グループ = 32768 ブロック = **128 MiB** です(32768 という数は「4 KiB の
ビットマップ1ブロックで管理できるビット数」から来ています)。各グループは
自分の区画に関する台帳一式を自分の中に持ちます。

```
 デバイス全体
 ┌──────────────┬──────────────┬──────────────┬───
 │ グループ 0    │ グループ 1    │ グループ 2    │ …
 └──────────────┴──────────────┴──────────────┴───
   1グループの中身(128 MiB)
 ┌───────────┬──────────┬─────────┬─────────┬──────────┬──────────────┐
 │スーパー    │グループ   │ブロック  │inode    │ inode    │ データブロック │
 │ブロック(*) │記述子(*)  │ビットマップ│ビットマップ│ テーブル  │              │
 └───────────┴──────────┴─────────┴─────────┴──────────┴──────────────┘
  (*) 全グループにはなく、一部のグループにだけ複製が置かれる(sparse_super)
```

- **スーパーブロック** — ファイルシステム全体の台帳(総ブロック数、
  ブロックサイズ、機能フラグ…)。前々章で VFS のオブジェクトとして見た
  ものの、ディスク上の実体です。壊れると全体が読めなくなるため、
  一部のグループに複製が置かれます
- **ブロックビットマップ** — グループ内の各ブロックが使用中か空きかを
  1ブロック=1ビットで記録する一覧表
- **inode ビットマップ** — 同じことを inode について記録する一覧表
- **inode テーブル** — inode の実物がぎっしり並んだ配列。**inode 番号とは、
  実はこの全グループ通しの配列の添字**です。番号が分かれば、計算だけで
  ディスク上の位置が決まります(探索は要りません)

なぜ全体を1つの大きなビットマップと1つの大きな inode テーブルで管理せず、
わざわざ区画に分けるのでしょうか。答えは**局所性(locality)**です。
グループという単位があることで、「ファイルの inode と、その中身のデータ
ブロックを、なるべく**同じグループに**割り当てる」「同じディレクトリの
ファイルたちを近くに置く」という方針が実装できます。台帳(inode)を読んで
から中身(データ)を読む、という必ず連続する2つのアクセスが、ディスク上でも
近所で済む——回転する円盤とシークの時代に生まれた設計ですが、「関連する
ものは近くに、まとめて」という原則は SSD でも(並べ替えやまとめ書きの
効率として)意味を持ち続けています。

なお現代の ext4 は **flex_bg** という機能で、複数グループ(既定16個)分の
ビットマップと inode テーブルを先頭のグループに寄せて置きます。台帳が
一か所に固まることで台帳自身の読み書きが連続になり、残りのグループが
まるごとデータ用の連続領域になる——局所性の原則を一段大きな単位で
適用し直したものです。

### マッピング(1) — 旧来の答え: 間接ブロック

では inode はどうやって中身を指すのか。ext4 の前身(ext2/ext3)の方式から
見るのが理解の近道です。inode には中身の位置を書くための領域が 60 バイト
(32 ビットのブロック番号 15 個分)あります。ext2/ext3 はここを
次のように使いました。

- 先頭 **12 個** — ファイルの先頭 12 ブロック(48 KiB)の番号を**直接**書く
- 13 個目 — **間接ブロック(indirect block)**の番号。ブロック番号だけが
  1024 個並んだ「目次専用ブロック」を1枚挟み、そこから先の 4 MiB 分を指す
- 14 個目 — **二重間接**。目次の目次(1024 × 1024 ブロック = 4 GiB 分)
- 15 個目 — **三重間接**。目次の目次の目次(4 TiB 分)

小さいファイルは inode だけで完結し、大きくなるほど目次の階層が深くなる
——多段ページテーブル(`02_process_kernel/03`)とそっくりな、「疎に、
必要な分だけ表を持つ」設計です。

しかしこの方式には構造的な弱点があります。**ブロック1つにつき番号を
1つ書く**ため、たとえ 1 GiB のファイルが完全に連続した領域に置かれて
いても、26 万個あまりのブロック番号を延々と記録しなければなりません。
「連続していること」を表現する語彙がないのです。

### マッピング(2) — 現在の答え: エクステント

ext4 の標準方式は**エクステント(extent)**です。エクステントは
「**論理ブロック番号 L から始まる長さ N の範囲は、物理ブロック番号 P から
連続して置いてある**」という(L, N, P)の3つ組で、**連続した範囲を1レコードで**
表します。1 GiB の連続ファイルなら、理屈の上ではレコード1件で済みます
(実際のエクステント1件の上限は 32768 ブロック = 128 MiB です)。

inode 内の 60 バイトには、エクステントが**4件まで**そのまま入ります。
大半のファイルは4件で足りるため、inode を読んだ時点でマッピングも手に
入ります。断片化などで4件を超えると、間接ブロックと同じ発想で
「エクステントの目次ブロック」を挟んだ**エクステントツリー**(B木)に
成長します。

```
        inode 内(60バイト)                エクステントが増えたら…
 ┌────────────────────────┐        ┌──────────┐
 │ ヘッダ                  │        │ ヘッダ    │  inode 内は「索引」になり
 │ (L=0,   N=8,  P=8461)  │        │ 索引 ──────→ 目次ブロック(エクステント
 │ (L=8,   N=120,P=9000)  │        │ 索引 ──────→ が何百件も並ぶ)を指す
 │ (空き)                  │        │  …       │
 │ (空き)                  │        └──────────┘
 └────────────────────────┘
   ファイルの中身が連続していれば、ツリーは浅いまま
```

ここで前章の宿題を回収します。エクステントには**未書き込み(unwritten)**
という印を付けられます。「ブロックは割り当て済みだが、中身はまだ書かれて
いない。読まれたらゼロを返せ」という状態です。`fallocate` システムコール
(`man 2 fallocate`)による領域の事前確保はこれを使います——確保の時点で
ブロックを押さえる(容量不足で後から失敗しない・なるべく連続で取れる)一方、
確保しただけの範囲を読んでも、そのブロックに以前入っていた他人のデータは
決して見えません。前章で「XFS は古いデータの露出を unwritten で防ぐ」と
述べたのと同じ仕掛けを、ext4 もエクステントの属性として持っているのです。

逆に、**割り当てないことで表現する**ものもあります。書かれたことのない
範囲(マッピングに対応レコードがない「穴」)を持つファイルを
**スパースファイル(sparse file)**と呼びます。読むとカーネルがゼロを
返すだけで、ディスクは1ブロックも消費しません。仮想マシンのディスク
イメージ(「1 TiB のディスク」を作っても実使用分しか消費しない)が
代表例です。`ls -l` のサイズ(論理サイズ)と `du` の使用量(実割り当て)が
食い違う正体がこれです。

### 割り当て — どこに書くかを決める仕事

新しいブロックが必要になったとき、ext4 のアロケータ(mballoc:
multiple block allocator)は、ビットマップから単に「最初に見つかった
空き」を取るのではなく、次の方針で**場所を選びます**。

- **目標(goal)から探す** — そのファイルの既存の最終ブロックの続き、
  それがなければ inode と同じ(flex_bg)グループ、を第一候補にする。
  局所性の原則の実装です
- **1個ずつではなく、まとめて取る** — 追記が続きそうなファイルには
  要求より大きめの連続領域を予約(プリアロケーション)し、後続の追記を
  その中で受ける。細切れの割り当てが交互に起きてファイルどうしが
  縞模様に混ざる(断片化の典型)のを防ぎます

そして割り当ての効果を最大化しているのが、前々章のライトバックと
組み合わさった**遅延割り当て(delayed allocation)**です。write の時点では
ブロックを割り当てず、「このページはいずれ書く」という予約だけしておき、
**実際の割り当てをライトバック(書き戻し)の瞬間まで遅らせます**。書き
戻しの時点では「このファイルにこれから何ブロック書くのか」の全体像が
見えているため、ぴったりの大きさの連続領域を一発で選べます。write の
たびに少しずつ割り当てていたら、こうはいきません。「遅らせると、
まとめられて、賢く決められる」——デマンドページング、ライトバック、
XFS の遅延ロギングと同じ、この分野で繰り返し現れる設計原理です。

なお、前章のトラブルシューティングで見た「rename 置き換え直後のクラッシュで
0 バイト」問題は、この遅延割り当てが関与しています。write 済みでも割り当て
すら済んでいないデータは書き戻しの列に並ぶのが遅くなりがちで、メタデータ
(rename)だけが先にコミットされる窓が広がるためです。`auto_da_alloc` に
よる緩和(該当パターンの検出で書き戻しを先行)の背景がこれです。

### 名前の側の台帳 — ディレクトリとハードリンク

最後に、`01_intro/03` 以来の「名前 → inode」の対応表そのものを見ます。
ディレクトリも実体は「中身がデータブロックにあるファイル」で、その中身が
対応表です。1行(ディレクトリエントリ)は「inode 番号、レコード長、
名前の長さ、種別、名前」の可変長レコードで、ブロック内に前から詰めて
並びます。

ここで、**1つの inode 番号を複数の行が指してよい**ことに注目してください。
`ln A B` で作られる**ハードリンク(hard link)**は、既存の inode 番号を
指す行をもう1つ書き足すだけの操作です。inode 側には**リンクカウント**
(参照している名前の数)が記録され、`ln` で増え、unlink(rm)で減ります。
前々章で見た「rm は名前を消すだけ。実体はリンクカウントが 0 になり、かつ
誰も開いていなくなったときに消える」の、ディスク側の帳尻がこの数字です。
2つの名前は完全に対等で、「本体と分身」の区別はありません——どちらも
ただの行です。この点、「別のパス名を中身に持つ特殊ファイル」である
シンボリックリンク(`01_intro/03`)とは仕組みがまったく違います。

ハードリンクには構造由来の制約が2つあります。

- **ファイルシステムをまたげない**(EXDEV)。行に書けるのは inode
  **番号**だけで、番号は同じファイルシステム内でしか意味を持たないため
- **ディレクトリには作れない**(EPERM)。許すと親子関係が循環し、
  ディレクトリツリーが木でなくなってしまう(パス解決やツリー走査が
  無限ループしうる)ため、カーネルが禁じています。例外はシステムが
  自動で管理する `.`(自分)と `..`(親)で、これらもハードリンクの
  一種です。ディレクトリのリンクカウントが「2+サブディレクトリ数」に
  なるのは、自分の名前+自分の `.`+各子の `..` を数えた結果です

## 内部動作の詳細

### ext4 の inode の中身

ディスク上の inode(struct ext4_inode)は既定 **256 バイト**の固定長
レコードです。主なフィールドを挙げます(Documentation/filesystems/ext4/
の inode の節に基づく)。

| フィールド | 内容 | これまでの章との対応 |
|---|---|---|
| i_mode | 種別(通常/ディレクトリ/リンク…)+パーミッション 12 ビット | `01_intro/03` の 9+3 ビット |
| i_uid / i_gid | 所有者 UID / GID | `01_intro/03` |
| i_size | 論理サイズ(バイト) | スパースなら実割り当てと乖離 |
| i_atime / i_mtime / i_ctime | 参照/内容変更/属性変更の日時 | `stat` の3つの時刻 |
| i_crtime | 作成日時(256 バイト inode の拡張領域) | `stat` の Birth |
| i_links_count | リンクカウント | 本章。0 で実体の回収候補に |
| i_blocks | 実際に割り当て済みのブロック数(512 バイト単位) | `du` が数えるもの |
| i_flags | エクステント使用、immutable 等のフラグ | |
| i_block[15](60 バイト) | マッピングの根(下記) | 本章の主役 |

要点は2つあります。第一に、**60 バイトの i_block 領域は用途が可変**です。
エクステントツリーの根(現代の通常ファイル)、間接ブロック方式の 15 個の
番号(ext2/3 互換)のほか、**短いシンボリックリンクはリンク先のパス文字列を
ここに直接書きます**(fast symlink——60 文字未満のリンク先ならブロックを
1つも消費しない)。`01_intro/03` で「別のパス名を中身として持つ」と説明した
中身の置き場所は、実はたいてい inode の中なのです。

第二に、**inode の総数はファイルシステム作成時に固定**されます。inode
テーブルは mkfs の時点で領域を確保し尽くす配列であり、後から増えません
(既定ではおおよそ 16 KiB につき 1 inode の比率。`mkfs.ext4 -i` で変更)。
ブロックに空きがあっても inode が尽きれば、それ以上ファイルは作れません
——`df` は空きがあるのに書けない、という後述の典型症状の原因です。

### 巨大ディレクトリと htree

ディレクトリエントリの素朴な線形リストは、エントリが数万〜数百万になると
名前1つの探索に全ブロックの走査が必要になり、破綻します。ext4 は
`dir_index` 機能(既定で有効)により、一定サイズを超えたディレクトリを
**htree**——名前のハッシュ値をキーにした木——に切り替えます。名前から
ハッシュを計算し、木をたどって候補ブロックを1つに絞る。深さは実用上
最大2段で、数百万エントリでも読むブロックは数個で済みます。

観察できる副作用が1つあります。htree 化したディレクトリの `ls` の既定
出力や readdir の列挙順は、作成順でも辞書順でもなく**ハッシュ順**に
なります(`ls` は既定でソートし直すので、`ls -f` や find で見えます)。
「巨大ディレクトリの列挙順が意味不明」は故障ではなく htree の仕様です。

もう1つ、線形リスト部分の削除の実装は示唆的です。エントリを消すとき、
ext4 は**行を詰め直しません**。直前のエントリのレコード長を伸ばして
「跡地を併合」するだけで、消した名前の文字列自体はゴミとして残ります
(次の作成で再利用されるまで)。前々章の「rm は名前を消すだけ」は、
ディスクのバイト列のレベルでは「名前すら消えていない」のです。

### 割り当ての帳尻とクラッシュ — 前章との接続

ビットマップ・inode・エクステントツリーは、まさに前章で見た「1操作で
複数ブロックの更新を要する台帳ども」です。ブロックを1つ割り当てる操作は
ブロックビットマップ+グループ記述子の空き数+inode(エクステント)の
最低3か所を書き、そのすべてが jbd2 のトランザクションに束ねられます。
本章の台帳群は、前章のジャーナリングが守っている対象の実物にほかなりません。

1つだけ、ジャーナルでは守り切れない隙間があります。「unlink 済みだが
まだ開かれているファイル」(前々章)です。リンクカウント 0・実体ありの
状態は正常ですが、その最中にクラッシュすると、再起動後は誰もその inode を
参照していないのに使用中、という宙に浮いた状態になります。ext4 はこれを
**孤児リスト(orphan list)**——「削除待ちの inode」の一覧をスーパー
ブロックから辿れる形でディスクに記録しておく仕組み——で処理し、
リカバリ時(前章の再生の後)にまとめて回収します。

### XFS の流儀 — ビットマップではなく B+木

前章に続き、XFS を対比として見ると設計の輪郭がはっきりします。

- **区画の目的が違う。** XFS の区画は**アロケーショングループ(AG)**と
  呼ばれ、既定で数 GiB と大きめです。主目的は局所性よりも**並列性**——
  AG ごとに独立した空き領域管理を持つため、複数 CPU が別々の AG で
  同時に割り当てを進められます。メタデータ並列性に強い(前章)の実体です
- **空き領域はビットマップではなく B+木で持つ。** 各 AG は空き領域を
  「開始位置順」と「サイズ順」の**2本の B+木**で二重に索引します。
  「位置 X の近くの空き」も「長さ N 以上の連続した空き」も対数時間で
  引ける——ビットマップの線形走査と違い、**連続領域を探す**という
  アロケータの本業に最適化されたデータ構造です
- **inode は動的に作られる。** ext4 と違い固定の inode テーブルを持たず、
  必要に応じて inode 用の領域を確保し、B+木で索引します。「inode が
  尽きて書けない」が原理的に起きにくい設計です(そのぶん inode 番号から
  位置が単純計算では出ず、木を引きます)
- マッピングがエクステントであること、遅延割り当てを行うことは ext4 と
  共通です(歴史的には XFS が先行し、ext4 が追随した機能群です)

ext4 は「固定配置+ビットマップ+計算で引ける inode」という素直で頑健な
構成、XFS は「動的配置+B+木ずくめ」というスケール志向の構成——前章の
物理/論理ロギングの対比と同様、同じ問題への異なる流儀として整理できます。

## 実行例 — 台帳の中身を観察する

前提は Ubuntu Server 26.04 LTS(ext4)です。

inode 番号とリンクカウント。ハードリンクを作ると、同じ inode 番号を
2つの名前が指し、Links が増えます:

```console
$ echo hello > A
$ ln A B
$ ls -li A B
524401 -rw-rw-r-- 2 user user 6 ... A    ← 先頭が inode 番号。完全に同じ
524401 -rw-rw-r-- 2 user user 6 ... B    ←(「2」がリンクカウント)
$ rm A
$ cat B                                  ← 名前を1つ消しても実体は無事
hello
```

stat で inode の主なフィールドが読めます。Size(論理)と Blocks
(実割り当て)の区別に注意してください:

```console
$ truncate -s 1G sparse.img              ← 1 GiB の「穴だけ」のファイル
$ stat sparse.img
  Size: 1073741824      Blocks: 0          IO Block: 4096   regular file
                                ↑ 1バイトも割り当てられていない
$ du -h sparse.img; ls -lh sparse.img
0       sparse.img                       ← du は実割り当てを数える
-rw-rw-r-- 1 user user 1.0G ... sparse.img  ← ls は論理サイズ
```

エクステントの実物は filefrag で見えます。fallocate で確保した範囲には
unwritten の印が付きます:

```console
$ fallocate -l 8M prealloc.dat
$ filefrag -v prealloc.dat
 ext:   logical_offset:   physical_offset: length:   expected: flags:
   0:      0..    2047:  34816..   36863:   2048:             last,unwritten,eof
                          ↑ 2048ブロック(8MiB)が1エクステントで連続確保。
                            中身は未書き込み(読めばゼロ)
```

inode の使用状況。ブロックの空き(df)とは独立の資源であることが
分かります:

```console
$ df -i /
Filesystem      Inodes  IUsed   IFree IUse% Mounted on
/dev/vda2      6291456 218452 6073004    4% /
                ↑ 総数は mkfs 時に固定。ブロックが余っていても IFree が
                  尽きればファイルは作れない
```

## トラブルシューティング

- **「No space left on device」なのに `df` では空きがある**: まず
  `df -i` を確認します。IUse% が 100% なら inode の枯渇です。典型は
  メールキュー・セッションファイル・キャッシュなど「小さいファイルを
  無数に作る」ディレクトリ。対処は不要ファイルの削除で、恒久策は
  作成時の `-i` 調整(作り直し)か、そうした用途を想定した設計
  (XFS の採用も選択肢)です
- **`ls -l` の合計と `du` が一致しない**: どちらも正しく、測っている
  ものが違います(論理サイズ vs 実割り当て)。du が小さければスパース
  ファイル、逆に大きければ削除済みだが開かれたままのファイル
  (前々章)や、間接的な消費を疑います。スパースファイルは cp や
  バックアップの仕方によっては穴が実データに膨らむ(1 TiB の穴が
  本当に 1 TiB 書かれる)点にも注意してください
- **巨大ディレクトリの操作が遅い・列挙順がおかしい**: 数十万エントリ
  規模なら htree が働いていても readdir 全体の列挙・stat の繰り返しは
  重い操作です。列挙順のハッシュ順は仕様(本文)。設計としては
  1ディレクトリに溜めず階層に分散するのが定石です。また、エントリを
  大量削除してもディレクトリ自身のサイズは自動では縮みません
- **断片化を心配すべきか**: 遅延割り当てとエクステントにより、通常運用の
  ext4/XFS で断片化が問題になることはまれです。長期間の追記を続けた
  ファイルの状態は `filefrag`(ext4)や `xfs_bmap`(XFS)で確認でき、
  必要なら `e4defrag` がありますが、「まず測ってから」が原則です
- **ハードリンクを別パーティションに作れない(EXDEV)**: 仕様です。
  inode 番号はファイルシステム内でしか通用しません(本文)。またぐ
  必要があるならシンボリックリンクかコピーを使います。`mv` が同一
  ファイルシステム内では一瞬(rename = ディレクトリの行の書き換え)
  なのに、またぐと時間がかかる(コピー+削除に切り替わる)のも
  同じ理由です
- **リンクカウントが 0 にならないのに実体が消えない/消えたのに容量が
  減らない**: 後者は前々章のとおり「開いている誰か」がいます
  (`lsof +L1` で「リンク 0 だが open 中」のファイルを一覧できます)。
  クラッシュを挟んだ場合の回収は孤児リストが担うため、fsck を待たず
  次回マウントで解消されます

## 演習・確認問題

1. ブロックグループという区画がなぜ存在するのか、「inode を読んでから
   中身を読む」というアクセスの連なりと局所性の観点から説明してください。
   flex_bg はその原則をどう発展させたものですか
2. 間接ブロック方式とエクステント方式のマッピングを比べ、「1 GiB の
   完全に連続したファイル」を表現するのに各方式が何を記録する必要が
   あるかを述べてください。エクステントの (L, N, P) はそれぞれ何ですか
3. 遅延割り当てが「割り当ての賢さ」をどう向上させるかを、write 時点と
   ライトバック時点で分かっている情報の差から説明してください。また、
   この設計が前章の「rename 置き換えで 0 バイト」問題とどう関係するかを
   述べてください
4. `fallocate` で確保しただけの領域を読んでもゼロしか見えないのは
   なぜですか。unwritten エクステントが防いでいる情報漏えいの筋書きと
   合わせて説明してください
5. ハードリンクとシンボリックリンクの違いを、ディレクトリエントリ・
   inode 番号・リンクカウントの言葉で説明してください。ハードリンクが
   ファイルシステムをまたげない理由と、ディレクトリに作れない理由も
   それぞれ述べてください
6. 「df では空きがあるのに No space left on device」という症状の原因を
   特定する手順と、その原因が ext4 の設計(inode テーブルの固定確保)に
   どう由来するかを説明してください。XFS ではなぜ起きにくいのですか

## まとめ

- ext4 はデバイスをブロックグループ(128 MiB)に区画し、各グループが
  ビットマップと inode テーブルを持つ。inode 番号はテーブルの添字で、
  位置は計算で決まる。区画の目的は「台帳と中身を近くに置く」局所性
- マッピングの現代の答えはエクステント——「L から N 個は P から連続」の
  3つ組で連続範囲を1レコードで表す。inode 内に4件、超えると B木に成長。
  unwritten の印が事前確保(fallocate)と古いデータの露出防止を両立し、
  レコードのない「穴」がスパースファイルを表現する
- 割り当ては「目標から・まとめて・遅らせて」。遅延割り当てはライトバック
  時点の全体像で連続領域を一発確保する——「遅らせるとまとめられて賢く
  なる」というこの分野の反復原理の一例
- 名前の側は可変長のディレクトリエントリの列(大きくなると htree)。
  ハードリンクは同じ inode 番号を指す行の追加にすぎず、帳尻はリンク
  カウントが取る。FSをまたげない・ディレクトリ不可は構造由来の制約
- XFS は AG による並列性、位置順+サイズ順の2本の B+木による空き管理、
  動的 inode と、同じ問題を B+木ずくめで解く。ext4 の固定配置+
  ビットマップとの対比で覚える
