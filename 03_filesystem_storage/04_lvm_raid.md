# LVM と RAID — ブロックデバイスの翻訳と冗長化

## 概要

これまでの3章は「ファイルシステムは1つのブロックデバイスの上に載っている」
ことを暗黙の前提にしてきました。この章はその足元——ブロックデバイスの層——を
主題にし、デバイスの合成・分割・引っ越しを可能にする LVM と、故障を織り込んで
冗長化する RAID を扱います。前提知識は `02_process_kernel/03`(仮想メモリの
「翻訳表を挟む」という型)、前章・前々章(ファイルシステムの台帳と
クラッシュ整合性)です。基準環境は Linux 7.0 / Ubuntu Server 26.04 LTS です。

## 導入 — 「1台のディスクの連続領域」という前提を疑う

ext4 の章で見たとおり、ファイルシステムは「ブロック 0 から N まで整然と並んだ、
1つの連続した記憶領域」を前提に台帳を組み立てます。ところが、その前提を
物理的なディスクにそのまま重ねると、3つの現実問題にぶつかります。

1. **サイズが固定である。** インストール時に「/ に 100 GiB、/var に 50 GiB」と
   区切ったが、1年後に /var だけが溢れた——区画の切り直しは、データを
   退避して作り直す大工事になります。逆に、ディスクを1台買い足しても、
   既存のファイルシステムをそこへ「続きとして」延ばす手段がありません
2. **ディスクは壊れる。** 可動部を持つ HDD はもちろん、SSD にも寿命と突然死が
   あります。1台のディスクにファイルシステムを直接置くことは、その1台を
   **単一障害点(single point of failure)**にすることと同じです。サーバーは
   「壊れたら再インストール」では済みません
3. **性能と容量が1台の上限で頭打ちになる。** どれほど速いディスクでも
   1台のスループットには物理的な上限があり、容量も1台分が天井です

3つに共通するのは、「**ファイルシステムが見ているブロックの並び**」と
「**物理的なディスク**」が1対1で直結していることが原因だ、という点です。
この構図、既視感がないでしょうか。プログラムが見るアドレスと物理メモリを
直結すると隔離も配置の自由も失われる——だから間に**翻訳表**(ページテーブル)を
挟んだ、というのが仮想メモリの章(`02_process_kernel/03`)の答えでした。

ブロックデバイスの世界も、まったく同じ型で解きます。ファイルシステムには
これまでどおり「連続したブロックの並び」という**眺め**を見せ、その眺めと
物理ディスクの間に翻訳の層を挟む。翻訳表の書き方次第で、複数台を1つに
束ねる(結合・ストライピング)、1つの書き込みを複数台へ写す(ミラーリング)、
後から対応先を差し替える(オンラインの引っ越しや拡張)が、**上に載った
ファイルシステムに一切気づかれずに**できるようになります。

この層の代表が、配置の自由を担う **LVM**(Logical Volume Manager)と、
冗長性・性能を担う **RAID** です。両者は排他ではなく、実務では
「RAID で冗長化した土台の上に LVM で自由に区画を切る」ように重ねて使います。

## 理論

本節の内容は、主に **Linux カーネルドキュメントの md
(Documentation/admin-guide/md.rst)と device mapper
(Documentation/admin-guide/device-mapper/ 以下)**、`man 8 lvm`、
`man 7 lvmraid`、`man 8 mdadm`、および RAID の原論文
(Patterson, Gibson, Katz, "A Case for Redundant Arrays of Inexpensive
Disks (RAID)", 1988)に基づきます。

### ブロックデバイス — ファイルシステムの足元にあるもの

まず足元の用語を正確にします。**ブロックデバイス(block device)**とは、
「固定長の区画(セクタ)が番号順に並んでいて、どの番号にも順不同で
読み書きできる装置」というカーネルの抽象です。HDD・SSD・NVMe はもちろん、
後述の仮想的なデバイスもすべてこの同じ顔でユーザーに見えます
(`/dev/sda`、`/dev/nvme0n1` など。`ls -l` の先頭が `b` の特殊ファイル)。
対になる抽象は、端末のようにバイトが流れてくるだけで「番地」を持たない
キャラクタデバイス(先頭 `c`)です。

ファイルシステムの mount(`03_filesystem_storage/01`)が要求するのは、
この「ブロックデバイス」という顔だけです。裏が1台の物理ディスクか、
100台の束か、mount は関知しません。**VFS がファイルシステム実装の違いを
隠したのと同じ分業を、ブロックデバイスの抽象は装置の違いに対して行って
いる**わけです。

### 積み重ねられるという性質 — パーティションから device mapper へ

この抽象の決定的な性質は、**ブロックデバイスの上にブロックデバイスを
作れる**(積み重ねられる)ことです。最初の例はすでに使っています。
**パーティション(partition)**——1台のディスクの先頭に区画の一覧表
(現代の標準は GPT: GUID Partition Table)を書き、「セクタ 2048 から
100 GiB 分」のような範囲1つ1つを独立したブロックデバイス
(`/dev/sda1`、`/dev/sda2`…)として見せる仕組みです。`/dev/sda1` への
書き込みは、カーネルが開始位置をずらして `/dev/sda` へ転送するだけ——
つまりパーティションとは**最も素朴な翻訳(足し算1回)**です。

この「範囲を区切って転送する」を一般化したのが、カーネルの
**device mapper(dm)**です。device mapper は特定の機能の名前ではなく、
「新しい仮想ブロックデバイスを1つ作り、その区間ごとに『この範囲への
アクセスは、あのデバイスのあの位置へこの方式で転送せよ』という
**マッピングテーブル**を与える」ための汎用の枠組みです。転送方式
(**ターゲット**と呼びます)は差し替え可能な部品になっています。

| ターゲット | 転送のしかた | 使われ先の例 |
|---|---|---|
| linear | 別デバイスの指定位置へそのまま転送 | LVM の通常の LV |
| striped | 複数デバイスへ一定幅で交互に転送 | LVM のストライプ LV |
| mirror / raid | 複数デバイスへ複製・パリティ付きで転送 | LVM RAID |
| snapshot / thin | 書き込み時に退避・割り当てを行う | スナップショット、シンプロビジョニング |
| crypt | 暗号化して転送 | ディスク暗号化(LUKS)の土台 |

区間ごとに転送先を差し替えられる仮想デバイス——これはまさに、ページ単位で
物理フレームへの対応を差し替えられるページテーブルの、ブロック版です。
本書の言い方でまとめれば、**ブロックの世界にも翻訳表を挟む**。この章の
残りは、この翻訳表をどう書くと何ができるか、の各論です。

### LVM — 置き場所の自由を取り戻す3層モデル

LVM は、device mapper の翻訳表を人間が管理しやすい形で組み立てるための
体系で、3層のモデルを使います。

```
  LV(論理ボリューム)      lv_root(50GiB)   lv_var(200GiB)   lv_snap …
     ↑ LE→PE の対応表         │ ファイルシステムはここに載る
  ─────────────────────────────────────────────
  VG(ボリュームグループ)   vg0 ── PE(4MiB)の大きなプール
  ─────────────────────────────────────────────
     ↓ PE に等分
  PV(物理ボリューム)       /dev/sda2      /dev/sdb1      /dev/sdc1
```

- **PV(Physical Volume / 物理ボリューム)** — LVM の材料として登録した
  ブロックデバイス。登録(pvcreate)すると先頭に管理用のラベルと
  メタデータ領域が書かれ、残りが **PE(Physical Extent / 物理エクステント。
  既定 4 MiB)**という固定サイズの粒に等分されます
- **VG(Volume Group / ボリュームグループ)** — PV を束ねた**プール**。
  この時点で「どのディスクか」という個体差は消え、「PE が全部で何個あるか」
  だけの世界になります
- **LV(Logical Volume / 論理ボリューム)** — プールから PE を必要な数だけ
  集めて作る仮想ブロックデバイス。LV 側の粒(LE: Logical Extent)と PE の
  対応表が、まさに翻訳表です。ファイルシステムはこの LV の上に作ります

用語の注意を1つ。この「エクステント」は **ext4 のエクステント
(`03_filesystem_storage/03`)とは別物**です。ext4 のそれは可変長の連続
範囲を表すレコード、LVM のそれは固定長の割り当て単位で、本書では LVM の
文脈では「PE / LE」と書いて区別します。

3層に分ける利点は、**変更したい操作がすべて「翻訳表の書き換え」に
還元される**ことです。

- **LV の拡張(lvextend)** — プールから PE を追加で払い出し、対応表に
  行を足すだけ。LV は使用中(mount 中)のままで構いません。ファイル
  システム側のサイズは別管理なので、続けて resize2fs 等で「器が広がった」
  ことを台帳に反映させます(ext4 はオンライン拡張可)
- **プールの拡張(vgextend)** — 新しいディスクを PV にして VG へ足すと、
  既存の LV たちの拡張余地が増えます。「ディスクを買い足して続きに使う」が
  実現します
- **引っ越し(pvmove)** — ある PV 上の PE を別の PV へコピーし、対応表を
  差し替えます。**使用中のまま**ディスクを交換前に空にできます
- **スナップショット** — ある瞬間の LV の凍結像をもう1つの LV として
  見せます(仕組みは内部動作の節で)

大事な分業を1つ。カーネルの中に「LVM」というコードはほぼありません。
LVM の実体は**ユーザーランドのツール群と、PV 上のメタデータの形式**であり、
起動時の activation(`vgchange -ay`)がメタデータを読んで dm の
マッピングテーブルを組み立て、カーネルに渡します。以後の毎回の I/O を
翻訳するのは device mapper です。「方針はユーザーランド、機構はカーネル」
という、スケジューラの章(`02_process_kernel/04`)でも見た役割分担です。

### RAID — 故障を織り込む

LVM が解いたのは配置の自由(問題1)でした。問題2(故障)と問題3(性能)に
答えるのが **RAID(Redundant Array of Independent Disks)**——複数台の
ディスクを束ねて1つのブロックデバイスに見せ、**冗長性**(何台か壊れても
データが失われない)と**性能**(複数台の並列)を得る技術です。束ね方は
「レベル」として類型化されています(1988年の原論文に由来する呼び名です)。

- **RAID 0(ストライピング)** — データを**チャンク**という一定幅で
  輪切りにし、複数台へ交互に置く。読み書きが全台に並列化され、容量も
  全台分使えるが、冗長性はゼロ。それどころか**1台でも壊れると全体が
  失われる**ため、故障率は台数分だけ悪化する
- **RAID 1(ミラーリング)** — 同じ内容を全台(通常2台)へ複製する。
  1台壊れても残りで続行できる。容量効率は 1/2。読み出しは2台に
  分散できるため読みは速くなりうる
- **RAID 5(パリティ)** — n 台のうち各ストライプの1台分を**パリティ**
  (誤り訂正用の冗長データ)に充て、n−1 台分の容量で1台の故障に耐える。
  パリティの置き場所はストライプごとに輪番で分散する
- **RAID 6(二重パリティ)** — パリティを2種類(P と Q)持ち、
  n−2 台分の容量で**2台**の故障に耐える
- **RAID 10** — ミラーしたペアをストライプで束ねる(1+0)。容量効率は
  1/2 だが、パリティ計算がなく性能・再構築とも素直

パリティの原理は、XOR(排他的論理和)の「同じものを2回足すと消える」
という性質だけで説明できます。3台のデータブロック A, B, C に対し
パリティ P = A ⊕ B ⊕ C を4台目に書いておくと、たとえば B のディスクが
壊れても **B = A ⊕ C ⊕ P** で復元できます——**失われた1台は、残り全部の
XOR に等しい**。これが「n−1 台分の容量で1台の故障に耐える」の正体です。
RAID 6 の2つ目のパリティ Q は XOR ではなくガロア体上の係数付きの和
(リード・ソロモン符号)で、P と連立させることで2台分を解けます。

RAID には、専用コントローラが担う**ハードウェア RAID** と、カーネルが
担う**ソフトウェア RAID** があります。本書が扱うのは後者、Linux の
**md(Multiple Devices)ドライバ**です(管理コマンドは mdadm。組んだ
配列は `/dev/md0` などのブロックデバイスとして現れ、状態は
`/proc/mdstat` で見えます)。なお LVM 自身にも RAID LV を作る機能が
ありますが(`man 7 lvmraid`)、その実装は dm の raid ターゲットが
**md の各レベルの実装(パーソナリティ)を内部で再利用**したものです。
mdadm 経由でも LVM 経由でも、ストライプとパリティを計算している
カーネル内のコードは同じ——顔は2つ、実装は1つです。

## 内部動作の詳細

### 翻訳表の実物 — dm のマッピングテーブル

dm のマッピングテーブルは `dmsetup table` で生で見ることができます。
たとえば2つの PV にまたがって拡張された LV は、こう見えます
(数値の単位はすべて 512 バイトのセクタです)。

```console
$ sudo dmsetup table vg0-lv_data
0 2097152 linear 8:2 2048        ← LV の先頭 1GiB は sda2(8:2)のセクタ2048から
2097152 1048576 linear 8:17 2048 ← 続く 512MiB は sdb1(8:17)のセクタ2048から
```

「区間の開始・長さ・ターゲット・転送先」の行の並び——これが翻訳表の
すべてです。LV への I/O が来るたび、dm は該当する行を引いて宛先を
書き換え、下のデバイスへ渡します。ページテーブルの表引きと違うのは、
粒度が PE 単位と粗く、区間の数が高々数個〜数百個なので、表引きの
コストが実質無視できることです。

### LVM のメタデータ — 翻訳表の永続化と復元

dm のテーブルはメモリ上にしかなく、再起動で消えます。永続するのは
**PV の先頭に書かれた LVM メタデータ**です。中身は VG 全体の構成
(どの PV があり、どの LV がどの PE を使っているか)を記した
**テキスト**で、更新のたびに新しい版を循環バッファへ書き足し、既定では
**VG 内の全 PV に同じものを複製**します。どれか1台が生きていれば構成を
復元できるうえ、テキストなので人間にも読めます(`/etc/lvm/backup/` に
控えも自動保存され、`vgcfgrestore` で書き戻せます)。「翻訳表そのものを
失うとデータが無事でも読み出せない」——ページテーブルを失ったメモリと
同じ——ことへの、多重の備えです。

### スナップショット — CoW の再演

スナップショットは「ある瞬間の LV の内容を凍結した、もう1つのブロック
デバイス」です。全量コピーしていたら瞬間になりません。答えはもう
おなじみ、**コピーオンライト**です(`02_process_kernel/01` の fork と
同じ型)。

伝統的な方式(dm-snapshot)は、退避領域を1つ用意し、スナップショット
作成後に**元の LV へ書き込みが来た瞬間、上書きされる前の古いブロックを
退避領域へ写してから**書き込みを通します。スナップショット側の読み出しは
「退避領域にあればそれ、なければ元 LV の現物」を返す——凍結像が保たれます。
代償は2つ。元 LV への書き込みが最悪3回の I/O(旧データ読み+退避書き+
本体書き)になること、そして**退避領域が溢れるとスナップショットは
無効化される**ことです。

現代の LVM はもう1つ、**シンプロビジョニング(thin provisioning)**方式を
持ちます。プール上に「書かれたブロックにだけ実領域を割り当てる」LV を作る
仕組みで(スパースファイルのブロックデバイス版と考えてください)、
スナップショットは「同じブロックを共有する2つ目の対応表」を作るだけで
済みます。書き込み時は共有を解いて新しい領域に書く——退避コピーが不要で、
スナップショットを何世代も持てます。ただし「見かけの合計サイズ >
プールの実容量」を許す設計なので、**プールを使い切ったときの破綻**
(全 thin LV への書き込みがエラーになる)がオーバーコミット
(`02_process_kernel/03`)と同型の問題として現れます。

### md のストライプとパリティ更新 — RMW ペナルティ

md の RAID 5 は、データを**チャンク**(既定 512 KiB)単位で輪切りにし、
横一列(**ストライプ**)ごとにパリティを1つ置きます。

```
        sdb        sdc        sdd
      ┌────────┬────────┬────────┐
 帯0  │ D0     │ D1     │ P(0,1) │   P = D0 ⊕ D1
      ├────────┼────────┼────────┤
 帯1  │ D2     │ P(2,3) │ D3     │   ← パリティは輪番で分散
      ├────────┼────────┼────────┤
 帯2  │ P(4,5) │ D4     │ D5     │
      └────────┴────────┴────────┘
```

問題は書き込みです。D0 だけを書き換えたくても、P も整合するように
更新しなければなりません。md は2つの戦略を使い分けます。

- **read-modify-write(RMW)** — 旧 D0 と旧 P を読み、
  新 P = 旧 P ⊕ 旧 D0 ⊕ 新 D0 を計算して、新 D0 と新 P を書く。
  小さな書き込み1つが**読み2回+書き2回**に化けます
- **reconstruct write** — ストライプの大部分を書き換えるなら、残りの
  データを読んでパリティを作り直したほうが安い

前者が、RAID 5 が「小さなランダム書き込みに弱い」と言われる理由
(**RMW ペナルティ**)です。逆に、ストライプ幅にきれいに揃った大きな
書き込みは読み戻しなしで全台並列に書けます。md がストライプ単位の
組み立てバッファ(stripe cache)で書き込みをためてから処理するのも、
「遅らせると、まとめられて、賢く決められる」(前章の反復原理)の
一例です。

### 縮退と再構築 — 冗長性が試されるとき

1台が脱落した配列は**縮退(degraded)**状態になります。RAID 5 なら
読み出しのたびに欠けた分を残り全台の XOR で作りながら、動き続けます。
ここで大事なのは、**縮退した配列は「壊れていない」のではなく「次の
1台で全損する」状態**だということです。交換ディスクを入れると
**再構築(recovery / resync)**が走り、全ストライプを読んで欠けた台の
中身を計算・書き込みします(`/proc/mdstat` に進捗が出ます。速度は
`/proc/sys/dev/raid/speed_limit_min` / `speed_limit_max` で通常 I/O との
綱引きを調整できます)。

再構築には固有のリスクがあります。全台を端から端まで読む作業なので、
**残った台の読み出し不能セクタ(URE)を踏むと、その箇所は復元不能**に
なります。ディスクの大容量化で「全読み」のビット数が桁で増えた結果、
RAID 5(冗長度1)の再構築中は無防備だという認識が広まり、大容量では
RAID 6 や RAID 10 が選ばれるようになりました。予防としては、md の
**スクラブ**(`sysfs` の `sync_action` に `check` を書くと全ストライプの
整合を検査する。Ubuntu では定期実行が設定されています)で、不良セクタを
冗長性が残っているうちに見つけて書き直させるのが定石です。

### RAID の write hole — ブロック層にもあるクラッシュ整合性

前章の主役だった問いを、この層でももう一度立てます。ストライプの更新は
「D0 の書き込み」と「P の書き込み」という**別々のディスクへの2つの
書き込み**であり、原子的ではありません。D0 だけ書けた瞬間に電源断が
起きると、そのストライプはパリティと矛盾した状態になります。データ
そのものは新旧どちらかで筋が通っていても、**その後に別の1台が壊れて
XOR で復元した瞬間、矛盾したパリティからゴミが合成される**——これが
RAID 5 の **write hole** と呼ばれる問題です。

上に載ったファイルシステムがジャーナリングしていても防げません。
ext4 の日誌が守るのは「ファイルシステムの台帳どうし」の整合であって、
その下の層がブロックを正しく保持していることは前提だからです。
**層ごとの整合性は、層ごとに自分で守るしかない**。md の答えは、
これも前章と同じ型です。

- **クラッシュ後の resync** — 正常に閉じられなかった配列は、全パリティを
  検査・修復してから運用に戻る(素朴だが全読みで時間がかかる)
- **write-intent bitmap** — 「この区域にいま書き込み中」という印を先に
  記録しておき、クラッシュ後は印のある区域だけ resync する。全読みが
  差分になる(大きな配列では既定で有効)
- **ジャーナル / PPL** — 書き込み内容そのものを先に専用デバイス
  (`--write-journal`。SSD を充てる)へ記録してからストライプに反映する、
  文字どおりの WAL。PPL(Partial Parity Log)は追加デバイスなしで
  各メンバー上に部分パリティを先行記録する軽量版

「先に日誌へ書き、印で完結を示す」——ジャーナリングファイルシステムと
同じ解法が、1つ下の層で独立にもう1回使われているわけです。

## 実行例 — 翻訳表と配列を観察する

LVM の3層は、ループデバイス(ファイルをブロックデバイスに見せる機能)で
安全に試せます。

```console
$ sudo pvcreate /dev/loop0 /dev/loop1
$ sudo vgcreate vg_demo /dev/loop0 /dev/loop1
$ sudo lvcreate -L 1G -n lv_data vg_demo
$ sudo lvs -o lv_name,lv_size,seg_pe_ranges vg_demo
  LV      LSize PE Ranges
  lv_data 1.00g /dev/loop0:0-255     ← PE 4MiB × 256個 = 1GiB
$ sudo dmsetup table vg_demo-lv_data
0 2097152 linear 7:0 2048            ← 翻訳表の実物(単位は512Bセクタ)
```

拡張は使用中のまま、翻訳表への行の追加として起きます。

```console
$ sudo mkfs.ext4 -q /dev/vg_demo/lv_data && sudo mount /dev/vg_demo/lv_data /mnt
$ sudo lvextend -L +512M -r /dev/vg_demo/lv_data   ← -r はFS拡張(resize2fs)も併せて実行
$ sudo dmsetup table vg_demo-lv_data
0 2097152 linear 7:0 2048
2097152 1048576 linear 7:0 2099200   ← 追加された区間の行
```

md の配列の状態は `/proc/mdstat` に集約されています。

```
$ cat /proc/mdstat
Personalities : [raid1] [raid6] [raid5] [raid4]
md0 : active raid5 sdd1[3] sdc1[1] sdb1[0]
      41908224 blocks super 1.2 level 5, 512k chunk, algorithm 2 [3/3] [UUU]
      bitmap: 0/1 pages [0KB], 65536KB chunk
```

`[3/3] [UUU]` が「定員3台、現在3台、全員稼働(Up)」の意味で、縮退すると
`[3/2] [U_U]` のように欠けます。再構築中は進捗バーの行が加わります。
また `lsblk` は、この章で見た「積み重ね」を木としてそのまま表示します
(ディスク → パーティション → md → LVM → マウント先)。

## トラブルシューティング

- **lvextend したのに df の容量が増えない**: LV(器)とファイルシステム
  (台帳)は別の層で、それぞれにサイズを持ちます。翻訳表を広げただけでは
  台帳は昔のまま——resize2fs(または lvextend -r)で台帳側も広げます
- **起動時にルートの LV / md が見つからない**: LVM の activation も md の
  組み立ても**ユーザーランドの仕事**なので、ルートファイルシステム自身が
  LVM/RAID 上にある場合は initramfs の中で行われます(`01_intro/04` で
  「initramfs は RAID の組み立てなどルートをマウントするための準備を担う」と
  述べたことの正体です)。initramfs 内の設定(mdadm.conf 等)が実態と
  ずれていると、本体は正常でも起動だけが失敗します。なお UEFI が直接読む
  ESP(EFI システムパーティション)は、ファームウェアが LVM も md も
  理解しないため、素のパーティションのままにする必要があります
- **スナップショットが突然消えた(Input/output error)**: 伝統的方式の
  スナップショットは退避領域が溢れた時点で無効化されます。`lvs` の
  Data% を監視し、溢れる前に拡張するか、thin 方式を使います。ただし
  thin はプール自体の使い切りがより深刻(全 LV に波及)なので、
  監視対象がプールに変わるだけとも言えます
- **縮退に気づかず数か月**: RAID の皮肉は、1台壊れても**何事もなく
  動き続ける**ことです。`/proc/mdstat` の `[U_]` は誰かが見なければ
  ただのファイルです。mdadm --monitor やメール通知、監視システム
  (分野09)への組み込みが冗長化とセットで必須になります
- **再構築が遅い/日中の I/O を圧迫する**: speed_limit_min / max の
  綱引きの調整対象です。ただし「早く終わらせたい」と「業務を止めたく
  ない」は正面から衝突するため、どちらを優先するかの判断が先です
- **「RAID があるからバックアップは不要」**: RAID が守るのは**装置の
  故障**に対する可用性だけです。rm も、ランサムウェアの暗号化も、
  アプリのバグの上書きも、ミラーは**忠実に全台へ複製**します。誤操作・
  論理破壊への備えはバックアップ(分野07)の領分で、代替になりません

## 演習・確認問題

1. 「ファイルシステムと物理ディスクの直結」が生む3つの問題を挙げ、
   「間に翻訳表を挟む」ことがそれぞれをどう解くか、仮想メモリの
   ページテーブルとの対応を意識して説明してください
2. LVM の PV / VG / LV の3層それぞれの役割を述べ、lvextend・vgextend・
   pvmove の3操作が「翻訳表の書き換え」としてそれぞれ何をしているかを
   説明してください
3. RAID 5(4台構成)で、データブロック1つだけを書き換えるときに
   発生する I/O を read-modify-write 方式で数え上げてください。
   新パリティの計算式に XOR のどの性質が使われていますか
4. 縮退した RAID 5 の再構築中に残存ディスクの読み出し不能セクタ(URE)を
   踏むと何が起きますか。大容量化がこのリスクをどう変え、RAID 6 や
   定期スクラブがどう緩和するかを説明してください
5. RAID 5 の write hole がなぜ起きるのかを「複数台への書き込みは原子的で
   ない」という観点から説明し、write-intent bitmap とジャーナルのそれぞれが
   前章のジャーナリングのどの考え方に対応するかを述べてください。また、
   ext4 のジャーナリングでは write hole を防げない理由も説明してください
6. 伝統的スナップショット(退避方式)と thin 方式のスナップショットの
   仕組みの違いを、書き込み時に起きる I/O と「溢れたとき」の影響範囲の
   両面から比較してください

## まとめ

- ファイルシステムと物理ディスクの直結が、サイズ固定・単一障害点・
  1台の上限という3つの問題を生む。解法は仮想メモリと同じ
  「**ブロックの世界にも翻訳表を挟む**」——その汎用機構が device mapper で、
  翻訳表(マッピングテーブル)は dmsetup table で生に見える
- LVM は PV(材料)→ VG(プール)→ LV(払い出し)の3層で翻訳表を管理する
  体系。拡張・増設・引っ越し・スナップショットがすべて使用中のままの
  翻訳表書き換えに還元される。実体はユーザーランドのツールとメタデータで、
  毎回の翻訳はカーネル(dm)が行う
- RAID は複数台を束ねて冗長性と性能を得る。パリティの原理は XOR の
  「失われた1台は残り全部の XOR」。小さな書き込みは RMW ペナルティ
  (読み2+書き2)を払う
- 縮退した配列は「次の1台で全損」の状態。再構築は全読みを伴い URE リスクが
  あるため、大容量では冗長度2(RAID 6 等)とスクラブが定石。冗長化は
  監視(気づく仕組み)とバックアップ(論理破壊への備え)とセットで初めて
  意味を持つ
- ストライプ更新は原子的でなく、write hole というブロック層固有の
  クラッシュ整合性問題がある。解法(bitmap、ジャーナル)は前章の WAL と
  同じ型——**層ごとの整合性は、層ごとに自分で守る**
