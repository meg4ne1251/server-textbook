# cgroups とネームスペース — コンテナの土台

## 概要

分野05(仮想化・コンテナ基盤)の最初の章です。「コンテナ」の正体が、
**見え方を分けるネームスペース**と**使える量を決める cgroups** という
2つの汎用カーネル機構の組み合わせであることを、両機構の設計と内部動作から
理解します。前提知識は分野02(プロセス、スケジューラ、仮想メモリ)と
分野04第3章(ネットワークネームスペース)。基準環境は Linux 7.0 /
Ubuntu Server 26.04 LTS、cgroups は **v2(unified hierarchy)のみ**を
基準とします。

## 導入 — 「コンテナ」というモノは、カーネルの中にない

前の分野の最後で、1台のマシンの中に独立したネットワークスタックを
複数作れることを見ました。あの章で予告したとおり、本章はその話を
一般化します——ネットワーク以外の「眺め」も分けられるのか。そして、
眺めを分けるだけでは足りない問題とは何か。

出発点として、サーバー運用の古典的な要求を思い出します。**1台の物理
マシンに複数のアプリケーションを同居させたい**。しかし素朴に同居させる
と、互いに干渉します。干渉の中身を分類すると、実は2種類しかありません。

1. **見えてしまう干渉。** 相手のプロセスが `ps` で見える。同じポート
   80 番を取り合う。同じ `/etc` を書き換え合う。ホスト名が1つしかない
2. **奪ってしまう干渉。** 片方が CPU を食い尽くすと、もう片方が
   遅くなる。片方のメモリリークが、もう片方を OOM killer の巻き添えに
   する(分野02で見たとおり、OOM killer は点数の高い者から殺します——
   それが「無実の隣人」であっても)

この2種類に、Linux は別々の機構で答えます。

- 見えてしまう干渉には**ネームスペース(namespace)**——カーネルが
  管理する資源の「眺め」をプロセスのグループごとに複製し、互いに
  見えなくする
- 奪ってしまう干渉には**cgroups(control groups)**——プロセスを
  グループに束ね、グループ単位で CPU 時間・メモリ量・I/O 帯域の
  「取り分」を制限・分配する

そして重要なのは、**カーネルには「コンテナ」という概念そのものは
存在しない**ということです。カーネルソースを `container` で検索しても、
コンテナを表す構造体は出てきません。あるのはネームスペースと cgroups
という2つの汎用機構だけで、「コンテナ」とは、ユーザーランドのツール
(Docker、containerd 等)がこの2つ(+ルートファイルシステムの
差し替えなど)を**決まった組み合わせで束ねたものに付けた名前**です。
ちょうど「プロセス」がカーネルにとっては task_struct と資源への参照の
束であったように、コンテナはネームスペースと cgroup への所属の束です。

仮想マシン(次章の KVM/QEMU)との違いも、この見方から明確になります。
仮想マシンは**ハードウェアの層で切る**——ゲストごとに仮想の CPU・
メモリ・ディスクを見せ、その上でゲストが**自前のカーネル**を動かします。
コンテナは**カーネルの中で切る**——全員が**同じカーネル**の上で動く
ただのプロセスであり、カーネルが見せる眺めと取り分だけが違います。
分野03の NAS/SAN で使った整理(どの層で切るかが、共有できるものと
責任の在り処を決める)の再演です。切る層が低い(VM)ほど隔離は強く
オーバーヘッドは大きく、層が高い(コンテナ)ほど軽く、その代わり
カーネルは1つを共有します。

## 理論

本節の内容は、主に **`man 7 namespaces`**(および `man 7` の各
ネームスペース個別ページ)、**Linux カーネルドキュメント
(Documentation/admin-guide/cgroup-v2.rst)**、`man 7 cgroups`、
カーネルソース(include/linux/nsproxy.h、kernel/cgroup/)に基づきます。

### ネームスペース総論 — 「眺めの複製」の一般形

前分野で見たネットワークネームスペースの型——**カーネル内のグローバルな
状態を1式複製し、プロセスごとに「どの複製を見るか」を差し替える**——は、
ネットワーク以外にも適用されています。`man 7 namespaces` が列挙する
ネームスペースは現在 **8種類**です。

| 種類 | 分けられる眺め | 導入版(参考) |
|---|---|---|
| mount | マウントの一覧(ファイルシステムツリーの見え方) | 2.4.19(2002年、最古) |
| UTS | ホスト名・ドメイン名 | 2.6.19 |
| IPC | System V IPC・POSIX メッセージキューの空間 | 2.6.19 |
| PID | プロセス番号の空間 | 2.6.24 |
| network | ネットワークスタック一式(前分野で詳説) | 2.6.24 から段階的 |
| user | UID/GID の空間と capability | 3.8(2013年)で完成 |
| cgroup | cgroup ツリーの見え方(根の位置) | 4.6 |
| time | CLOCK_MONOTONIC 等の起点 | 5.6 |

種類は増えてきましたが、**共通の設計**はすべて同じです。前分野で
ネットワークネームスペースについて学んだ次の事実は、そのまま全種類に
一般化されます。

- **所属は task_struct → nsproxy でたどられる。** タスクは種類ごとに
  「どのネームスペースに属するか」の参照を持ち、その束が nsproxy
  構造体です(user ネームスペースだけは別経路で、身分情報 cred 側に
  ぶら下がります——理由は後述)
- **fork した子は親の眺めを引き継ぐ。** ネームスペースは「設定」では
  なく「所属」なので、家系に沿って自然に伝播します
- **入り方は3つ。** 新しく作って子をそこに入れる(clone のフラグ)、
  自分自身を新しいネームスペースへ移す(unshare)、既存のネームスペース
  へ合流する(setns)
- **`/proc/<PID>/ns/` に全種類の名札がある。** 各ファイルは
  `pid:[4026531836]` のような inode 番号を持ち、番号が同じなら同じ
  ネームスペースです。この fd を開いて持っておく・どこかへ bind mount
  する、がネームスペースを空のまま延命する方法である点も前分野と
  同じです
- **寿命は参照カウント。** 所属プロセスも、開かれた fd も、bind mount
  もなくなれば、カーネルが片付けます

つまりネームスペースの学習で本当に必要なのは、共通の型を1回
(前分野で済んでいます)と、**種類ごとの固有の癖**です。以下、癖の
大きい順に見ます。

### PID ネームスペース — 番号の空間は入れ子になる

PID ネームスペースは、他のネームスペースにない特徴を2つ持ちます。

第一に、**入れ子の階層**をなします(最大32段)。新しい PID
ネームスペースは親ネームスペースの「中」に作られ、**親からは子の中の
プロセスが見えます**——ただし親の番号体系で。子の中からは親側は
一切見えません。つまり1つのプロセスは、自分が属するネームスペースから
根まで、**階層の段数ぶんの PID を同時に持ちます**。コンテナの中では
PID 1 に見えるプロセスが、ホストの `ps` では PID 34567 に見える——
これは「変換」されているのではなく、**最初から両方の番号を持っている**
のです(内部構造は後述)。

第二に、**unshare しても自分は移らない**という癖があります。
unshare(CLONE_NEWPID) を呼んだプロセス自身は元のネームスペースに
留まり、**次に fork した子**が新しいネームスペースの最初のプロセス=
**そのネームスペースの PID 1** になります。getpid() が返す値が突然
変わると、自分の PID を覚えているライブラリ・ランタイムが壊れるため、
という実装上の理由が `man 2 unshare` に記されています。

そして、その「ネームスペースの PID 1」は、分野02で学んだ本物の PID 1
と同じ責務と特権を、そのネームスペースの中で持ちます。

- **孤児の引き取り先になる。** ネームスペース内で親を失ったプロセスは、
  ホストの PID 1 ではなく、そのネームスペースの PID 1 に養子縁組
  されます(分野02の reparenting のネームスペース版)
- **シグナルから保護される。** 同じネームスペース内からのシグナルは、
  PID 1 がハンドラを登録しているものしか届きません。デフォルト処分
  (分野02)による暗黙の終了が起きないのです。SIGKILL ですら同一
  ネームスペース内からは効きません(祖先ネームスペースからは効きます)
- **死ぬと道連れ。** ネームスペースの PID 1 が終了すると、カーネルは
  その中の全プロセスに SIGKILL を送り、以後そのネームスペースでの
  fork は失敗します。「init が死んだら世界の終わり」のミニチュア版です

なお、PID の眺めを本当に分けるには **mount ネームスペースとの併用が
必須**です。`ps` はカーネルに問い合わせるのではなく `/proc` を読む
コマンドなので(分野02)、`/proc` をマウントし直さない限り、新しい
PID ネームスペースの中でもホストの `/proc` が見えたままになります。
ネームスペースが「単品」ではなく「組み合わせ」で使われる典型例です。

### mount ネームスペース — ツリーの眺めの複製

mount ネームスペース(最古のネームスペース)は、分野03で学んだ
「単一ツリーへの接ぎ木」の一覧、すなわちマウントテーブルの眺めを
複製します。作成時点のマウント一覧のコピーから始まり、以後の
mount / umount は原則としてそのネームスペースの中だけの出来事に
なります。

「原則として」と書いたのは、完全に切り離すと不便な場合がある
からです——ホスト側で USB ディスクをマウントしたら、コンテナにも
見せたい、など。このためマウント1件ごとに**伝播(propagation)の
方針**(shared / private / slave)を設定でき、systemd はホストの
ルートを shared にしています。伝播の規則は本書では深追いせず、
「ネームスペース間でマウントイベントを転送する仕組みがある」という
位置づけの理解にとどめます(Documentation/filesystems/sharedsubtree.rst
が一次情報源です)。

コンテナの文脈では、mount ネームスペースは**ルートファイルシステムの
差し替え**の土台です。新しいネームスペースの中で pivot_root により
ルートを別のディレクトリツリー(コンテナイメージの中身)へ切り替える
ことで、「コンテナの中では `/` がまるごと別物」が実現されます。
この手順の詳細はコンテナランタイムの章(本分野第3章)で扱います。

### UTS・IPC・time — 小さな眺めたち

残りの「小物」は要点だけで足ります。

- **UTS ネームスペース**はホスト名・NIS ドメイン名だけを分けます
  (名前の UTS は uname の構造体名 utsname の名残)。コンテナごとに
  hostname が違って見えるのはこれです
- **IPC ネームスペース**は、分野02で学んだ System V IPC(共有メモリ・
  セマフォ・メッセージキュー)と POSIX メッセージキューの名前空間を
  分けます。キー(key_t)で待ち合わせる旧世代 IPC は、名前が
  グローバルであるがゆえに隔離が必要になる、という関係です
- **time ネームスペース**は CLOCK_MONOTONIC / CLOCK_BOOTTIME の
  起点オフセットだけを分けます。壁時計(CLOCK_REALTIME)は分け
  ません。主な動機はプロセスのチェックポイント/リストア(マシンを
  またいだ移送)で、「起動からの経過時間」が移送先で巻き戻らない
  ようにするためです

### user ネームスペース — 隔離の世界の「偽の root」

user ネームスペースは、後発(3.8 で完成)ながら最も重要な癖を
持ちます。**UID/GID の番号空間を分け、外の番号との対応表を持つ**の
です。

新しい user ネームスペースを作ると、その作成者は**中では UID 0
(root)**になれます。ただしそれは対応表(`/proc/<PID>/uid_map`)に
「中の 0 は外の 1000 に対応する」と書かれた上での話で、この
プロセスがファイルを作れば、ディスク上の inode に記録される所有者は
**外の番号(1000)**です。つまり中の root は、外から見ればただの
一般ユーザーのままです。

それでも「中で root になれる」ことには決定的な意味があります。
分野01で「root の全能性は capability に分割されている」と予告した
話がここで効いてきます——**capability の判定は、user ネームスペース
ごとに行われる**のです。新しい user ネームスペースの作成者は、
**そのネームスペースが支配する範囲に対して**フルの capability を
持ちます。そして他のネームスペースは、すべて**どれか1つの user
ネームスペースに所有される**設計になっています(nsproxy ではなく
cred 側にぶら下がる、と先に述べた理由がこれです。user ネームスペース
は「眺め」であると同時に「権限の文脈」なのです)。

この設計の帰結が**非特権コンテナ**です。一般ユーザーが user
ネームスペースを作り、その中でなら(自分の支配範囲に限り)mount や
PID などのネームスペースも root 権限なしで作れる——コンテナを
root なしで動かす rootless コンテナの土台です。同時に、これは
「一般ユーザーがカーネルの広い攻撃面に触れる入り口」でもあり、
セキュリティ上の議論が続いてきました。Ubuntu は近年のリリースで
非特権 user ネームスペースの作成を AppArmor で制限しており
(要検証: Ubuntu 26.04 での既定)、実行例の節で挙動に触れます。
capability の体系そのものは分野07で扱います。

### cgroup ネームスペースと、cgroups 本体へ

8種類の最後、cgroup ネームスペースは「cgroup ツリーの眺め(根の
位置)」を分けるものですが、これを理解するには cgroups 本体が先です。
ここからが本章の後半の主役です。

### cgroups — プロセスを束ね、取り分を決める木

cgroups(control groups)の発想は2段階に分かれます。

**第1段階: プロセスをグループに束ね、木に並べる。**
システム上の全プロセスを、ディレクトリのような階層(木)に
振り分けます。「Web サーバー一式」「バッチ処理一式」「ユーザー A の
セッション一式」のように、**制御したい単位はプロセス1個ではなく
プロセスの集団**である、という運用の現実がこの設計の根拠です。
プロセスは常に木のどこか1箇所に属し、fork した子は親と同じ場所に
生まれます(ネームスペースと同じ「所属の継承」)。

**第2段階: 木の節ごとに、資源の取り分をコントローラが決める。**
CPU 時間・メモリ量・I/O 帯域・プロセス数など、資源の種類ごとに
**コントローラ(controller)**があり、木の各グループに「重み」や
「上限」を設定します。制限の実行はカーネルの既存機構——CPU なら
スケジューラ、メモリなら回収と OOM killer——がグループ単位の会計に
基づいて行います。cgroups は新しい資源管理を発明したのではなく、
**分野02で学んだ既存の資源管理に「グループ」という軸を追加した**もの
です。

このインタフェースは、疑似ファイルシステム **cgroupfs**
(`/sys/fs/cgroup` にマウント)として提供されます。「すべては
ファイル」(分野01)の再演です——グループの作成は mkdir、プロセスの
所属変更は `cgroup.procs` ファイルへの PID の write、制限の設定は
`cpu.max` 等への write、現在値の観察は read。専用のシステムコールは
ほぼなく、シェルだけで一通り操作できます。

### v1 の失敗と v2 の統一階層

cgroups には歴史の断層があります。Linux 2.6.24(2008年。奇しくも
PID/network ネームスペースと同じ版)で入った初代——今でいう
**cgroup v1** ——は、**コントローラごとに独立した木**を持てる設計
でした。CPU の木ではこう分類し、メモリの木では別の分類で、という
自由度です。

この自由度は失敗と総括されました。コントローラごとに木の形が違うと、
「このプロセス集団のメモリが逼迫したから I/O を絞る」といった
**コントローラ間の連携**が原理的に取れません(同じ「集団」が木ごとに
違うので)。実運用では結局ほぼ全員が全コントローラに同じ木を作って
いました。そこで Linux 4.5(2016年)で正式化された **cgroup v2** は、
**木は全体で1本(unified hierarchy)**に改めました。すべての
コントローラが同じ木を共有し、グループ「Web サーバー一式」は CPU に
とってもメモリにとっても同じ集団です。Ubuntu は 21.10 以降 v2 のみを
マウントする構成が既定で、本書も v2 のみを扱います(v1 の記述は
古い資料を読むときの注意として、トラブルシューティングで触れます)。

v2 の設計規則で重要なものを2つ挙げます(いずれも
cgroup-v2.rst に規定があります)。

- **プロセスは原則、木の葉にだけ置く**(no internal process 制約)。
  子グループを持つグループに直接プロセスを入れることは、root
  cgroup を除き許されません。「グループ A の取り分」を子たちに
  分配するとき、A 直属のプロセスと子グループが同じ土俵で競合すると
  意味論が濁るため、「分配する節」と「プロセスを収める葉」を分離
  させる規則です
- **コントローラは部下に対して有効化する。** 各グループの
  `cgroup.controllers` が使えるコントローラの一覧、
  `cgroup.subtree_control` に書き込むと**子グループたちに**その
  コントローラの設定ファイル(`cpu.*` 等)が現れます。木の上から
  順に、使う分だけ配る方式です

設定ファイルの記法も v2 で統一されました。相対配分は
**weight**(`cpu.weight`、`io.weight`: 既定 100、範囲 1〜10000、
兄弟間の比率で分配)、絶対上限は **max**(`memory.max`、`pids.max`、
`cpu.max`: 「これ以上は使わせない」)という2本立てが基本形です。
weight は混雑時だけ効く「保証の比率」、max は空いていても超えられない
「天井」——前分野の HTB の rate と ceil、その前のスケジューラの
nice と帯域制限、と同じ二本柱がここでも現れます。

## 内部動作の詳細

### struct pid — 多層の番号を最初から持つ

PID ネームスペースの「両方の番号を最初から持っている」を、データ
構造で確かめます。分野02では task_struct が pid/tgid を持つと
説明しましたが、正確には番号は task_struct 直下の整数ではなく、
**struct pid** という独立の構造体(kernel/pid.c)が担います。

```
  struct pid(1プロセス分)
  ┌───────────────────────────────┐
  │ level = 2        (属する深さ)  │
  │ numbers[0] = { nr=34567, 根のns }   ← ホストでの番号
  │ numbers[1] = { nr=12,    中間のns } │
  │ numbers[2] = { nr=1,     自分のns } ← コンテナ内での番号
  └───────────────────────────────┘
```

struct pid は、根のネームスペースから自分の属するネームスペースまで、
**階層の各段での番号(upid)の配列**を持ちます。getpid() は「呼び出した
プロセスのネームスペースの段の番号」を返し、親ネームスペースの `ps` は
親の段の番号を読む——どの窓から見るかで違う番号が見えるだけで、
変換テーブルはどこにもありません。シグナルの宛先解決(kill に渡された
番号がどのタスクか)も、常に「送り手のネームスペースの段」で照合される
ため、番号の解釈がずれることもありません。

### css_set — task_struct と cgroup ツリーの接点

cgroups 側の接点も task_struct にあります。タスクは自分の属する
cgroup への参照を、**css_set** という中間構造体(kernel/cgroup/)
経由で持ちます。

```
 task_struct ──→ css_set ──→ cgroup_subsys_state(cpu 用)──→ cgroup "web"
   (多数)      (同じ所属の      cgroup_subsys_state(memory 用)──→ 〃
                タスクで共有)    cgroup_subsys_state(io 用)──→ 〃
```

同じグループに属するタスクは同じ css_set を共有します(数千の
タスクがいても所属パターンの数だけしか css_set は要らない、という
省メモリ設計)。css_set が指す先の **cgroup_subsys_state(css)** が、
「このグループの、このコントローラにとっての状態」——CPU なら
グループ用のスケジューリングエンティティ、メモリならグループの
会計簿——を持ちます。「木は1本、コントローラごとの状態は css として
節に併設」が v2 の内部の姿です。

### cpu コントローラ — スケジューラのグループ化(分野02の伏線回収)

分野02のスケジューラ章で「グループ単位の公平分配は分野05で」と
予告しました。ここで回収します。

cpu コントローラの実体は、fair クラス(EEVDF)の**グループ
スケジューリング**です。仕組みは再帰で、グループはランキュー上で
**1個のスケジューリングエンティティ**として振る舞います。親の
ランキューでは「グループ web」全体が1人のタスクのように weight を
持って競争し、web が選ばれたら、今度は web の**内部のランキュー**で
所属タスク(または子グループ)が競争する——木を根から葉へ降りながら
EEVDF の選択を繰り返す入れ子構造です。

この見方をすると、`cpu.weight` は **nice 値の一般化**だと分かります。
nice がタスク1個の weight を決めた(1024 が基準、比率で分配)のと
同じ会計に、グループという中間層を挿入しただけです。既定の
weight 100 のグループが2つあれば CPU 混雑時に 50:50、200 と 100 なら
2:1。中のタスクが何百個あっても、まずグループ間で分けてから中で
分ける——「タスクの数で取り分が決まってしまう」素朴な公平の問題
(コンテナ A は1プロセス、B は100プロセスでも、A:B は weight 比)を
解くのがグループ化の意義です。

`cpu.max` は比率ではなく絶対上限で、「期間(period)あたりの使用時間」を
`$MAX $PERIOD`(µs 単位、既定 `max 100000`)で指定します。
100000 µs 中 50000 µs と書けば、空いていても 0.5 CPU 分しか
走れません。混雑時の比率(weight)と絶対の天井(max)の二本柱、
前節で見た型のとおりです。

### memory コントローラ — 会計簿と、グループ限定の OOM

メモリの制限は CPU と違う難しさを持ちます。CPU 時間は「次の tick で
誰を走らせるか」でいくらでも調整できますが、メモリは**すでに渡して
しまったページを後から取り上げるのが難しい**からです(回収=分野02で
見たとおり、書き戻しや破棄のコストがかかります)。

memory コントローラは、ページ単位の**記帳(charge)**で会計します。
プロセスがページフォールトで物理フレームを得るとき(分野02)、また
ファイルを読んでページキャッシュ(分野03)が作られるとき、その
ページは「使わせた先のグループ」の勘定に記帳されます。重要なのは
**ページキャッシュも勘定に入る**ことです。「アプリの使用メモリは
小さいのに memory.current が大きい」の大半はこれで、カーネルは
上限に近づくとそのグループのキャッシュから回収します(実害は
小さいことが多い)。

主なつまみは3つの水位です。

- `memory.max` — 硬い天井。超過しそうになると、まずグループ内で
  回収を試み、それでも足りなければ**そのグループの中だけを対象に
  OOM killer が走ります**。分野02で見た OOM killer の点数勘定が、
  システム全体ではなくグループ内で行われる形です。巻き添えが
  グループの塀の中に閉じ込められる——これが「奪ってしまう干渉」への
  回答です
- `memory.high` — 柔らかい天井。超えてもプロセスは殺さず、超過分の
  回収をそのグループのプロセス自身に(メモリを要求した文脈で)
  肩代わりさせ、実質的に減速させます。殺すほどではないが野放しにも
  しない、という中間の運用に使います
- `memory.low` — 保護の下限。システム全体が逼迫したとき、この量
  までは「他人のための回収」の対象になりにくくします(weight 系の
  発想のメモリ版)

OOM が起きたかどうかは `memory.events` の `oom_kill` カウンタで
観察できます。コンテナ運用で頻出の「exit code 137」の正体は
トラブルシューティングで扱います。

### pids・io — 数の上限と、帯域の分配

残る主要コントローラは要点のみ述べます。

- **pids コントローラ**は、グループ内のタスク数(`pids.current`)に
  上限(`pids.max`)を課します。守っている資源はメモリでも CPU でも
  なく、**PID 空間と task_struct という有限資源**です。暴走した
  fork(いわゆる fork 爆弾)がホスト全体の PID を食い尽くすのを、
  塀の中で止めます
- **io コントローラ**は、ブロックデバイス(分野03)への I/O に
  weight(`io.weight`)と上限(`io.max`: デバイスごとに bps / iops)を
  課します。ライトバック(分野03)のように「書いたプロセスと
  ディスクへ流すタイミングが分離している」I/O を誰の勘定に付けるか
  という難問があり、v2 はページの記帳(memory)と連動させて
  書き戻し I/O を発生源のグループに帰属させます——これこそ v1 の
  「木がバラバラでは連携できない」問題の代表例で、v2 統一階層の
  動機そのものです

なお、資源の逼迫を「どれだけ待たされたか」の側から測る PSI
(Pressure Stall Information)も cgroup 単位で観察できますが、
可観測性の話題として分野09に譲ります。

### 組み合わせて「コンテナ」になる

部品が出揃いました。コンテナランタイム(本分野第3章)が1個の
コンテナを作るとき、カーネルに対して行うことの骨子は次の通りです。

1. **ネームスペースを一式作る**(PID・mount・network・UTS・IPC、
   構成により user・cgroup・time も)——「見えてしまう干渉」の遮断
2. **cgroup を1つ作って全プロセスを収め、取り分を書き込む**
   (`cpu.weight`、`memory.max` 等)——「奪ってしまう干渉」の遮断
3. mount ネームスペースの中で**ルートをイメージへ差し替える**
   (pivot_root)
4. network ネームスペースへ **veth の片割れを挿し、ホスト側で
   ブリッジに束ねる**(前分野で見た標準構成そのもの)。コンテナの
   帯域制限は、この veth への qdisc の取り付けとして実現されます
   (前分野最終章の予告の回収)
5. 仕上げに capability の削減や seccomp によるシステムコール制限
   などの**攻撃面の縮小**を施す(分野07で扱う話題)

どの行にも新しいカーネル機構は登場しません。**コンテナとは、本書で
ここまで学んだ機構の編成(オーケストレーション)である**——これが
本章の結論です。

## 実行例

以下は Ubuntu Server 26.04 LTS での例です。まず、自分がすでに
cgroup ツリーの中にいることを確かめます(systemd は起動時から
全プロセスを木に配置しています——詳細は分野06)。

```console
$ cat /proc/self/cgroup
0::/user.slice/user-1000.slice/session-4.scope   ← v2 は 1 行だけ("0::")
$ ls /sys/fs/cgroup/user.slice/
cgroup.controllers  cgroup.procs  cpu.weight  memory.current ...
$ cat /sys/fs/cgroup/user.slice/cgroup.controllers
cpuset cpu io memory pids                        ← この節で使えるコントローラ
```

メモリ上限の効き方は、systemd 経由で一時的な cgroup を作ると
安全に観察できます。

```console
$ systemd-run --user --scope -p MemoryMax=100M \
      python3 -c 'a = bytearray(300 * 1024 * 1024)'
Running scope as unit: run-r1b2....scope
Killed                                           ← グループ内 OOM で SIGKILL
$ echo $?
137                                              ← 128 + 9(SIGKILL)
```

PID ネームスペースは unshare で体験できます(`-r` で user
ネームスペースを併用し中の root になる。`-f` の意味は
トラブルシューティング参照)。

```console
$ unshare -r -f -p --mount-proc bash    ← user+PID ns、/proc も貼り直し
# ps aux
USER  PID %CPU %MEM ... COMMAND
root    1  0.0  0.0 ... bash             ← 自分が PID 1。ホストの他プロセスは見えない
root   15  0.0  0.0 ... ps aux
# readlink /proc/self/ns/pid
pid:[4026532745]                         ← ホスト側の値とは別の inode 番号
```

(非特権 user ネームスペースの作成が AppArmor で制限されている
環境では、`unshare -r` が Permission denied になることがあります。
Ubuntu 26.04 の既定は要検証です。)

## トラブルシューティング

- **コンテナに SIGTERM を送っても終わらない(docker stop が
  10秒待って強制殺害になる)。** コンテナ内の PID 1 には、ハンドラを
  登録していないシグナルのデフォルト処分が適用されません(本章
  「PID 1 の特別扱い」)。単なるアプリをそのまま PID 1 に据えると、
  SIGTERM を「握りつぶす」のではなく**そもそも届かない**状態に
  なります。アプリ側でハンドラを入れるか、シグナルを扱える小さな
  init(tini 等)を PID 1 に立てるのが定石です
- **exit code 137 で死ぬ。** 137 = 128 + 9、すなわち SIGKILL です。
  コンテナ文脈での最頻の原因はグループ内 OOM(`memory.max` 超過)。
  当該 cgroup の `memory.events` の `oom_kill` が増えていれば確定です。
  `memory.current` にはページキャッシュも含まれる点に注意して
  上限を見積もります
- **`unshare -p bash` が「Cannot allocate memory」等で即死する。**
  PID ネームスペースは unshare した本人を移しません。fork した子から
  が新世界なので、`-f`(unshare 自身が fork してから bash を実行)が
  必要です。また `--mount-proc` を忘れると、新しいネームスペースなのに
  `ps` にはホストの全プロセスが見えます(`/proc` が古いままのため)
- **古い資料と `/sys/fs/cgroup` の中身が食い違う。** `memory/` や
  `cpu/` というコントローラ別ディレクトリ、`cpu.shares`(v2 では
  `cpu.weight`)、`memory.limit_in_bytes`(v2 では `memory.max`)が
  出てくる資料は v1 のものです。v1/v2 は記法も規則も別物として
  読み替えが必要です
- **cgroup にプロセスを入れられない(No such process ではなく
  Device or resource busy / Invalid argument)。** 子グループを持つ
  節に直接プロセスを書き込もうとした場合、v2 の no internal process
  制約に当たっている可能性があります。プロセスは葉へ、分配の設定は
  節へ、が v2 の規則です

## 演習・確認問題

1. 「カーネルにはコンテナという概念がない」とはどういう意味か。
   ネームスペースと cgroups がそれぞれ何の干渉を防ぐ機構かを添えて、
   仮想マシンとの違いと合わせて説明せよ。
2. コンテナ内で PID 1 のプロセスが、ホストの `ps` では PID 34567 と
   表示される。この2つの番号はどこにどう保持されているか。
   struct pid の構造に触れて説明せよ。
3. `cpu.weight=100` のグループ A(タスク40個)と `cpu.weight=200` の
   グループ B(タスク1個)が1つの CPU を取り合うとき、長期的な
   CPU 時間の比はどうなるか。グループスケジューリングの入れ子構造から
   説明せよ。
4. cgroup v1 の「コントローラごとに別の木」は、なぜ v2 で1本に
   統一されたか。書き戻し I/O の帰属を例に説明せよ。
5. `unshare -r` で user ネームスペースを作ると、なぜ一般ユーザーの
   ままで PID や mount のネームスペースを作れるようになるのか。
   capability と user ネームスペースの関係から説明せよ。

## まとめ

- コンテナはカーネルの機構ではなく、**眺めを分けるネームスペース**と
  **取り分を決める cgroups** をユーザーランドが束ねた編成である。
  全員が同じカーネルの上のプロセスで、仮想マシンとは切る層が違う
- ネームスペースは8種類あるが設計は共通(nsproxy による所属、fork での
  継承、clone/unshare/setns、/proc の名札)。癖が大きいのは入れ子と
  PID 1 特権を持つ **PID ns** と、権限の文脈そのものを分ける **user ns**
- cgroup v2 は**1本の木**に全コントローラが載る。相対配分は weight、
  絶対上限は max の二本柱で、プロセスは葉に置く
- cpu コントローラの実体は EEVDF の入れ子(グループ=親のキュー上の
  1エンティティ)、memory は記帳とグループ限定 OOM——いずれも
  分野02の機構に「グループの軸」を足したものである
